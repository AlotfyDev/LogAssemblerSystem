# 1. Purpose

## 1.1

The Canonical Kernel is the shared governance layer used by both Assembler skills:

1. Content Editor Skill,
2. Query Expert Skill.

## 1.2

The kernel defines the canonical distinctions, rules, authority model, evidence expectations, and validation principles that both skills must obey.

## 1.3

The kernel does not execute edits.

## 1.4

The kernel does not answer user queries directly.

## 1.5

The kernel does not run adapters.

## 1.6

The kernel does not define runtime-specific behavior.

## 1.7

The kernel is the common semantic constraint layer beneath both skill cores.

# 6. Responsibility vs Obligation

## 6.1 Responsibility Meaning

A responsibility answers:

```text id="d97i1t"
What must be owned or carried?
```

## 6.2 Obligation Meaning

An obligation answers:

```text id="f0srjs"
What must be preserved, enforced, guaranteed, or prevented?
```

## 6.3 Delegation Difference

A responsibility may be delegated to a primary entity and supporting entities.

An obligation may be delegated to enforcement entities and preserving objects.

## 6.4 Kernel Rule

```text id="qfkv1u"
Do not collapse responsibilities into obligations.
Do not collapse obligations into responsibilities.
```

## 6.5 Query Implication

A responsibility query must not be answered only from obligation files unless the answer clearly identifies the obligation relationship.

## 6.6 Editor Implication

An edit that changes a responsibility may require obligation-link validation.

An edit that changes an obligation may require responsibility-link validation.

---

# 7. Entity vs Governed Object

## 7.1 Entity Meaning

An entity can carry, moderate, assemble, manage, validate, or dispatch responsibility.

## 7.2 Governed Object Meaning

A governed object is controlled, validated, emitted, stored, resolved, or interpreted by an entity.

## 7.3 Kernel Rule

```text id="xny3lo"
Delegated entities carry responsibility.
Governed objects are controlled by responsibility carriers.
```

## 7.4 Invalid Collapse

Invalid:

```text id="coasqj"
Envelope Slot Occupancy owns Log Level Envelope lifecycle.
```

Valid:

```text id="dt89x8"
Envelope Slot Occupancy records slot binding/state.
Lifecycle ownership remains elsewhere.
```

## 7.5 Editor Implication

If an edit promotes a governed object into a delegated entity, it must be recorded as a taxonomy or entity update.

## 7.6 Query Implication

If a user asks “who owns this,” the query skill must distinguish actor/entity ownership from governed-object participation.

---

# 8. Enablement vs Implementation

## 8.1 Enablement Meaning

Enablement is the canonical capability family that makes responsibility execution possible.

## 8.2 Implementation Meaning

Implementation is concrete code, runtime behavior, service logic, database design, or environment-specific mechanism.

## 8.3 Kernel Rule

```text id="qtm3mf"
Enablement families are canonical capability classes.
They are not implementation modules.
```

## 8.4 Editor Implication

Adding implementation detail must not rewrite enablement taxonomy unless the change is explicitly a canonical enablement update.

## 8.5 Query Implication

The Query Expert Skill may explain enablement requirements, but must not infer implementation code behavior from enablement labels.

---

# 9. Surface vs Port vs Actor

## 9.1 Surface Meaning

A surface is an exposed or interaction-facing boundary.

## 9.2 Port Meaning

A port is a more specific delivery, registry, or contract-facing surface.

## 9.3 Actor Meaning

An actor carries assigned responsibility.

## 9.4 Kernel Rule

```text id="nbh9au"
A surface exposes interaction.
A port structures access or delivery.
An actor carries delegated responsibility.
```

```text id="tz0lrc"
Registry / Delivery Port is the final outbound actor.
```

Valid:

```text id="ss7wao"

```
