# Canonical Assignment Principles
## Delegation, Obligation, Governed Object, and Enablement Rules

---

# 0. Document Purpose

## 0.1
This document defines the canonical assignment principles used by the **Canonical Delegation & Enablement Matrix V2**.

## 0.2
It explains how responsibilities and obligations are assigned to architectural entities, how governed objects are distinguished from actors, and how enablements and surfaces are derived.

## 0.3
This file is part of:

```text
01_reference_markdown/
````

and supports the main reference file:

```text
01_reference_markdown/01.01_Canonical_Delegation_Enablement_Matrix_V2.md
```

## 0.4

The principles here are derived from:

1. the contract expansion from external relations into structured contract families,
2. the unified taxonomy of external, internal, runtime, carrier, contract, doctrine, and relation layers,
3. the corrected Batch 4 delegation matrix,
4. the gap / closure corrections integrated into the release.

---

# 1. Architectural Transformation Chain

## 1.1

The Assembler Subsystem is analyzed through the following transformation chain:

```text
External Relation
→ Contract
→ Responsibility
→ Obligation
→ Delegation
→ Enablement
→ Capability Candidate
```

## 1.2

Each stage answers a different architectural question.

| Stage                | Question Answered                                        |
| -------------------- | -------------------------------------------------------- |
| External Relation    | Who interacts with whom, and in what direction?          |
| Contract             | What formal interaction law governs that relation?       |
| Responsibility       | What must the system or component achieve?               |
| Obligation           | What must be guaranteed or prevented while achieving it? |
| Delegation           | Which architectural entity carries the responsibility?   |
| Enablement           | What capability or surface must that entity expose?      |
| Capability Candidate | What may later become implementation-level behavior?     |

## 1.3

The current release focuses on:

```text
Responsibility / Obligation
→ Delegation
→ Enablement
```

## 1.4

It does not finalize:

1. implementation packages,
2. classes,
3. function signatures,
4. database tables,
5. runtime scheduling algorithms.

---

# 2. Responsibility Definition

## 2.1

A **responsibility** is an architectural duty derived from a contract, closure, doctrine, or runtime model.

## 2.2

A responsibility states:

```text
what must be achieved
```

## 2.3

A responsibility is not necessarily:

1. a class,
2. a method,
3. a function,
4. a runtime thread,
5. a package.

## 2.4

Example:

```text
RSP-CAN-025
Update delivery-cycle flags after delivery round completion.
```

This is a responsibility because the system must achieve a post-delivery state update.

## 2.5

Responsibility assignment does not automatically imply ownership of every object affected by the responsibility.

## 2.6

Example:

```text
Dispatcher handles envelope handoff.
```

This does not mean:

```text
Dispatcher owns the envelope lifecycle after registry handoff.
```

---

# 3. Obligation Definition

## 3.1

An **obligation** is a constraint, guarantee, prohibition, or condition attached to a responsibility.

## 3.2

An obligation states:

```text
what must be guaranteed while achieving the responsibility
```

## 3.3

Example:

```text
Responsibility:
RSP-CAN-025
Update delivery-cycle flags after delivery round completion.

Obligation:
Delivery-cycle flags must not be conflated with detailed dispatcher feedback reports.
```

## 3.4

Obligations may:

1. restrict a responsibility,
2. clarify a boundary,
3. prevent a false ownership assumption,
4. preserve a doctrine rule,
5. enforce a closure decision.

## 3.5

A single responsibility may have:

1. one obligation,
2. several obligations,
3. one primary obligation plus sub-obligations,
4. sub-responsibility obligations, such as delivery-cycle flags or dispatcher reports.

---

# 4. System Accountability vs Delegated Entity

## 4.1

The **Assembler Subsystem** remains the system-level accountability owner for assembler-side responsibilities.

## 4.2

However, this does not mean every responsibility is directly delegated to the Assembler Subsystem.

## 4.3

The corrected principle is:

```text
Assembler Subsystem = accountable
Narrow mature architectural entity = delegated carrier
```

## 4.4

Examples:

| Responsibility Area                  | System Accountability Owner | Delegated Entity         |
| ------------------------------------ | --------------------------- | ------------------------ |
| Envelope assembly                    | Assembler Subsystem         | Envelope Assembler       |
| Policy assignment                    | Assembler Subsystem         | PolicyAssigner           |
| Registry handoff                     | Assembler Subsystem         | Dispatcher               |
| Outbound registry contract execution | Assembler Subsystem         | Registry / Delivery Port |
| Single queue-container moderation    | Assembler Subsystem         | Single Queue Moderator   |
| Queue bundle orchestration           | Assembler Subsystem         | Queue Bundle Agent       |

## 4.5

This separation prevents overloading `E-INT-001 Assembler Subsystem` with operational details that should belong to narrower architectural carriers.

---

# 5. Primary Delegated Entity

## 5.1

The **primary delegated entity** is the entity that directly carries the responsibility.

## 5.2

The primary delegated entity should be the narrowest mature entity capable of carrying the responsibility.

## 5.3

Assignment should prefer a mature concrete architectural carrier over a broad logical grouping.

## 5.4

Example:

```text
Timestamp stabilization
→ Timestamp Stabilizer
```

not:

```text
Timestamp stabilization
→ Assembler Subsystem
```

## 5.5

Example:

```text
Queue-container moderation
→ Single Queue Moderator
```

not:

```text
Queue-container moderation
→ Queue Runtime Layer
```

---

# 6. Supporting Entity

## 6.1

A **supporting entity** participates in fulfilling a responsibility without being the primary carrier.

## 6.2

Supporting entities may:

1. provide data,
2. provide a subordinate mechanism,
3. receive a handoff,
4. participate in composition,
5. maintain related state.

## 6.3

Example:

| Responsibility                            | Primary Delegated Entity | Supporting Entities                                |
| ----------------------------------------- | ------------------------ | -------------------------------------------------- |
| Assemble Log Level Envelope               | Envelope Assembler       | Validator, Metadata Injector, Timestamp Stabilizer |
| Preserve registry handoff discipline      | Dispatcher               | Registry / Delivery Port                           |
| Restore waiting discipline after delivery | Single Queue Moderator   | Queue Bundle Agent                                 |

## 6.4

Supporting status must not be confused with primary responsibility ownership.

---

# 7. Governed Object

## 7.1

A **governed object** is the object, carrier, state structure, contract artifact, or runtime artifact affected by a responsibility.

## 7.2

A governed object is not automatically an actor.

## 7.3

Common governed objects include:

1. Policy Object,
2. AssemblerPolicyBundle,
3. Log Level Envelope,
4. Queue Container,
5. Slot,
6. Envelope Slot Occupancy,
7. Slot Lifecycle State,
8. Queue Container Identifier,
9. Delivery-Cycle Flag Set,
10. Dispatcher Feedback Report.

## 7.4

Example:

```text
Dispatcher hands off Log Level Envelope.
```

Here:

```text
Dispatcher = delegated entity
Log Level Envelope = governed object
```

## 7.5

Example:

```text
Single Queue Moderator maintains Envelope Slot Occupancy.
```

Here:

```text
Single Queue Moderator = delegated entity
Envelope Slot Occupancy = governed object
```

## 7.6

The release must not promote governed objects into actors unless a later taxonomy revision explicitly does so.

---

# 8. Enablement

## 8.1

**Enablement** is the required capability, surface, or mechanism that allows the delegated entity to carry the responsibility.

## 8.2

Enablement is not final code.

## 8.3

Enablement answers:

```text
What must this entity be able to do?
```

## 8.4

Examples:

| Responsibility                           | Delegated Entity       | Required Enablement                           |
| ---------------------------------------- | ---------------------- | --------------------------------------------- |
| Assign Policy Objects                    | PolicyAssigner         | policy assignment surface                     |
| Admit envelope into queue-container slot | Assembler Ingress Port | envelope admission and slot targeting surface |
| Update delivery-cycle flags              | Dispatcher             | delivery flag state update surface            |
| Produce feedback reports                 | Dispatcher             | diagnostic/reporting surface                  |
| Moderate one queue-container             | Single Queue Moderator | single queue moderation surface               |

## 8.5

Enablement is grouped into canonical enablement families for queryability and reuse.

---

# 9. Surface Type

## 9.1

A **surface type** classifies the level or plane at which a responsibility is carried.

## 9.2

Canonical surface types are:

| Surface Type   | Meaning                                                      |
| -------------- | ------------------------------------------------------------ |
| Administrative | Governance, assignment, provisioning, control, configuration |
| Managerial     | Runtime realization, orchestration, lifecycle, moderation    |
| Consuming      | Content-facing, API-facing, consumer admission               |
| Boundary       | Authority edge, subsystem separation, handoff edge           |
| Contractual    | Typed/template contract law and admissibility                |
| Observability  | Reporting, diagnostics, feedback artifacts                   |

## 9.3

A responsibility may have:

1. one primary surface,
2. one or more secondary surfaces.

## 9.4

Example:

```text
RSP-CAN-035 Preserve template contract doctrine
```

Primary surface:

```text
Contractual
```

## 9.5

Example:

```text
RSP-CAN-048 Produce detailed dispatcher feedback reports
```

Primary surface:

```text
Managerial / Observability
```

# 10. Port Assignment Principle

## 10.1

The corrected model replaces a generic “contract layer” with concrete port-like architectural surfaces.

## 10.2

Canonical ports and contract-side objects are:

| ID        | Name                            | Role                                   |
| --------- | ------------------------------- | -------------------------------------- |
| E-INT-011 | Envelope Assembler              | client-side composed contract object   |
| E-INT-012 | Assembler Ingress Port          | server-side receiving contract port    |
| E-INT-013 | Administrative / Ecosystem Port | administrative/control contract port   |
| E-INT-014 | Registry / Delivery Port        | outbound registry-facing contract port |

## 10.3

Ports are assigned when the responsibility concerns:

1. contract admission,
2. payload boundary,
3. inbound administrative control,
4. outbound registry contract execution,
5. authority separation.

## 10.4

The Dispatcher remains the actor for outbound delivery, while `Registry / Delivery Port` is the contract surface.

## 10.5

The Log Level API remains external, while `Envelope Assembler` is the client-side composed contract object associated with it.

# 11. Policy Object Assignment Principle

## 11.1

The **Policy Object** is a generic assignable architectural object.

## 11.2

It governs mechanism semantics.

## 11.3

It does not execute behavior.

## 11.4

The component to which the Policy Object is assigned hosts and realizes the mechanism.

## 11.5

Canonical form:

```text
Policy Object = semantic governor
Component = mechanism host
```

## 11.6

Policy assignment responsibilities are delegated to:

```text
PolicyAssigner
```

## 11.7

Policy-hosting responsibilities are delegated to policy-consuming components.

## 11.8

Metadata Injector is not a policy-consuming component.

## 11.9

Policy-consuming components include:

1. Dispatcher,
2. Writing / Feeding Round Manager,
3. Dispatching / Delivery Round Manager,
4. Queue Bundle Agent,
5. Single Queue Moderator,
6. Timestamp Stabilizer,
7. Validator.

---

# 12. Metadata Assignment Principle

## 12.1

Metadata is schema/directive-governed.

## 12.2

Metadata may be changed by schema-guided directive or field value update.

## 12.3

Metadata does not require Policy Object hosting.

## 12.4

Therefore:

```text
Metadata Injector ≠ Policy Host
```

## 12.5

Metadata-related responsibilities are delegated to:

1. Administrative / Ecosystem Port,
2. Metadata Injector,
3. Envelope Assembler where envelope composition is involved.

## 12.6

Metadata authority must not collapse into consuming content ingress.

---

# 13. Envelope Assignment Principle

## 13.1

The **Log Level Envelope** is the authoritative runtime carrier.

## 13.2

It carries:

1. accepted content,
2. metadata,
3. stabilized timestamp,
4. runtime transport identity,
5. lifecycle and handoff semantics.

## 13.3

The envelope is normally a governed object.

## 13.4

The envelope is assembled by:

```text
Envelope Assembler
```

## 13.5

The envelope is received by:

```text
Assembler Ingress Port
```

## 13.6

The envelope is moved through round progression by:

1. Writing / Feeding Round Manager,
2. Dispatching / Delivery Round Manager.

## 13.7

The envelope is handed off by:

```text
Dispatcher
```

through:

```text
Registry / Delivery Port
```

## 13.8

After valid handoff, lifecycle authority transfers to the Log Level Records Registry.

---

# 14. Queue / Container Assignment Principle

## 14.1

The corrected runtime law is:

```text
Queue = Container
```

## 14.2

There is no separate runtime container above the queue.

## 14.3

A Queue Container contains Slots.

## 14.4

A Log Level Envelope occupies a Slot through Envelope Slot Occupancy.

## 14.5

Slot Lifecycle State applies to the Slot, not to the envelope lifecycle after registry handoff.

## 14.6

Queue runtime responsibilities must be delegated to:

1. Queue Bundle Agent,
2. Single Queue Moderator,
3. Writing / Feeding Round Manager,
4. Dispatching / Delivery Round Manager.

## 14.7

`Queue Runtime Layer` may appear only as a classification label, not as a delegated entity.

---

# 15. Waiting List Assignment Principle

## 15.1

Round Manager waiting lists store queue-container identifiers.

## 15.2

They do not store envelopes.

## 15.3

Canonical form:

```text
Writing Round Waiting List
→ Queue Container Identifier
→ Queue Container
```

```text
Delivery Round Waiting List
→ Queue Container Identifier
→ Queue Container
```

## 15.4

Writing / Feeding Round Manager governs writing-side availability indexing.

## 15.5

Dispatching / Delivery Round Manager governs delivery-side availability indexing.

## 15.6

Queue Bundle Agent may expose or update bundle-level availability.

## 15.7

Single Queue Moderator may requalify a queue-container ID back into the delivery waiting list after moderation.

---

# 16. Dispatcher Assignment Principle

## 16.1

Dispatcher is the final assembler-side boundary actor.

## 16.2

Dispatcher carries responsibilities related to:

1. registry readiness observation,
2. delivery gating,
3. envelope handoff,
4. delivery-cycle flag updates,
5. detailed dispatcher feedback reports,
6. lifecycle transfer discipline,
7. dispatcher boundary law.

## 16.3

Dispatcher acts through the Registry / Delivery Port for outbound registry contract execution.

## 16.4

Dispatcher must not own registry internals.

## 16.5

Dispatcher must not claim registry-side lifecycle after valid handoff.

---

# 17. Delivery Flags vs Dispatcher Reports Principle

## 17.1

Delivery-cycle flags and detailed dispatcher feedback reports must remain distinct.

## 17.2

Canonical distinction:

```text
Delivery-cycle flags = state updates
Detailed dispatcher feedback reports = diagnostic / reporting artifacts
```

## 17.3

Delivery-cycle flags are governed by:

```text
BCO-013 Delivery-Cycle Flag Set
```

## 17.4

Detailed dispatcher feedback reports are governed by:

```text
BCO-014 Dispatcher Feedback Report
```

## 17.5

Both are Dispatcher-related, but they require different enablement surfaces.

# 18. Assignment Decision Rules

## 18.1 Rule A — Assign to the Narrowest Mature Entity

### 18.1.1

If a mature entity exists in the taxonomy or corrected release, assign responsibility to it instead of a broader subsystem.

### 18.1.2

Example:

```text
Timestamp stabilization
→ Timestamp Stabilizer
```

not:

```text
Timestamp stabilization
→ Assembler Subsystem
```

---

## 18.2 Rule B — Preserve System Accountability Separately

### 18.2.1

Assembler Subsystem may remain accountable even when execution is delegated.

### 18.2.2

Do not confuse accountability with operational execution.

---

## 18.3 Rule C — Use Objects as Governed Objects Unless Promoted

### 18.3.1

Objects such as `Log Level Envelope`, `Policy Object`, `Queue Container`, `Slot`, and `Envelope Slot Occupancy` are governed objects unless explicitly promoted.

---

## 18.4 Rule D — Do Not Create False Runtime Containers

### 18.4.1

Do not introduce a `Runtime Container` above Queue Container.

### 18.4.2

The correct doctrine is:

```text
Queue = Container
```

---

## 18.5 Rule E — Split Flags from Reports

### 18.5.1

Do not merge delivery-cycle flags and dispatcher feedback reports.

### 18.5.2

Use:

1. Delivery-Cycle Flag Set for state updates,
2. Dispatcher Feedback Report for diagnostics/reporting.

---

## 18.6 Rule F — Use Ports for Boundary Contract Surfaces

### 18.6.1

Do not use a generic “contract layer” when a specific port is known.

### 18.6.2

Use:

1. Envelope Assembler,
2. Assembler Ingress Port,
3. Administrative / Ecosystem Port,
4. Registry / Delivery Port.

---

## 18.7 Rule G — Preserve Typed Template Contract Doctrine

### 18.7.1

Assembler contracts are typed/template laws.

### 18.7.2

They are not runtime COM interfaces.

---

## 18.8 Rule H — Preserve Policy Host Doctrine

### 18.8.1

Policy Object governs mechanism semantics.

### 18.8.2

Component hosts and realizes the mechanism.

---

# 19. Canonical Assignment Statement

## 19.1

A responsibility should be assigned to the entity that can most directly realize the architectural behavior while preserving the system’s authority boundaries, contract doctrine, and runtime object distinctions.

## 19.2

An obligation should be assigned to the entity that must enforce the relevant guarantee or prohibition.

## 19.3

A governed object should be recorded separately from the delegated entity so that carrier/state structures do not become false actors.

## 19.4

Enablement should describe the capability surface required by the delegated entity, not the final implementation.

## 19.5

The Assembler Subsystem remains accountable, but Batch 4 V2 delegates execution to mature architectural carriers such as:

1. Envelope Assembler,
2. Assembler Ingress Port,
3. Administrative / Ecosystem Port,
4. Registry / Delivery Port,
5. PolicyAssigner,
6. Dispatcher,
7. Queue Bundle Agent,
8. Single Queue Moderator,
9. Round Managers,
10. Validator,
11. Timestamp Stabilizer,
12. Metadata Injector where schema/directive preparation applies.

# 21. Final Principle Summary

## 21.1

The canonical assignment model preserves the following rules:

1. Assembler Subsystem is accountable, but narrow entities carry responsibilities.
2. Responsibility states what must be achieved.
3. Obligation states what must be guaranteed.
4. Delegated entity is the actor or carrier of responsibility.
5. Governed object is the affected object, artifact, or state structure.
6. Enablement is required capability, not final implementation.
7. Ports replace generic contract-layer ambiguity.
8. Metadata Injector is not a policy host.
9. Policy Object governs semantics; components host mechanisms.
10. Log Level Envelope is a governed carrier, not an actor.
11. Queue equals Container.
12. Queue Runtime Layer is not a delegated entity.
13. Delivery-cycle flags are separate from detailed dispatcher feedback reports.
14. Dispatcher remains the final assembler-side boundary.
15. Contracts are typed/template laws, not runtime COM interfaces.
