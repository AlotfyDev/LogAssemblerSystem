````markdown id="v2n3k9"
## 0.1
This document defines the corrected textual reference version of the **Canonical Delegation & Enablement Matrix V2** for the Assembler Subsystem.

It upgrades the legacy Batch 4 delegation and enablement layer using:
1. the canonical contract matrix,

3. the canonical responsibility and obligation matrix,

The document answers the following question:

> Given a canonical responsibility or obligation, which architectural entity carries it, what object does it govern, what enablement does it require, and how is the mapping traced back to contracts, taxonomy, and closures?
````

# 2. Canonical Assignment Principles

## 2.1 System Accountability vs Delegated Execution

### 2.1.1

The **Assembler Subsystem** remains the system-level accountability owner for assembler-side responsibilities.

### 2.1.2

However, execution must be delegated to the narrowest mature architectural carrier.

### 2.1.3

Therefore:

```text
Assembler Subsystem = accountable
Component / Port / Contract Object = delegated carrier
```

### 2.1.4

This prevents the old Batch 4 pattern where too many responsibilities remained broadly assigned to the Assembler Subsystem.

---

## 2.2 Responsibility vs Obligation

### 2.2.1

A **responsibility** states what must be achieved.

### 2.2.2

An **obligation** states what must be guaranteed, constrained, or prevented while fulfilling the responsibility.

### 2.2.3

Example:

```text
Responsibility:
Update delivery-cycle flags after delivery round completion.

Obligation:
Delivery-cycle flags must not be treated as detailed feedback reports.
```

---

## 2.3 Delegated Entity vs Governed Object

### 2.3.1

The **delegated entity** performs or carries the responsibility.

### 2.3.2

The **governed object** is the object, state structure, carrier, or artifact affected by the responsibility.

### 2.3.3

Example:

```text
Dispatcher handles Log Level Envelope handoff.
Log Level Envelope is governed object, not actor.
```

---

## 2.4 Primary Delegated Entity vs Supporting Entity

### 2.4.1

The **primary delegated entity** is the main responsibility carrier.

### 2.4.2

The **supporting entity** participates in fulfilling the responsibility but is not the primary carrier.

### 2.4.3

Example:

```text
Envelope Assembler assembles the envelope.
Metadata Injector and Timestamp Stabilizer support the assembly.
Log Level Envelope is the governed carrier.
```

---

## 2.5 Enablement

### 2.5.1

Enablement is the required capability, surface, or mechanism that allows the delegated entity to carry the responsibility.

### 2.5.2

Enablement is not final implementation. It is an architectural capability requirement.

---

## 2.6 Surface Type

### 2.6.1

Each responsibility is associated with one or more surface types.

### 2.6.2

Canonical surface types are:

1. Administrative,
2. Managerial,
3. Consuming,
4. Boundary,
5. Contractual,
6. Observability.

# 3. Corrected Delegation Entities and Artifacts

## 3.1 Envelope Assembler

### 3.1.1

`Envelope Assembler` is introduced as:

```text
E-INT-011 Envelope Assembler
```

### 3.1.2

It is a client-side composed contract object associated with the consuming entry path.

### 3.1.3

It composes:

1. Validator,
2. Metadata Injector,
3. Timestamp Stabilizer,
4. envelope assembly routine.

### 3.1.4

It is injected into, or composed inside, the Log Level API.

### 3.1.5

Its output is:

```text
BCO-012 Log Level Envelope
```

---

## 3.2 Assembler Ingress Port

### 3.2.1

`Assembler Ingress Port` is introduced as:

```text
E-INT-012 Assembler Ingress Port
```

### 3.2.2

It is the server-side receiving contract port.

### 3.2.3

It receives the Log Level Envelope and admits it into a suitable Slot inside the target Queue Container.

---

## 3.3 Administrative / Ecosystem Port

### 3.3.1

`Administrative / Ecosystem Port` is introduced as:

```text
E-INT-013 Administrative / Ecosystem Port
```

### 3.3.2

It receives administrative and ecosystem-level payloads, including:

1. policy bundles,
2. metadata directives,
3. operational directives,
4. ecosystem-level control/configuration.

### 3.3.3

It prevents administrative payloads from entering through the consuming ingress path.

---

## 3.4 Registry / Delivery Port

### 3.4.1

`Registry / Delivery Port` is introduced as:

```text
E-INT-014 Registry / Delivery Port
```

### 3.4.2

It is the outbound registry-facing contract port used by the Dispatcher.

### 3.4.3

The Dispatcher remains the actor.

### 3.4.4

The port is the contract execution surface.

---

## 3.5 Delivery-Cycle Flag Set

### 3.5.1

`Delivery-Cycle Flag Set` is introduced as:

```text
BCO-013 Delivery-Cycle Flag Set
```

### 3.5.2

It is a dispatcher state artifact.

### 3.5.3

It is not a diagnostic report.

---

## 3.6 Dispatcher Feedback Report

### 3.6.1

`Dispatcher Feedback Report` is introduced as:

```text
BCO-014 Dispatcher Feedback Report
```

### 3.6.2

### 3.6.3

It is not a delivery-cycle state flag.

# 4. Core Corrections Applied

## 4.1 Metadata Injector Is Not a Policy Host

### 4.1.1

`Metadata Injector` is schema/directive-governed.

### 4.1.2

It may apply metadata directives or schema-guided values.

### 4.1.3

It does not host Policy Object mechanisms.

---

## 4.2 Queue Runtime Layer Is Not a Delegated Entity

### 4.2.1

`Queue Runtime Layer` remains a classification layer.

### 4.2.2

Runtime responsibilities must be delegated to:

1. Queue Bundle Agent,
2. Single Queue Moderator,
3. Writing / Feeding Round Manager,
4. Dispatching / Delivery Round Manager.

---

## 4.3 Delivery Flags and Dispatcher Reports Are Separate

### 4.3.1

The corrected distinction is:

```text
Delivery-cycle flags = state updates
Detailed dispatcher feedback reports = diagnostic / reporting artifacts
```

### 4.3.2

Both are Dispatcher-related, but they are different governed artifacts and require different enablements.

---

## 4.4 Timestamp Stabilizer Is Mature

### 4.4.1

The old timestamp placeholder is closed.

### 4.4.2

`Timestamp Stabilizer` is now a mature component.

### 4.4.3

It participates in every accepted ingress path.

---

## 4.5 Queue Equals Container

### 4.5.1

The corrected runtime law is:

```text
Queue = Container
```

### 4.5.2

There is no separate runtime container above the queue.

### 4.5.3

All ownership, occupancy, and slot lifecycle terms must use `Queue Container`.

---

Accepted content must be transformed into a Log Level Envelope and must not remain raw content as runtime carrier.

## 17.1

The corrected Batch 4 delegation model assigns responsibilities not to broad logical layers, but to concrete architectural carriers.

The primary corrected carriers are:

1. Envelope Assembler,
2. Assembler Ingress Port,
3. Administrative / Ecosystem Port,
4. Registry / Delivery Port,
5. PolicyAssigner,
6. Dispatcher,
7. Queue Bundle Agent,
8. Single Queue Moderator,

10. Dispatching / Delivery Round Manager.

The primary governed objects and artifacts are:

1. Log Level Envelope,
2. Policy Object,
3. Queue Container,
4. Slot,
5. Envelope Slot Occupancy,
6. Slot Lifecycle State,
7. Delivery-Cycle Flag Set,
8. Dispatcher Feedback Report.

## 17.4

The model preserves the following laws:

1. Metadata Injector is not a policy host.
2. Policy Object governs mechanism semantics; components host mechanisms.
3. Dispatcher remains the final assembler-side boundary.
4. Delivery-cycle flags and dispatcher feedback reports are distinct.
5. Timestamp Stabilizer is mature and mandatory in ingress.
6. Queue equals Container.
7. Queue Runtime Layer is not a delegated entity.
8. Contracts are typed/template laws, not runtime COM interfaces.
