# Release Query Guide
## Canonical Delegation & Enablement Matrix V2

---

# 0.1 Purpose

This guide explains how to query the **Canonical Delegation & Enablement Matrix V2 Release** after installing the files under:

```text
canonical_delegation_enablement_matrix_v2_release/
````

It focuses on practical lookup paths, join keys, and common questions.

The release is designed to support queries across:

1. responsibilities,
2. obligations,
3. delegated entities,
4. supporting entities,
5. governed objects,
6. enablement families,
7. surface types,
8. contract traces,
9. gap / closure traces,
10. taxonomy update traces.

---

# 0.2 Main Query Principle

The release is not organized as one monolithic table.

Instead, it is organized as linked indexes and matrices.

The core query pattern is:

```text
responsibility_id
→ delegation_id
→ primary_delegated_entity_id
→ enablement_family_id
→ governed_object_id
→ trace ids
```

For obligations:

```text
obligation_id
→ responsibility_id
→ delegated entity
→ governed object
→ obligation focus
```

For traces:

```text
contract_id / gap_id / taxonomy_trace_id
→ responsibility_id
→ entity_id
→ enablement_family_id
```

# 3.1 Common Query Patterns

## 3.1.1 Find the Entity Responsible for a Responsibility

Question:

> Who carries `RSP-CAN-025`?

Query:

1. Open `03_delegation_matrix/03.01_delegation_matrix_v2.csv`.
2. Filter `responsibility_id = RSP-CAN-025`.
3. Read:

   1. `primary_delegated_entity_id`,
   2. `primary_delegated_entity_name`,
   3. `supporting_entity_ids`,
   4. `governed_object_ids`,
   5. `enablement_family_ids`.

Expected result:

```text
RSP-CAN-025
→ Dispatcher
→ Delivery-Cycle Flag Set
→ EN-FAM-006 Feedback / Reporting Enablement
```

Important distinction:

```text
RSP-CAN-025 = delivery-cycle flags
RSP-CAN-048 = detailed dispatcher feedback reports
```

---

## 3.1.2 Find All Responsibilities Assigned to an Entity

Question:

> What responsibilities are assigned to `Dispatcher`?

Query:

1. Open `03_delegation_matrix/03.01_delegation_matrix_v2.csv`.
2. Filter rows where:

   1. `primary_delegated_entity_id = E-INT-002`, or
   2. `primary_delegated_entity_name = Dispatcher`.
3. Optionally include rows where `supporting_entity_ids` contains `E-INT-002`.

Expected responsibility areas:

1. registry readiness,
2. delivery gating,
3. handoff,
4. delivery-cycle flags,
5. detailed feedback reports,
6. boundary discipline,
7. lifecycle transfer.

---

## 3.1.3 Find All Obligations Related to an Entity

# 5.1 Special Query Cases

## 5.1.1 Distinguish Delivery Flags from Dispatcher Reports

Use:

1. `04_obligation_delegation/04.03_delivery_cycle_flags_delegation.csv`,
2. `04_obligation_delegation/04.04_dispatcher_feedback_reports_delegation.csv`.

Do not infer one from the other.

Canonical rule:

```text
Delivery-cycle flags are state updates.
Detailed dispatcher feedback reports are diagnostic/reporting artifacts.
```

---

## 5.1.2 Verify Metadata Injector Is Not a Policy Host

Use:

1. `02_entity_enablement_indexes/02.06_policy_consuming_components_index.csv`,
2. `03_delegation_matrix/03.01_delegation_matrix_v2.csv`,
3. `04_obligation_delegation/04.01_obligation_delegation_matrix_v2.csv`.

Expected result:

```text
Metadata Injector
→ schema/directive-governed
→ not policy-consuming component
```

It may appear in metadata and envelope-preparation responsibilities, but not as a Policy Object host.

---

## 5.1.3 Verify Queue Runtime Layer Is Not a Delegated Entity

Use:

1. `02_entity_enablement_indexes/02.01_delegation_entities_index.csv`,
2. `03_delegation_matrix/03.01_delegation_matrix_v2.csv`.

Expected result:

```text
Queue Runtime Layer
→ classification only
→ not delegated entity
```

Runtime responsibilities should be assigned to:

1. `Queue Bundle Agent`,
2. `Single Queue Moderator`,
3. `Writing / Feeding Round Manager`,
4. `Dispatching / Delivery Round Manager`.

---

## 5.1.4 Verify Timestamp Placeholder Closure

Use:

1. `05_trace_coverage_registers/05.02_gap_closure_coverage_v2.csv`,
2. `03_delegation_matrix/03.01_delegation_matrix_v2.csv`.

Expected result:

```text
G-004
→ Timestamp Stabilizer
→ Envelope Assembler
→ no artificial placeholder entity
```

---

## 5.1.5 Verify Queue-as-Container Correction

Use:

1. `03_delegation_matrix/03.01_delegation_matrix_v2.csv`,
2. `04_obligation_delegation/04.01_obligation_delegation_matrix_v2.csv`,
3. `05_trace_coverage_registers/05.02_gap_closure_coverage_v2.csv`.

Expected rule:

```text
Queue = Container
Queue Container contains Slot
Log Level Envelope occupies Slot through Envelope Slot Occupancy
Slot Lifecycle State is not Log Envelope lifecycle ownership
```

# 6.1 Suggested SQL-Like Filters

These examples assume CSVs are imported into a query engine.

## 6.1.1 Responsibilities by Entity

```sql
SELECT responsibility_id,
       responsibility_name,
       responsibility_family_id,
       governed_object_ids,
       enablement_family_ids
FROM delegation_matrix_v2
WHERE primary_delegated_entity_name = 'Dispatcher';
```

---

## 6.1.2 Obligations by Governed Object

```sql
SELECT obligation_id,
       responsibility_id,
       obligation_statement,
       primary_delegated_entity_name
FROM obligation_delegation_matrix_v2
WHERE governed_object_ids LIKE '%Queue Container%';
```

---

## 6.1.3 Gap Coverage

```sql
SELECT gap_id,
       gap_name,
       covered_by_responsibility_ids,
       covered_by_entity_ids,
       coverage_status
FROM gap_closure_coverage_v2
WHERE gap_id = 'G-003';
```

---

## 6.1.4 Enablement Load by Entity

```sql
SELECT primary_delegated_entity_name,
       enablement_family_ids,
       COUNT(*) AS responsibility_count
FROM delegation_matrix_v2
GROUP BY primary_delegated_entity_name, enablement_family_ids;
```

---

## 6.1.5 Contract Trace

```sql
SELECT contract_id,
       responsibility_id,
       delegated_entity_id,
       governed_object_id,
       enablement_family_id
FROM contract_to_responsibility_to_entity_trace
WHERE contract_id = 'CC-ENV-001';
```

# 7.1 Recommended Review Queries

Before accepting the release, run or manually check the following.

## 7.1.1 Dispatcher Review

Confirm:

1. `Dispatcher` owns handoff responsibilities.
2. `Dispatcher` owns delivery-cycle flag update responsibility.
3. `Dispatcher` owns detailed feedback report responsibility.
4. `Registry / Delivery Port` supports contract execution.
5. `Dispatcher` does not own registry lifecycle after valid handoff.

---

## 7.1.2 Policy Review

Confirm:

1. `PolicyAssigner` handles policy intake and assignment.
2. `Policy Object` is governed semantic object, not executor.
3. Policy-consuming components exclude `Metadata Injector`.
4. Components host mechanism semantics locally.

---

## 7.1.3 Ingress Review

Confirm:

1. `Envelope Assembler` handles ingress preparation.
2. `Validator` only participates in validating ingress.
3. `Timestamp Stabilizer` participates in all ingress paths.
4. `Assembler Ingress Port` receives the produced envelope.

---

## 7.1.4 Runtime Review

Confirm:

1. `Queue Bundle Agent` handles bundle-level orchestration.
2. `Single Queue Moderator` handles one queue-container.
3. Round Managers handle waiting-list indexing.
4. Queue-container IDs, not envelopes, are stored in waiting lists.

---

# 8.1 Query Safety Notes

## 8.1.1

Do not treat comma-separated ID fields as normalized many-to-many tables unless you split them during import.

## 8.1.2

For precise joins, split fields such as:

1. `supporting_entity_ids`,
2. `governed_object_ids`,
3. `enablement_family_ids`,
4. `contract_trace_ids`,
5. `gap_closure_trace`.

## 8.1.3

Future normalized versions may create bridge tables for these fields.
