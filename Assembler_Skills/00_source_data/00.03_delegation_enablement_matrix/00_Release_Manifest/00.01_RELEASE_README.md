# Canonical Delegation & Enablement Matrix V2 Release

## 0.1 Purpose

This release defines the corrected **Canonical Delegation & Enablement Matrix V2** for the Assembler Subsystem.

It upgrades the legacy Batch 4 delegation and enablement material into a taxonomy-aware, closure-covered, queryable release.

The release answers the following architectural question:

> Given a canonical responsibility or obligation, which architectural entity carries it, what objects does it govern, what enablement does it require, and how can the mapping be traced back to contracts, taxonomy, and gap closures?

## 0.4 What This Release Produces

The release maps:

```text
Responsibility / Obligation
→ Primary Delegated Entity
→ Supporting Entities
→ Governed Objects
→ Enablement Family
→ Surface Type
→ Trace
````

This means the release does **not** merely restate responsibilities.
It assigns each responsibility to the most appropriate mature architectural carrier and describes what that carrier must be able to do.

---

## 0.5 What This Release Does Not Do

This release does not define:

1. final implementation packages,
2. final code classes,
3. database schema,
4. PostgreSQL derivation,
5. runtime execution algorithms,
6. final deployment topology.

Those are downstream activities.

This release remains an architectural delegation and enablement layer.

## 2.1 Core Architectural Corrections Integrated

This release integrates the following corrections.

### 2.1.1 Metadata Injector Is Not a Policy Host

`Metadata Injector` is schema/directive-governed.

It may receive metadata directives or schema-guided field updates, but it does not host `Policy Object` mechanisms.

Therefore, it is excluded from the canonical list of policy-consuming components.

---

### 2.1.2 Envelope Assembler Is a Client-Side Composed Contract Object

`Envelope Assembler` is introduced as a composed contract object associated with the consuming entry path.

It composes:

1. `Validator`,
2. `Metadata Injector`,
3. `Timestamp Stabilizer`,
4. envelope assembly routine.

It is injected into, or composed inside, the `Log Level API`.

It produces or prepares the `Log Level Envelope`.

---

### 2.1.3 Assembler Ingress Port Is a Server-Side Receiving Port

`Assembler Ingress Port` is introduced as the Assembler-side contract port that receives the `Log Level Envelope`.

Its responsibility is to admit the envelope into the suitable slot inside the target `Queue Container`.

---

### 2.1.4 Administrative / Ecosystem Port Separates Administrative Inputs

`Administrative / Ecosystem Port` is introduced for:

1. policy bundles,
2. metadata directives,
3. operational directives,
4. ecosystem-level control payloads.

This prevents administrative/control payloads from entering through the consuming content ingress path.

---

### 2.1.5 Registry / Delivery Port Separates Outbound Registry Contracts

`Registry / Delivery Port` is introduced as the outbound registry-facing contract surface used by the `Dispatcher`.

The `Dispatcher` remains the actor.

The port is the contract execution surface.

---

### 2.1.6 Queue Runtime Layer Is Not a Delegated Entity

`Queue Runtime Layer` remains a classification layer only.

Actual runtime responsibilities are delegated to:

1. `Queue Bundle Agent`,
2. `Single Queue Moderator`,
3. `Writing / Feeding Round Manager`,
4. `Dispatching / Delivery Round Manager`.

---

### 2.1.7 Delivery-Cycle Flags Are Not Detailed Feedback Reports

The release preserves a strict distinction:

```text
delivery-cycle flags = state updates
detailed dispatcher feedback reports = diagnostic / reporting artifacts
```

Both are Dispatcher-related, but they require different enablement surfaces.

---

### 2.1.8 Timestamp Stabilizer Is Mature

The old timestamp placeholder has been resolved.

`Timestamp Stabilizer` is now treated as a mature component participating in every accepted ingress path.

No additional placeholder entity is introduced.

---

### 2.1.9 Queue Equals Container

The runtime model preserves the corrected law:

```text
Queue = Container
```

There is no separate runtime container above the queue.

All ownership, occupancy, and slot lifecycle wording must use `Queue Container`.

## 3.1 Core Concepts

### 3.1.1 Responsibility

A responsibility defines what must be achieved by the system or one of its architectural entities.

It is not necessarily a method or a code function.

---

### 3.1.2 Obligation

An obligation defines what must be guaranteed, constrained, or prevented while fulfilling a responsibility.

A responsibility answers:

```text
What must be achieved?
```

An obligation answers:

```text
What must be guaranteed while achieving it?
```

---

### 3.1.3 Delegated Entity

A delegated entity is the architectural entity that practically carries the responsibility.

The Assembler Subsystem may remain accountable at the system level, while a narrower entity performs the responsibility.

---

### 3.1.4 Supporting Entity

A supporting entity participates in fulfilling the responsibility but is not the primary carrier.

---

### 3.1.5 Governed Object

A governed object is the object, carrier, state structure, or artifact affected by the responsibility.

Examples include:

1. `Policy Object`,
2. `Log Level Envelope`,
3. `Queue Container`,
4. `Slot`,
5. `Envelope Slot Occupancy`,
6. `Delivery-Cycle Flag Set`,
7. `Dispatcher Feedback Report`.

---

### 3.1.6 Enablement

Enablement is the required capability, surface, or mechanism that allows the delegated entity to carry the responsibility.

---

### 3.1.7 Surface Type

Surface type classifies the level of interaction:

1. `Administrative`,
2. `Managerial`,
3. `Consuming`,
4. `Boundary`,
5. `Contractual`,
6. `Observability`.
