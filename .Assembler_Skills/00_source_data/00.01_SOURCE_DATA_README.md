# 1. Purpose

## 1.1

The purpose of this document is to explain how canonical source data is packaged inside the Assembler Skills package.

## 1.2

The Assembler Skills package must be portable across:

1. Codex,
2. Kilo Code,
3. OpenCode,
4. GitHub,
5. Google Drive,
6. Local Runtime,
7. future runtime environments.

## 1.3

To achieve this portability, the package must not rely on external workspace-specific folder paths.

## 1.4

Instead, all required canonical source data must be vendored, snapshotted, mirrored, or exported into:

```text id="hl5ag8"
08_assembler_skills/00_source_data/
```

## 1.5

The Source Data Snapshot layer is now the first layer of the skills architecture.

## 1.6

The governing architecture law is:

```text id="z42gus"
Source data is vendored.
Kernel governs.
Core reasons.
CLI expresses.
Adapter executes.
Runtime hosts.
```

# 2. Why Source Data Is Vendored

## 2.1

The broader Assembler Subsystem workspace contains source material distributed across multiple top-level directories.

## 2.2

The subsystem tree includes source directories such as:

```text id="knf0vc"
00.01_Architectural_Taxonomy/
00.02_Assembler_Canonical_Responsibilities/
00.03_Canonical_Delegation_Enablement_Matrix_V2/
```

The current subsystem tree confirms that these are separate source folders, with taxonomy, canonical responsibilities, and delegation/enablement release files stored under different branches. 

## 2.3

That layout is valid for the broader subsystem workspace.

## 2.4

However, a skill package must not require that exact workspace layout to exist.

## 2.5

If the skill package depended directly on sibling paths such as:

```text id="fqb6as"
../00.03_Canonical_Delegation_Enablement_Matrix_V2/
```

then the skill would fail or drift in environments where:

1. only `08_assembler_skills/` is installed,
2. source folders have different casing,
3. Google Drive exports use different folder names,
4. repository paths differ,
5. a runtime adapter works from a temporary workspace,
6. a packaged distribution includes only the skill files.

## 2.6

Vendoring source data solves this by making package `08` self-contained with respect to canonical evidence.

# 3. What Source Data Means Here

## 3.1 Definition

### 3.1.1

Source data means canonical source material that the skills must read, validate, query, trace, or update through controlled workflows.

### 3.1.2

It includes:

1. taxonomy source files,
2. responsibility and obligation source files,
3. delegation and enablement source files,
4. entity and enablement indexes,
5. delegation matrices,
6. obligation matrices,
7. trace coverage registers,
8. rollup indexes,
9. join-key references,
10. query pattern references,
11. schema files.

## 3.2 Source Data Is Not New Semantics

### 3.2.1

The vendored source snapshot does not create new canonical semantics.

### 3.2.2

It is a local operating copy of upstream canonical material.

### 3.2.3

The skill may read it, validate it, query it, or propose controlled updates to it.

### 3.2.4

The skill must not invent source facts that are not present or formally proposed through an editor workflow.

# 11. Snapshot vs Authority

## 11.1 Snapshot Definition

The vendored source data is the local operating snapshot used by the skills.

## 11.2 Authority Definition

Canonical authority remains with the governed source content represented by the snapshot.

## 11.3 Practical Rule

The skill reads the snapshot as local evidence.

## 11.4 Staleness Rule

If the snapshot is stale, the correct behavior is to:

1. report staleness,
2. run a controlled refresh workflow,
3. record the refresh,
4. validate the refreshed content,
5. proceed only after source-data status is clear.

## 11.5 No Inference Rule

If a source fact is missing, the skill must not invent it.

It must instead return:

```text id="wa920r"
missing_source_evidence
```

or produce a controlled editor proposal.
