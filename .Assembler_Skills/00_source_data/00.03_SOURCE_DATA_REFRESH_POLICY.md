# 1. Purpose

## 1.1

This document defines how vendored canonical source data may be refreshed inside the Assembler Skills package.

## 1.2

The refresh policy exists because the skills package must be portable and must not depend on hardcoded external workspace paths.

## 1.3

A refresh operation may read from upstream sources, repositories, Drive exports, generated exports, or local files.

## 1.4

However, after refresh, normal skill operation must use only the vendored source-data snapshot under:

```text id="n8zsgu"
08_assembler_skills/00_source_data/
```

## 1.5

This policy governs refresh behavior for:

1. architectural taxonomy source data,
2. canonical responsibility and obligation source data,
3. delegation and enablement matrix source data,
4. source-specific manifests,
5. source-data file map,
6. source-data validation records.

---

# 2. Core Policy Statement

## 2.1

Refreshing source data is a controlled maintenance operation.

## 2.2

It is not the same as normal query execution.

## 2.3

It is not the same as free-form content editing.

## 2.4

It is not a license for adapters to choose semantic sources.

## 2.5

The governing rule is:

```text id="xzj169"
External sources may be read during refresh.
Runtime skill operation must use vendored source data.
```

## 2.6

The package-level operating law remains:

```text id="x1g0va"
Source data is vendored.
Kernel governs.
Core reasons.
CLI expresses.
Adapter executes.
Runtime hosts.
```

# 3. Refresh Authority

## 3.1 Who May Request Refresh

A refresh may be requested by:

1. a human maintainer,
2. the Content Editor Skill,
3. a source-data maintenance workflow,
4. an approved adapter operating in editor or maintenance mode.

## 3.2 Who May Not Request Refresh

A refresh must not be initiated by:

1. the Query Expert Skill during normal query execution,
2. an adapter acting without a declared refresh command,
3. a runtime environment implicitly,
4. a missing-file fallback routine,
5. a query answer generation routine.

## 3.3 Refresh Is Not Query

The Query Expert Skill must operate in read-only mode.

If source data is missing or stale, the query skill must report:

```text id="x9qr1h"
source_data_missing
```

or:

```text id="ca8nnk"
source_data_stale
```

It must not refresh or repair the snapshot silently.

# 5. Allowed Refresh Modes

## 5.1 `manual_copy`

### 5.1.1 Definition

A human maintainer copies source files into the vendored source-data tree.

### 5.1.2 Required Record

The refresh record must include:

1. copied files,
2. source origin,
3. copied timestamp if known,
4. reason,
5. validation result.

### 5.1.3 Use Case

Appropriate for small controlled updates.

---

## 5.2 `drive_export`

### 5.2.1 Definition

A Google Drive, Docs, Sheets, or Slides source is exported into Markdown, CSV, JSON, or another canonical local format.

### 5.2.2 Required Record

The refresh record must include:

1. Drive file title,
2. Drive file ID or URL if available,
3. export MIME type,
4. exported file path,
5. export timestamp if available,
6. validation result,
7. known conversion limitations.

### 5.2.3 Constraint

After export, skill operation must use the exported vendored file, not a live Drive path.

---

## 5.3 `git_checkout_copy`

### 5.3.1 Definition

Source data is copied from a repository checkout into `00_source_data/`.

### 5.3.2 Required Record

The refresh record must include:

1. repository name if available,
2. branch or commit if available,
3. copied paths,
4. changed files,
5. validation result.

### 5.3.3 Constraint

The repository path is origin metadata, not a runtime dependency.

---

## 5.4 `repository_subtree_snapshot`

### 5.4.1 Definition

A repository subtree is snapshotted into the source-data root.

### 5.4.2 Required Record

The refresh record must include:

1. subtree source,
2. snapshot root,
3. included files,
4. excluded files,
5. commit or revision if available,
6. validation result.

### 5.4.3 Use Case

Appropriate when the upstream source is maintained in a repository.

---

## 5.5 `generated_export`

### 5.5.1 Definition

Source data is generated from another canonical format, such as exporting CSV indexes from a canonical spreadsheet or JSON indexes from a maintained source.

### 5.5.2 Required Record

The refresh record must include:

1. generator name,
2. generator version if available,
3. input source,
4. output files,
5. transformation rules,
6. validation result.

### 5.5.3 Constraint

Generated files must preserve canonical IDs and row grains.

---

## 5.6 `read_only_live_inspection_with_snapshot_output`

### 5.6.1 Definition

A live source is read for inspection, but the result is recorded as a snapshot artifact under `00_source_data/`.

### 5.6.2 Required Record

The refresh record must include:

1. live source inspected,
2. exact exported or captured output,
3. snapshot target,
4. limitations,
5. validation result.

### 5.6.3 Constraint

Live inspection must not become a runtime dependency for normal skill operation.

# 8. Validation After Refresh

## 8.1 Required Validation Classes

After refresh, validation must check:

1. source root existence,
2. source group completeness,
3. source-specific manifest presence,
4. file map coverage,
5. canonical ID grammar,
6. expected file count where known,
7. required CSV headers where known,
8. JSON parseability,
9. Markdown presence for reference documents,
10. anti-collapse rules,
11. rollup no-new-semantics rule,
12. query-readiness,
13. editor-propagation readiness.

## 8.2 Validation Outcomes

Allowed validation outcomes:

| Outcome   | Meaning                                            |
| --------- | -------------------------------------------------- |
| `passed`  | Refresh is valid for operation                     |
| `warning` | Refresh is usable but limitations must be reported |
| `failed`  | Refresh cannot be used for canonical operation     |
| `not_run` | Refresh is incomplete and not release-ready        |

## 8.3 Blocking Failures

Refresh must be blocked if:

1. `SOURCE_DATA_ROOT` is missing,
2. required source group root is missing,
3. source-specific manifest is missing,
4. required rollup files are missing for query skill,
5. external workspace paths are required at runtime,
6. query files are modified during read-only execution,
7. IDs are malformed,
8. anti-collapse guards are weakened,
9. rollup files introduce new semantics,
10. refresh has no record.

# 9. Staleness Detection

## 9.1 Staleness Definition

A source-data snapshot is stale when it is known or suspected to differ from the intended upstream canonical source.

## 9.2 Staleness Signals

Staleness may be indicated by:

1. source-specific manifest timestamp older than expected,
2. upstream revision newer than snapshot revision,
3. missing source files,
4. count mismatches,
5. schema mismatch,
6. query result mismatch,
7. manifest/file-map contradiction,
8. known user-provided update not reflected in source data.

## 9.3 Staleness Status Values

Allowed staleness statuses:

1. `current`,
2. `unknown`,
3. `suspected_stale`,
4. `confirmed_stale`,
5. `refresh_required`,
6. `refresh_blocked`.

## 9.4 Query Behavior on Staleness

If staleness is detected during query mode, the Query Expert Skill must:

1. report staleness,
2. continue only if evidence is still sufficient,
3. mark confidence accordingly,
4. avoid refreshing,
5. avoid inventing missing facts.

## 9.5 Editor Behavior on Staleness

If staleness is detected during editor mode, the Content Editor Skill must:

1. classify whether refresh is required,
2. recommend refresh before patching if needed,
3. avoid generating final edits from stale source data,
4. include staleness in validation notes.

# 10. Adapter Responsibilities

## 10.1 General Adapter Responsibility

Adapters may execute refresh operations only when:

1. the command is explicitly a refresh or editor-maintenance command,
2. the adapter declares the required capability,
3. the source-data manifest permits the source group,
4. the refresh record can be produced,
5. validation can be run or explicitly marked `not_run`.

## 10.2 Codex Adapter

The Codex adapter may support:

1. repository-relative source-data reads,
2. branch-based source-data updates,
3. diff generation,
4. validation command execution,
5. review-ready refresh summaries.

It must not:

1. choose canonical source folders independently,
2. refresh during query commands,
3. commit source-data updates without a refresh record.

## 10.3 Kilo Code Adapter

The Kilo Code adapter may support:

1. workspace discovery,
2. local file export into `00_source_data/`,
3. structured edit sessions,
4. validation feedback.

It must not:

1. treat workspace discovery as semantic authority,
2. infer missing source groups,
3. silently change source-data paths.

## 10.4 OpenCode Adapter

The OpenCode adapter may support:

1. filesystem copy,
2. shell-based validation,
3. local diff,
4. packaging workflows.

It must not:

1. read from external paths during normal query operation,
2. bypass source-data aliases,
3. hide shell or filesystem errors.

## 10.5 GitHub Adapter

The GitHub adapter may support:

1. repository file fetch,
2. branch-based source-data refresh,
3. pull-request refresh workflow,
4. commit comparison,
5. CI validation reporting.

It must not:

1. treat repository default branch as automatically current without manifest check,
2. modify source data on default branch without review workflow,
3. skip validation reporting.

## 10.6 Google Drive Adapter

The Google Drive adapter may support:

1. Drive file search,
2. Docs export,
3. Sheets export,
4. Slides export,
5. raw file download,
6. snapshot output into source data.

It must not:

1. treat live Drive files as runtime source dependencies,
2. silently overwrite vendored files,
3. hide export format limitations,
4. convert Sheets to CSV without recording sheet/range/export details.

## 10.7 Local Runtime Adapter

The Local Runtime adapter may support:

1. local relative file reads,
2. local file writes for editor workflows,
3. local refresh copies,
4. local validation scripts,
5. local artifact export.

It must not:

1. assume machine-specific absolute paths,
2. use drive-letter paths as canonical references,
3. bypass refresh records.

# 14. Refresh and Anti-Collapse Rules

## 14.1 Required Anti-Collapse Checks

After refresh, the following distinctions must remain intact:

1. Metadata Injector is not a policy host.
2. Queue Runtime Layer is classification-only.
3. Delivery-Cycle Flag Set is not Dispatcher Feedback Report.
4. Queue equals Container.
5. Waiting lists store Queue Container Identifiers.
6. Envelope Slot Occupancy is not lifecycle ownership.
7. Registry / Delivery Port does not replace Dispatcher.
8. Typed/template contracts are not runtime COM interfaces.

## 14.2 Blocking Anti-Collapse Failure

Refresh must be considered failed if it weakens any of these distinctions.

## 14.3 Warning Case

If a refreshed source omits an anti-collapse rule that existed previously, the refresh must be marked:

```text id="emv9ts"
validation_status = warning
```

or:

```text id="qgp9ac"
validation_status = failed
```

depending on severity.

# 15. Refresh and Rollup Semantics

## 15.1 Rollup Constraint

Rollup source files must remain navigation and validation artifacts.

## 15.2 Invalid Refresh Result

A refresh is invalid if it causes rollup files to become the only semantic origin for responsibilities, obligations, entities, or gaps.

## 15.3 Required Rule

The rollup layer must preserve:


If a rollup file changes, the editor must determine whether upstream reference, matrix, obligation, or trace files also changed.

A rollup-only semantic change is invalid unless explicitly explained as a derived update.

# 17. Human Review Requirements

## 17.1 Required Human Review

Human review is required when refresh:

1. changes canonical responsibility or obligation files,
2. changes delegation matrix files,
3. changes obligation delegation files,
4. changes trace coverage files,
5. changes rollup indexes,
6. changes anti-collapse guards,
7. removes source files,
8. changes file counts,
9. changes source-specific manifests,
10. changes schemas.

## 17.2 Optional Human Review

Human review may be optional when refresh:

1. only adds non-semantic metadata,
2. updates last-synced fields,
3. refreshes unchanged files,
4. updates checksums without content changes.

## 17.3 Review Output

Human review output should include:

1. refresh summary,
2. changed files,
3. validation result,
4. anti-collapse result,
5. downstream impact,
6. recommendation.

# 18. Relationship to Content Editor Skill

## 18.1 Editor Uses Refresh Policy

The Content Editor Skill must use this policy when a change request requires source-data refresh.

## 18.2 Refresh Before Edit

If the source snapshot is stale, the editor should recommend refresh before generating final patches.

## 18.3 Edit After Refresh

After a successful refresh, the editor may proceed with:

1. change classification,
2. impact analysis,
3. propagation planning,
4. patch generation,

## 18.4 Refresh Is Not Patch

A refresh updates the local source snapshot.

A patch proposes or applies controlled semantic or structural changes.

The editor must not confuse the two.

# 23. Refresh Result Status

## 23.1 Allowed Status Values

| Status                | Meaning                                                       |
| --------------------- | ------------------------------------------------------------- |
| `ready`               | Source data is valid for normal skill operation               |

## 23.2 Runtime Behavior by Status

| --------------------- | -------------------------------------- | ------------------------------- |
| `ready`               | normal                                 | normal                          |
| `ready_with_warnings` | answer with warnings                   | patch with warnings             |
| `partial`             | only evidence-limited queries          | no final patches                |
| `blocked`             | fail with source-data error            | fail with source-data error     |
| `not_validated`       | answer only if user accepts limitation | require validation before patch |

# 24. Final Policy Statement

## 24.1

Source-data refresh is a controlled maintenance workflow.

## 24.2

Normal skill operation must use vendored source data.

## 24.3

External sources are allowed only as refresh inputs or origin metadata.

## 24.4

Adapters execute refresh steps but do not decide canonical semantics.

## 24.5

The Query Expert Skill remains read-only.

## 24.6

The Content Editor Skill may coordinate refresh only through explicit impact, propagation, and validation workflows.

## 24.7

The final rule is:

```text id="nq10db"
Refresh may update the snapshot.
It must not bypass the canon.
```
