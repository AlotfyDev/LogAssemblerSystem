# 1. Query Design Principles

## 1.1 No New Semantics

### 1.1.1

Queries must not invent responsibilities, obligations, entities, governed objects, enablements, contracts, gaps, or doctrines.

### 1.1.2

The rollup query layer only navigates existing canonical records.

## 1.2 Stable Identifier First

### 1.2.1

Every query should prefer stable IDs over names.

### 1.2.2

Examples:

```text
RSP-CAN-027
OBL-CAN-027
E-INT-011
BCO-012
EN-FAM-003
CC-ENV-001
G-004
DOC-006
```

## 1.3 Names Are Secondary

### 1.3.1

Human-readable names may be used for display.

### 1.3.2

They should not be used as the only join basis when an ID exists.

## 1.4 Semicolon Fields Must Be Split

### 1.4.1

CSV multi-value fields use semicolon-separated IDs.

### 1.4.2

For reliable queries, split fields such as:

```text
primary_delegated_entity_ids
supporting_entity_ids
governed_object_ids
enablement_family_ids
surface_ids
contract_trace_ids
gap_trace_ids
doctrine_trace_ids
taxonomy_layer_ids
```

## 1.5 Anti-Collapse Queries Are First-Class

### 1.5.1

This release contains several semantic separation rules.

### 1.5.2

Queries must preserve them, especially:

1. Metadata Injector is not a policy host.
2. Queue Runtime Layer is classification-only.
3. Delivery-Cycle Flag Set is not Dispatcher Feedback Report.
4. Queue equals Container.
5. Waiting lists store Queue Container Identifiers.
6. Envelope Slot Occupancy is not Log Level Envelope lifecycle ownership.
7. Registry / Delivery Port does not replace Dispatcher.
8. Typed/template contracts are not runtime COM interfaces.

# 18. Downstream Query Engine Guidance

## 18.1 CSV Loader Behavior

A future query engine should:

1. parse CSV files with headers,
2. split semicolon-separated fields into arrays,
3. trim whitespace after splitting,
4. preserve exact ID casing,
5. validate all IDs against known prefixes,
6. normalize many-to-many relations into join tables.

## 18.2 JSON Loader Behavior

A future query engine should:

1. preserve nested rollups,
2. index all IDs into lookup maps,
3. expose reverse lookups,
4. validate counts,
5. validate anti-collapse guards,
