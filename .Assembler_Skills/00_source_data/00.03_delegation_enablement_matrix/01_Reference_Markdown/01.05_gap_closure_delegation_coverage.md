# 3. Coverage Method

## 3.1 Closure Coverage Definition

### 3.1.1

A gap is considered covered only when the release records all of the following:

1. `gap_id`,
2. closure statement,
3. covering responsibility IDs,
4. preserving obligation IDs,
5. delegated entity IDs,
6. governed object IDs,
7. enablement family IDs,
8. surface types,
9. contract trace IDs,
10. taxonomy or doctrine trace IDs.

---

## 3.2 Coverage Is Not Mere Mention

### 3.2.1

A gap is not considered covered merely because it is named.

### 3.2.2

Coverage requires executable architectural traceability at the delegation layer.

### 3.2.3

That means each closure must answer:

```text id="xe9i7a"
Which responsibility enforces the closure?
Which entity carries it?
Which object is governed?
Which enablement makes it possible?
Which obligation prevents regression?
```

---

## 3.3 Coverage Levels

| Coverage Level          | Meaning                                                            |
| ----------------------- | ------------------------------------------------------------------ |
| Contract coverage       | Closure is represented in canonical contracts or contract families |
| Responsibility coverage | Closure is represented by one or more RSP-CAN rows                 |
| Obligation coverage     | Closure is protected by one or more OBL-CAN rows                   |
| Delegation coverage     | Closure is assigned to concrete carriers                           |
| Enablement coverage     | Closure requires one or more enablement families                   |
| Doctrine coverage       | Closure preserves a taxonomy or constitutional law                 |
| Query coverage          | Closure can be found through stable join keys                      |

# 4. G-001 — Registry Abstraction

## 4.1 Gap Statement

### 4.1.1

`G-001 Registry Abstraction` addressed the risk that the Assembler could blur its boundary with the `Log Level Records Registry`.

### 4.1.2

The risk included:

1. treating the registry as an assembler-owned internal store,
2. hiding readiness and capacity dependencies,
3. bypassing the Dispatcher boundary,
4. failing to model receiver-side admission separately,
5. retaining lifecycle authority after handoff.

---

## 4.2 Closure Statement

### 4.2.1

The closure establishes that:

1. the `Dispatcher` is the final assembler-side actor,
2. the `Registry / Delivery Port` is the outbound contract surface,
3. the registry exposes only boundary-sufficient receiving access,
4. readiness must be observed before handoff,
5. lifecycle authority transfers only after valid handoff.

### 4.2.2

The corrected reference matrix covers this gap with Dispatcher, Registry / Delivery Port, RSP-CAN-021 through RSP-CAN-024, RSP-CAN-026, and RSP-CAN-030.

1. confusing ecosystem policy source with local policy assignment,
2. treating Policy Object as an executor,
3. bypassing PolicyAssigner,
4. treating Metadata Injector as a policy host,

5. `Metadata Injector` is excluded because metadata is schema/directive-governed.

### 5.2.2

The corrected delegation matrix explicitly identifies included policy-consuming components and excludes `Metadata Injector` because it is schema/directive-governed, not policy-hosted.

## 10.2 Closure Statement
