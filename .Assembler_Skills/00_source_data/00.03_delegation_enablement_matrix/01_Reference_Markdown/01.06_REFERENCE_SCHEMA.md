````markdown id="q8m1x6"
# Reference Schema
## Canonical Logical Schema for the Delegation, Enablement, Obligation, and Trace Release

---

# 0. Document Purpose

## 0.1
This document defines the canonical **logical reference schema** for the **Canonical Delegation & Enablement Matrix V2 Release**.

## 0.2
It is not a final database schema.

## 0.3
It is not a PostgreSQL derivation.

## 0.4
It is not a runtime implementation interface.

## 0.5
It defines the stable schema vocabulary, entity classes, logical records, join keys, constraints, and validation expectations required to keep all Markdown, CSV, JSON, and downstream rollup artifacts aligned.

## 0.6
The release itself is explicitly designed around mapping:

```text id="qyiqzt"
Responsibility / Obligation
→ Primary Delegated Entity
→ Supporting Entities
→ Governed Objects
→ Enablement Family
→ Surface Type
→ Trace
````

and the release README fixes these as the core mapping dimensions and query axes. 

---

# 1. Schema Scope

## 1.1 In Scope

### 1.1.1

This reference schema covers the following release domains:

1. release identity,
2. responsibility families,
3. responsibilities,
4. obligations,
5. sub-responsibilities,
6. delegated entities,
7. supporting entities,
8. governed objects,
9. enablement families,
10. surface types,
11. contract traces,
12. gap closure traces,
13. taxonomy and doctrine traces,
14. entity-to-responsibility load,
15. obligation delegation,
16. flag and feedback-report sub-delegation,
17. rollup and query indexes.

---

## 1.2 Out of Scope

### 1.2.1

This reference schema does not define:

1. final code packages,
2. final classes,
3. final method signatures,
4. runtime scheduling algorithms,
5. PostgreSQL physical tables,
6. indexes or materialized views,
7. deployment topology,
8. storage engine choices.

### 1.2.2

Those belong to downstream derivation work.

### 1.2.3

The assignment principles explicitly state that the current release focuses on `Responsibility / Obligation → Delegation → Enablement` and does not finalize implementation packages, classes, function signatures, database tables, or runtime scheduling algorithms.

# 2. Schema Design Principles

## 2.1 Stable Identity Before Representation

### 2.1.1

Every architectural record must have a stable identifier before it is represented in Markdown, CSV, JSON, or downstream tables.

### 2.1.2

Display names may change more easily than IDs.

### 2.1.3

IDs must remain canonical unless a formal migration is declared.

---

## 2.2 Logical Schema Before Physical Schema

### 2.2.1

This file defines logical records.

### 2.2.2

Logical records may later derive into:

1. CSV rows,
2. JSON objects,
3. graph nodes,
4. PostgreSQL tables,
5. validation reports,
6. query indexes.

### 2.2.3

No one-to-one mapping to final tables is assumed here.

---

## 2.3 Actor/Object Separation

### 2.3.1

Delegated entities and governed objects must remain distinct.

### 2.3.2

A governed object must not become an actor unless explicitly promoted by a later taxonomy revision.

### 2.3.3

The assignment principles define a governed object as the carrier, state structure, contract artifact, or runtime artifact affected by a responsibility, and explicitly state that it is not automatically an actor. 

---

## 2.4 Accountability/Execution Separation

### 2.4.1

`Assembler Subsystem` may remain accountable while execution is delegated to narrower mature architectural carriers.

### 2.4.2

This schema must therefore allow:

1. system accountability owner,
2. primary delegated entity,
3. supporting entities,
4. governed objects.

### 2.4.3

The assignment principles fix the separation as:

```text id="kpijms"
Assembler Subsystem = accountable
Narrow mature architectural entity = delegated carrier
```



---

## 2.5 Closure-Aware Traceability

### 2.5.1

A release record is incomplete if it cannot trace to relevant closure, contract, or taxonomy sources where applicable.

### 2.5.2

The gap closure coverage reference states that a gap is considered covered only when the release records gap ID, closure statement, responsibilities, obligations, delegated entities, governed objects, enablement families, surface types, contract traces, and taxonomy or doctrine traces. 

---

## 2.6 Queryability

### 2.6.1

Every schema class must support stable query joins.

### 2.6.2

The release README identifies the canonical join keys as:

```text id="gn07op"
responsibility_family_id

```

# 3. Canonical Identifier Grammar

## 3.1 Identifier Requirements

### 3.1.1

Every canonical identifier must be:

1. stable,
2. unique within its namespace,
3. prefix-qualified,
4. human-readable enough for trace review,
5. suitable as a CSV key,
6. suitable as a JSON string key,
7. suitable as a future relational primary or foreign key.

---

## 3.3 Canonical Casing Rules

### 3.3.1

Identifiers must use uppercase namespace prefixes.

### 3.3.2

6. If a responsibility covers a gap, `gap_ids` must not be empty.
7. If a responsibility enforces doctrine, `doctrine_trace_ids` must not be empty.

## 4.4 Obligation Record

### 4.4.1 Purpose

An obligation record states what must be guaranteed, constrained, or prevented while fulfilling a responsibility.

### 4.4.2 Logical Fields

| Field                    |    Required | Type   | Description                                                                  |
| ------------------------ | ----------: | ------ | ---------------------------------------------------------------------------- |
| `obligation_id`          |         yes | string | `OBL-CAN-###`                                                                |
| `responsibility_id`      |         yes | string | Associated responsibility                                                    |
| `obligation_statement`   |         yes | string | Required guarantee/prohibition                                               |
| `obligation_type`        |         yes | enum   | guarantee, prohibition, boundary, doctrine, lifecycle, readiness, separation |
| `enforced_by_entity_ids` |         yes | array  | Entity or entities enforcing it                                              |
| `governed_object_ids`    | conditional | array  | Objects constrained by obligation                                            |
| `surface_types`          |         yes | array  | Surface level                                                                |
| `enablement_family_ids`  | conditional | array  | Enablements preserving it                                                    |
| `gap_ids`                | conditional | array  | Gaps protected by obligation                                                 |
| `anti_collapse_rule`     | conditional | string | False collapse prevented                                                     |
| `notes`                  |          no | string | Explanation                                                                  |

### 4.4.3 Validation Rules

1. Every obligation must map to at least one responsibility.
2. Every obligation must identify its enforcing entity or boundary.
3. Every obligation that preserves a closure must include `gap_ids`.
4. Every obligation that prevents role confusion must include `anti_collapse_rule`.

### 4.4.4 Concept Rule

An obligation answers what must be guaranteed while achieving the responsibility; the assignment principles define obligations as constraints, guarantees, prohibitions, or conditions attached to responsibilities.

## 7.2 Required Join Validation

### 7.2.1

Every responsibility must join to:

1. one responsibility family,
2. at least one delegated entity,
3. at least one governed object or governed boundary artifact,
4. at least one enablement family,
5. at least one surface type,
6. at least one obligation.

---

## 7.3 Entity Classification Validation

### 7.3.1

A record with `delegation_status = classification_only` must not appear as `primary_delegated_entity_id`.

### 7.3.2

A record with `actor_status = not_actor` must not appear as an entity carrying execution.

### 7.3.3

A governed object may appear in `governed_object_ids`, not in `primary_delegated_entity_ids`.

---

## 7.4 Policy Host Validation

### 7.4.1

`Metadata Injector` must not be listed as a policy-consuming component.

### 7.4.2

`Metadata Injector` must not be assigned `EN-FAM-002`.

### 7.4.3

`Metadata Injector` may use:

1. `EN-FAM-009`,
2. relevant administrative/contract surface through `Administrative / Ecosystem Port`,
3. envelope composition support through `Envelope Assembler`.

---

## 7.5 Queue-as-Container Validation

### 7.5.1

No schema record may introduce a separate runtime container above `Queue Container`.

### 7.5.2

Waiting lists must store queue-container identifiers, not envelopes.

### 7.5.3

The assignment principles state that `Queue = Container`, there is no separate runtime container above the queue, and Round Manager waiting lists store queue-container identifiers rather than envelopes. 

---

## 7.6 Dispatcher Boundary Validation

### 7.6.1

Registry handoff responsibilities must preserve `Dispatcher` as the final assembler-side actor.

### 7.6.2

`Registry / Delivery Port` may be a contract surface, but must not replace `Dispatcher` as actor.

### 7.6.3

After valid handoff, lifecycle authority must transfer to the `Log Level Records Registry`.

---

## 7.7 Flag/Report Separation Validation

### 7.7.1

`Delivery-Cycle Flag Set` and `Dispatcher Feedback Report` must remain separate governed artifacts.

### 7.7.2

`RSP-CAN-025` must govern delivery-cycle flag updates.

### 7.7.3

`RSP-CAN-048` must govern detailed dispatcher feedback reports.

### 7.7.4

The assignment principles define the canonical distinction:

```text id="er82kw"
Delivery-cycle flags = state updates
Detailed dispatcher feedback reports = diagnostic / reporting artifacts
```



---

## 7.8 Ingress Variant Validation

### 7.8.1

Validating ingress and hot-path ingress must remain distinct.

### 7.8.2

`Validator` must participate only in validating ingress.

### 7.8.3

`Timestamp Stabilizer` must participate in every accepted ingress path.

### 7.8.4

Accepted content must be transformed into `Log Level Envelope`.
