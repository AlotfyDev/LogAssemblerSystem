
# Project Overview

## 1. Document Purpose

This document provides the architectural overview of the **Assembler System**.

It explains the project identity, architectural position, subsystem boundary, internal and external context, responsibility model, runtime coordination model, policy/governance model, and C++ implementation direction.

This document is broader than `README.md`.

The README introduces the system as an architectural landing page.  
This Project Overview explains how the major concepts relate to each other before the documentation proceeds into `VisionAndScope.md`, `SystemGoals.md`, `RoadMap.md`, and `Glossary.md`.

This document does not define final implementation packages, final C++ classes, final runtime algorithms, final database tables, PostgreSQL derivation, or deployment topology.

Those are later derivation targets.

---

## 2. Project Identity

Assembler System is a bounded C++ write-side preparation and dispatch subsystem responsible for transforming governed incoming content into authoritative **Log Level Envelopes** and handing them off through a dispatcher-mediated boundary toward downstream preservation and query destinations.

The system exists at the assembler-subsystem level.

It is not the full logging system.

It is not the full messaging system.

It is not the downstream registry.

It is not the storage engine.

It is not the query engine.

It is not the post-handoff lifecycle authority.

Its identity is defined by a narrow but strict preparation and dispatch responsibility.

The system receives or participates in governed incoming content paths, prepares content into typed envelope carriers, coordinates queue/container runtime state, and hands off prepared envelopes through the final assembler-side dispatcher boundary.

---

## 3. Architectural Problem

Incoming content may otherwise reach preservation, logging, or query-facing systems as informal payloads.

Assembler System prevents this.

The project introduces a bounded write-side preparation layer that ensures incoming material is not treated as downstream-ready until it has passed through assembler-governed preparation semantics.

The architectural problem can be stated as follows:

```text
How can incoming content be transformed into a typed, traceable, policy-aware,
metadata-prepared, timestamp-stabilized, dispatch-ready envelope before it is
allowed to cross into downstream preservation or query destinations?
````

Assembler System solves this problem by enforcing:

1. typed content admission,
2. contract discipline,
3. envelope realization,
4. metadata preparation,
5. timestamp stabilization,
6. policy assignment,
7. queue/container runtime coordination,
8. dispatcher-mediated handoff,
9. post-handoff authority exclusion.

The system is therefore not a message mover.

It is a preparation and dispatch authority with strict boundary rules.

---

## 4. System Position

Assembler System sits between consuming-side input paths and downstream preservation/query destinations.

The high-level position is:

```text
External Consuming / Administrative / Operational Context
        ↓
Assembler-Governed Boundary
        ↓
Envelope Assembly and Preparation
        ↓
Queue-as-Container Runtime Coordination
        ↓
Dispatcher-Mediated Handoff
        ↓
Registry / Delivery Port
        ↓
Downstream Preservation / Query Destination
```

The downstream destination may include the **Log Level Records Registry**.

However, the registry is not owned by the Assembler System.

The registry is treated as a downstream storage-side receiving subsystem.

The reading/querying world is also outside assembler authority.

Assembler System prepares and dispatches.

It does not own downstream persistence, indexing, querying, replay, mutation, or record lifecycle.

---

## 5. Boundary Overview

The Assembler boundary begins where governed assembler-facing interaction reaches the subsystem.

That interaction may come from:

1. consuming-side content paths,
2. administrative or ecosystem sources,
3. policy sources,
4. metadata directive sources,
5. operational directive sources,
6. downstream registry-facing delivery interaction.

The boundary ends at valid dispatcher-mediated handoff.

After valid handoff, assembler authority closes.

The Assembler must not claim lifecycle ownership over downstream registry-side records after this point.

The boundary is protected by explicit separation between:

1. consuming ingress,
2. administrative/ecosystem input,
3. operational/runtime coordination,
4. outbound registry/delivery interaction,
5. downstream preservation/query authority.

## 6. External Architectural Context

The external architectural entities are parties outside Assembler ownership that interact with, govern, receive from, or are explicitly excluded from the Assembler subsystem.

The canonical external context includes:

1. `Level API`
2. `Assemblers Administrator`
3. `Policy Registry / Policy Source`
4. `Metadata Directive Source`
5. `Scaling / Operational Directive Source`
6. `Log Level Records Registry`
7. `Reading / Querying Subsystem`

These external entities are not internal assembler actors.

They define the surrounding environment in which the Assembler operates.

### 6.1 Level API

The `Level API` is the consuming-side façade through which typed content entry is initiated.

It participates in the consuming path and may contain or inject the `Envelope Assembler`.

### 6.2 Assemblers Administrator

The `Assemblers Administrator` represents administrative governance and control input.

It is not a normal content source.

### 6.3 Policy Registry / Policy Source

The `Policy Registry / Policy Source` provides policy material.

It is external to the assembler runtime and supplies policy objects or policy-governance input.

### 6.4 Metadata Directive Source

The `Metadata Directive Source` supplies schema-guided or directive-driven metadata input.

This source is distinct from policy execution.

### 6.5 Scaling / Operational Directive Source

The `Scaling / Operational Directive Source` provides operational directives such as pause, drain, scale, activate, or related runtime control signals.

### 6.6 Log Level Records Registry

The `Log Level Records Registry` is a downstream stable receiving subsystem.

It may receive prepared envelopes or record material after dispatcher-mediated handoff.

It is not an internal assembler component.

### 6.7 Reading / Querying Subsystem

The `Reading / Querying Subsystem` is explicitly outside assembler authority.

Assembler System does not own read/query behavior.

## 7. Internal Architectural Overview

Assembler System contains internal responsibility carriers, boundary ports, coordination actors, and preparation components.

The canonical internal model distinguishes between:

1. subsystem accountability,
2. delegated execution,
3. governed carrier objects,
4. runtime state objects,
5. contract surfaces,
6. enablement families.

The Assembler Subsystem remains the system-level accountability owner.

However, execution is delegated to the narrowest mature architectural carrier.

The governing rule is:

```text
Assembler Subsystem = accountable
Component / Port / Contract Object = delegated carrier
```

This prevents broad, vague responsibility assignment to the whole subsystem.

---

## 8. Core Internal Entities

The internal architectural model includes the following responsibility carriers and boundary entities.

### 8.1 Assembler Subsystem

The `Assembler Subsystem` is the containing subsystem.

It is accountable for assembler-side preparation and dispatch responsibilities.

It does not personally perform every operation.

Its responsibilities are delegated to narrower components, ports, or contract objects.

### 8.2 Envelope Assembler

The `Envelope Assembler` is a client-side composed contract object associated with the consuming entry path.

It composes:

1. `Validator`,
2. `Metadata Injector`,
3. `Timestamp Stabilizer`,
4. envelope assembly routine.

It is injected into, or composed inside, the `Log Level API`.

Its output is the `Log Level Envelope`.

### 8.3 Assembler Ingress Port

The `Assembler Ingress Port` is the server-side receiving contract port.

It receives the `Log Level Envelope` and admits it into a suitable `Slot` inside the target `Queue Container`.

It does not replace the Envelope Assembler.

It does not receive administrative payloads.

It does not act as registry delivery output.

### 8.4 Administrative / Ecosystem Port

The `Administrative / Ecosystem Port` receives administrative and ecosystem-level payloads, including:

1. policy bundles,
2. metadata directives,
3. operational directives,
4. ecosystem-level control/configuration.

It prevents administrative payloads from entering through the consuming ingress path.

### 8.5 PolicyAssigner

The `PolicyAssigner` mediates assignment of `Policy Objects` to policy-consuming components.

It preserves the difference between:

1. policy object,
2. policy assignment,
3. policy-consuming component,
4. mechanism hosting,
5. policy-governed behavior.

### 8.6 Metadata Injector

The `Metadata Injector` applies metadata directives.

It is schema/directive-governed.

It is not a policy host.

This distinction is important because metadata preparation must not be collapsed into policy mechanism hosting.

### 8.7 Timestamp Stabilizer

The `Timestamp Stabilizer` stabilizes timestamp semantics in accepted ingress paths.

It is a mature preparation component.

It participates in all accepted ingress contract families.

### 8.8 Validator

The `Validator` participates in validating ingress paths.

It is not necessarily used in hot-path flows where validation has been externalized.

The architecture preserves the distinction between validating and hot-path ingress.

### 8.9 Queue Bundle Agent

The `Queue Bundle Agent` orchestrates queue bundles and waiting-list discipline.

It is not the same as a single queue moderator.

It works at the queue-bundle coordination level.

### 8.10 Single Queue Moderator

The `Single Queue Moderator` moderates one queue-container.

It preserves queue-container occupancy and lifecycle distinctions.

It must not confuse slot occupancy with envelope lifecycle ownership.

### 8.11 Writing / Feeding Round Manager

The `Writing / Feeding Round Manager` manages writing/feed-domain round progression.

It interacts with queue-container identifiers and waiting-list state.

### 8.12 Dispatching / Delivery Round Manager

The `Dispatching / Delivery Round Manager` manages dispatch/delivery-domain round progression.

It participates in moving envelopes toward dispatch readiness.

### 8.13 Dispatcher

The `Dispatcher` is the final assembler-side handoff actor.

It performs final outbound handoff.

It must not be bypassed.

### 8.14 Registry / Delivery Port

The `Registry / Delivery Port` is the outbound registry-facing contract surface used by the Dispatcher.

The Dispatcher remains the actor.

The port is the contract execution surface.

## 9. Canonical Responsibility Chain

Assembler System uses a formal derivation chain:

```text
External Relation
→ Contract
→ Responsibility
→ Obligation
→ Delegation
→ Enablement
→ Capability Candidate
```

This chain is the basis for moving from architectural knowledge to implementation-facing design.

### 9.1 External Relation

An external relation identifies a boundary interaction involving the Assembler.

It answers:

```text
Who or what interacts with the Assembler boundary?
```

### 9.2 Contract

A contract defines the typed law governing the interaction.

Contracts are architectural laws before they become C++ templates, traits, static checks, concepts, or validation mechanisms.

### 9.3 Responsibility

A responsibility states what must be achieved.

It defines the required architectural outcome.

### 9.4 Obligation

An obligation states what must be guaranteed, constrained, or prevented while fulfilling a responsibility.

It makes the responsibility reviewable and testable.

### 9.5 Delegation

Delegation identifies which architectural entity carries the responsibility.

Delegation must target the narrowest mature carrier.

### 9.6 Enablement

Enablement identifies the required capability, surface, or mechanism that allows the delegated entity to carry the responsibility.

Enablement is not final implementation.

### 9.7 Capability Candidate

A capability candidate is an implementation-facing possibility derived from responsibility, obligation, delegation, and enablement.

It is not automatically a final class, function, package, or runtime algorithm.

## 10. Architectural Taxonomy Overview

The current architecture is best understood as a taxonomy, not as a set of premature implementation classes.

The unified taxonomy has seven major layers:

1. `L-001 External Architectural Entities`
2. `L-002 Core Internal Architectural Entities`
3. `L-003 Runtime / State Entities`
4. `L-004 Boundary / Carrier / Runtime Objects`
5. `L-005 Contract / Template Families`
6. `L-006 Doctrine / Constitutional Rules`
7. `L-007 Relation Families`

This layered taxonomy prevents category collapse.

It avoids treating ports as actors, carriers as actors, runtime state as lifecycle ownership, policy objects as executors, or registry boundary surfaces as downstream registry internals.

## 11. Contract Family Overview

The architecture is governed by canonical contract/template families.

These families are architectural contract groups before they are implementation interfaces.

The main contract families are:

1. `Ingress Contract Family`
2. `Policy / Ecosystem Contract Family`
3. `Metadata / Preparation Contract Family`
4. `Operational / Scaling Contract Family`
5. `Registry / Dispatcher Contract Family`
6. `Envelope Carrier Contract Family`
7. `Exclusion Contract Family`
8. `Runtime State Contract Family`

### 11.1 Ingress Contract Family

Governs typed content admission from the consuming side.

It distinguishes validating and hot-path ingress variants.

### 11.2 Policy / Ecosystem Contract Family

Governs policy provisioning, policy assignment, and policy-hosting semantics.

### 11.3 Metadata / Preparation Contract Family

Governs metadata directive application and preparation semantics.

### 11.4 Operational / Scaling Contract Family

Governs runtime operational directives and queue realization.

### 11.5 Registry / Dispatcher Contract Family

Governs registry readiness, handoff eligibility, receiving admission, and dispatcher boundary behavior.

### 11.6 Envelope Carrier Contract Family

Governs envelope assembly, envelope identity, envelope movement, and lifecycle transfer.

### 11.7 Exclusion Contract Family

Governs explicit non-authority boundaries.

It prevents read/query, registry internals, and downstream lifecycle from becoming assembler responsibilities.

### 11.8 Runtime State Contract Family

Governs queue, slot, occupancy, waiting-list, lifecycle, and runtime state semantics.

## 12. Log Level Envelope Overview

The `Log Level Envelope` is the authoritative assembler-side runtime transport carrier.

It is the central carrier produced after typed content admission and preparation.

The official documentation term is:

```text
Log Level Envelope
```

Some source references may use:

```text
Log Envelope
```

Unless a later distinction is introduced, `Log Envelope` should be treated as a source-data/legacy term referring to the same carrier concept.

A Log Level Envelope is not raw content.

A Log Level Envelope is not an actor.

A Log Level Envelope is not a downstream registry record.

A Log Level Envelope is a governed carrier object.

It may carry or bind:

1. accepted typed content,
2. envelope identity,
3. metadata,
4. stabilized timestamp,
5. policy assignment,
6. queue slot occupancy relation,
7. round progression state,
8. dispatch eligibility state,
9. handoff trace material.

The envelope must remain the runtime transport unit.

Raw content must not replace it.

## 13. Ingress and Preparation Overview

The corrected ingress flow is:

```text
Log Level API
  contains / injects
    Envelope Assembler
      composes Validator, Metadata Injector, Timestamp Stabilizer
      produces Log Level Envelope

Assembler Ingress Port
  receives Log Level Envelope
  admits it into suitable Slot in target Queue Container
```

This means the consuming-side path begins with typed content through the `Log Level API`.

The `Envelope Assembler` is responsible for producing the `Log Level Envelope`.

The `Assembler Ingress Port` is not the object that assembles the envelope.

It is the receiving contract port that admits the prepared envelope into queue/container runtime state.

This sequence preserves the separation between:

1. typed content entry,
2. validation participation,
3. metadata preparation,
4. timestamp stabilization,
5. envelope realization,
6. server-side envelope admission,
7. queue/container slot placement.

## 14. Runtime Queue / Container Overview

The runtime model is governed by the correction:

```text
Queue = Container
```

A queue is not inside a separate runtime container.

A queue is itself the container that holds slots.

This rule affects all runtime terminology.

The runtime model includes:

1. `Queue Bundle`
2. `Queue Container`
3. `Single Queue Container`
4. `Pending / Waiting Queue Container`
5. `Slot`
6. `Envelope Slot Occupancy`
7. `Slot Lifecycle State`
8. `Writing Round Manager Waiting List`
9. `Delivery Round Manager Waiting List`
10. `Queue Container Identifier`

The queue model is not a broker model.

It is a bounded assembler-side runtime coordination model.

### 14.1 Queue Bundle

A queue bundle groups multiple queue-containers.

It is coordinated by the `Queue Bundle Agent`.

### 14.2 Queue Container

A queue container is a queue that contains slots.

It is the core runtime placement structure.

### 14.3 Slot

A slot is a position inside a queue-container.

### 14.4 Envelope Slot Occupancy

Envelope slot occupancy binds a Log Level Envelope to a slot.

This does not imply lifecycle ownership over the envelope.

### 14.5 Waiting Lists

Round-manager waiting lists carry queue-container identifiers.

They do not store envelopes as generic message payloads.

## 15. Policy and Governance Overview

The policy model preserves a strict distinction between policy material and behavior execution.

A `Policy Object` is a semantic governor.

It does not execute behavior by itself.

Policy-governed behavior is realized locally by policy-consuming components.

Policy-consuming components may include:

1. `Dispatcher`,
2. `Validator`,
3. `Timestamp Stabilizer`,
4. `Queue Bundle Agent`,
5. `Single Queue Moderator`,
6. `Writing / Feeding Round Manager`,
7. `Dispatching / Delivery Round Manager`.

The `Metadata Injector` is explicitly excluded from policy-hosting semantics.

It is schema/directive-governed, not policy-hosted.

This distinction prevents metadata preparation from being incorrectly treated as policy mechanism execution.

## 17. Dispatcher and Handoff Overview

The Dispatcher is the final assembler-side outbound actor.

It is responsible for handoff mediation.

The `Registry / Delivery Port` is the outbound registry-facing contract surface used by the Dispatcher.

The Dispatcher remains the actor.

The port is the contract execution surface.

This distinction is mandatory.

The Registry / Delivery Port must not:

1. replace the Dispatcher as actor,
2. own registry internals,
3. own registry-side lifecycle after valid handoff,
4. accept consuming content ingress,
5. accept administrative policy bundles,
6. collapse handoff into direct registry implementation access.

The handoff path must preserve:

1. registry readiness observation,
2. handoff eligibility,
3. receiver admission compatibility,
4. dispatcher boundary integration,
5. lifecycle transfer discipline,
6. registry internal non-ownership,
7. handoff eligibility verification.

## 18. Downstream Boundary and Authority Exclusion

The Assembler must not leak authority into downstream lifecycle domains.

The following are outside assembler authority:

1. registry internals,
2. storage lifecycle,
3. query lifecycle,
4. indexing,
5. downstream replay,
6. downstream mutation,
7. downstream inspection,
8. post-handoff correction,
9. database ownership,
10. downstream schema derivation.

After valid dispatcher-registry handoff, lifecycle authority transfers away from the Assembler.

The Assembler may preserve trace material or handoff evidence inside its own boundary if defined later, but it must not claim ownership over downstream records.

## 19. C++ Implementation Direction

Assembler System targets:

```text
C++17
```

The preferred implementation direction is:

```text
Header-only first
```

CMake may be used as the expected build-system candidate.

However, build tooling must not drive the architecture.

The architecture drives the code structure.

The source architecture defines typed/template contract doctrine and contract families.

The implementation direction maps this doctrine into C++17 header-first abstractions using templates and CRTP before concrete processing units are derived.

### 19.1 Templates and CRTP

Templates and CRTP are part of the architectural implementation strategy.

They are intended to support:

1. compile-time type discipline,
2. static polymorphism,
3. typed envelope categories,
4. typed contract expression,
5. capability specialization,
6. responsibility-derived abstraction layers,
7. processing-unit derivation after architectural stabilization.

### 19.2 Abstract Layers Before Processing Units

Concrete processing units must not be invented first.

The derivation path is:

```text
External Relation
→ Contract
→ Responsibility
→ Obligation
→ Delegation
→ Enablement
→ Capability Candidate
→ Template / CRTP Abstraction Layer
→ Concrete Processing Unit
```

This protects the project from premature implementation.

## 22. Summary

Assembler System is a bounded C++17 write-side preparation and dispatch subsystem.

It is architecturally governed by taxonomy, contracts, responsibilities, obligations, delegation, enablement, and anti-collapse doctrine.

It transforms governed typed content into authoritative Log Level Envelopes.

It coordinates queue-as-container runtime structures.

It preserves policy-object doctrine.

It separates metadata directives from policy-hosted mechanisms.

It distinguishes queue occupancy from envelope lifecycle ownership.

It uses the Dispatcher as the final assembler-side outbound actor.

It uses the Registry / Delivery Port as the outbound contract execution surface.

It excludes downstream registry, storage, query, and lifecycle authority after valid handoff.

The project is currently in the pre-implementation architecture phase.

The next documentation steps will refine vision, scope, goals, roadmap, and glossary before deriving final C++ packages, filesystem structures, template/CRTP abstraction layers, and concrete processing units.
