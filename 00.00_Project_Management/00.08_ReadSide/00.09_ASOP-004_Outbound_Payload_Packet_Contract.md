## 2. Document Purpose

### 2.1 Purpose Statement

This document defines the **Outbound Payload Packet Contract** for the Assembler System Outbound Ports layer.

It answers the question:

```text
What packet shape, descriptor, payload view, metadata set, version tag,
layout rule, lifetime compatibility rule, and validation requirement must exist
before ASOP may present outbound material to Thin C ABI passage without turning
the packet into a destination schema, add-on protocol, database record,
telemetry event, or final receiver message?
````

The Outbound Payload Packet Contract is not the ABI implementation.

It is not the final destination format.

It is not a database row.

It is not a telemetry event.

It is not an external receiver protocol.

It is the ASOP-owned packet-level contract that makes outbound material safe, describable, version-aware, lifetime-aware, and boundary-ready for Thin C ABI passage.

### 2.2 Why This Document Exists

`ASOP-003` established that Thin C ABI passage requires a passage-safe packet view but does not finalize the packet layout.

Therefore, ASOP needs a separate packet contract.

Without this document, the Thin C ABI boundary would either:

1. expose raw internal payload material,
2. invent layout rules inside the ABI crossing,
3. leak ASIP or ASCC internals,
4. collapse packet shape into destination schema,
5. confuse packet-readiness with receiver consumption,
6. let add-ons define ASOP-owned packet meaning,
7. lose version and lifetime discipline.

This document prevents those failures.

### 2.3 Position in the ASOP Pack

The ASOP sequence is:

```text id="asop-004-sequence"
ASOP-001_Unified_Outbound_Port_Foundation_Model.md
ASOP-002_Port_Owned_Delivery_Round_Model.md
ASOP-003_Thin_C_ABI_Outbound_Passage_Model.md
ASOP-004_Outbound_Payload_Packet_Contract.md
ASOP-005_Port_Diagnostics_And_Failure_Model.md
```

This document is the fourth ASOP document.

It receives:

1. ASOP-owned port identity from ASOP-001,
2. packet-readiness and ABI-readiness context from ASOP-002,
3. Thin C ABI passage constraints from ASOP-003.

It enables:

1. packet-level validation,
2. ABI passage safety,
3. port diagnostics,
4. failure classification,
5. future C++ packet abstraction derivation.

## 4. Core Definition

### 4.1 Outbound Payload Packet

An **Outbound Payload Packet** is the ASOP-governed, passage-safe representation of outbound material prepared for Thin C ABI passage.

It contains or references:

1. payload view,
2. packet descriptor,
3. metadata view,
4. version tag,
5. layout descriptor,
6. size descriptor,
7. lifetime mode,
8. correlation trace,
9. validation state,
10. authority boundary note.

It is designed to be passed across a thin C-compatible boundary without exposing internal C++ object ownership or final destination semantics.

### 4.2 Packet Contract

A **Packet Contract** is the architectural law that defines what an outbound packet must provide, what it may expose, how it is validated, how it is versioned, and what it must not claim.

The packet contract precedes final implementation.

It may later derive:

1. C++ packet view types,
2. C-compatible struct shapes,
3. descriptor traits,
4. validation helpers,
5. lifetime policy adapters,
6. ABI-facing packet handles.

It must not be treated as final code in this document.

### 4.3 Packet Contract Formula

```text id="asop-004-formula"
Outbound Payload Packet Contract
= Packet Identity
+ Payload View
+ Packet Descriptor
+ Metadata View
+ Layout / Size Rules
+ Version Tag
+ Lifetime Mode
+ Validation State
+ Trace Correlation
+ Destination-Semantic Exclusion
```

### 4.4 Packet Contract Non-Formula

The following definitions are invalid:

```text id="asop-004-non-formula"
Outbound Payload Packet = ASIP Internal Container
Outbound Payload Packet = ASCC Bridge Frame
Outbound Payload Packet = Port-Owned Delivery Round
Outbound Payload Packet = Thin C ABI Function
Outbound Payload Packet = Database Row
Outbound Payload Packet = Telemetry Event
Outbound Payload Packet = Receiver Message
Outbound Payload Packet = Add-on Command
Outbound Payload Packet = Final Delivery Proof
```

## 7. Payload View Rules

### 7.1 Payload View Definition

A **Payload View** is the packet-level presentation of outbound payload material.

It may expose material as:

1. pointer-and-size view,
2. immutable structured view,
3. opaque handle view,
4. copied packet body,
5. materialized snapshot,
6. bounded segment list where explicitly allowed.

It must not expose:

1. ASIP internal containers,
2. ASIP slots,
3. ASIP mutable pipeline state,
4. ASCC private bridge frames,
5. ASOP delivery round private context,
6. C++ STL internals,
7. C++ references,
8. template-only public types.

### 7.2 Candidate Payload View Kinds

|        Record ID | Payload View Kind          | Meaning                                                                   |
| ---------------: | -------------------------- | ------------------------------------------------------------------------- |
| ASOP-004-PVK-001 | `byte_span_view`           | Pointer-and-size view over packet-safe bytes                              |
| ASOP-004-PVK-002 | `structured_readonly_view` | Read-only structured view governed by layout descriptor                   |
| ASOP-004-PVK-003 | `opaque_handle_view`       | External handle to packet material under explicit release rules           |
| ASOP-004-PVK-004 | `copied_body`              | Packet body copied into ABI-safe storage                                  |
| ASOP-004-PVK-005 | `transferred_buffer`       | Buffer ownership transferred under explicit rules                         |
| ASOP-004-PVK-006 | `materialized_snapshot`    | Detached snapshot valid beyond source lifecycle                           |
| ASOP-004-PVK-007 | `segmented_view`           | Multiple bounded segments under a descriptor, allowed only when validated |

### 7.3 Payload View Constraints

Every payload view must declare:

1. pointer validity,
2. byte length,
3. alignment expectation where applicable,
4. mutability rule,
5. lifetime mode,
6. release obligation,
7. version compatibility,
8. validation status.

### 7.4 Payload View Non-Claims

A valid payload view does not imply:

1. full serialization success,
2. destination schema compatibility,
3. receiver acceptance,
4. persistence durability,
5. telemetry export readiness,
6. downstream query readiness.

## 9. Size and Layout Rules

### 9.1 Size Rule

Every packet must declare how its size is known.

Valid size models include:

|         Record ID | Size Model               | Meaning                                       |
| ----------------: | ------------------------ | --------------------------------------------- |
| ASOP-004-SIZE-001 | `fixed_size`             | Packet size is fixed by versioned contract    |
| ASOP-004-SIZE-002 | `bounded_dynamic_size`   | Packet size may vary within declared maximum  |
| ASOP-004-SIZE-003 | `descriptor_driven_size` | Packet size is resolved through descriptor    |
| ASOP-004-SIZE-004 | `segmented_bounded_size` | Packet consists of validated bounded segments |
| ASOP-004-SIZE-005 | `snapshot_size`          | Packet is detached snapshot with stable size  |

Invalid size models include:

```text id="asop-004-invalid-size"
unknown_unbounded
implicit_runtime_size
receiver_defined_size
database_defined_size
telemetry_sink_defined_size
raw_cpp_object_size
```

### 9.2 Layout Rule

Every packet must declare a layout family.

Valid layout families may include:

|           Record ID | Layout Family                  | Meaning                                                |
| ------------------: | ------------------------------ | ------------------------------------------------------ |
| ASOP-004-LAYOUT-001 | `opaque_payload_view`          | Payload interpreted only by authorized receiver/add-on |
| ASOP-004-LAYOUT-002 | `byte_sequence`                | Flat byte sequence with descriptor                     |
| ASOP-004-LAYOUT-003 | `structured_header_body`       | Header/body split under versioned rules                |
| ASOP-004-LAYOUT-004 | `descriptor_plus_payload`      | Descriptor controls payload interpretation             |
| ASOP-004-LAYOUT-005 | `multi_segment_packet`         | Multiple segments governed by segment descriptors      |
| ASOP-004-LAYOUT-006 | `materialized_snapshot_packet` | Detached packet snapshot layout                        |

### 9.3 Layout Non-Claims

A layout family is not:

1. full serialization framework,
2. destination schema,
3. database schema,
4. telemetry schema,
5. receiver-specific message format,
6. external runtime ABI implementation.

### 9.4 Alignment and Representation

Where applicable, a packet must define:

1. byte order expectation,
2. alignment expectation,
3. padding rule,
4. nullability rule,
5. pointer validity,
6. reserved field behavior,
7. future extension policy.

This document does not finalize those low-level values.

It requires that later implementation derivation define them before ABI freeze.

## 11. Lifetime Compatibility Rules

### 11.1 Lifetime Requirement

A packet must declare how long its payload view remains valid.

The packet contract must be compatible with the Thin C ABI lifetime mode.

No packet may be passed through the Thin C ABI boundary if its payload view may expire before the declared passage window closes.

### 11.2 Valid Lifetime Modes

|         Record ID | Lifetime Mode           | Meaning                                               |
| ----------------: | ----------------------- | ----------------------------------------------------- |
| ASOP-004-LIFE-001 | `borrowed_view`         | Valid only during a declared boundary passage window  |
| ASOP-004-LIFE-002 | `retained_handle`       | Handle remains valid until explicit release           |
| ASOP-004-LIFE-003 | `copied_packet`         | Packet copy is independent from ASOP internal storage |
| ASOP-004-LIFE-004 | `transferred_buffer`    | Ownership moves under explicit release contract       |
| ASOP-004-LIFE-005 | `materialized_snapshot` | Stable detached snapshot                              |
| ASOP-004-LIFE-006 | `single_pass_view`      | View may be consumed once during passage only         |

### 11.3 Lifetime Compatibility Matrix

| Packet Lifetime Mode         | Compatible With Thin C ABI Passage? | Required Guard                              |
| ---------------------------- | ----------------------------------: | ------------------------------------------- |
| `borrowed_view`              |                                 Yes | Passage window must be explicit             |
| `retained_handle`            |                                 Yes | Release API or release policy must exist    |
| `copied_packet`              |                                 Yes | Copy ownership must be clear                |
| `transferred_buffer`         |                                 Yes | Transfer and release rules must be explicit |
| `materialized_snapshot`      |                                 Yes | Snapshot stability must be validated        |
| `single_pass_view`           |                         Conditional | Reuse must be forbidden and traceable       |
| `implicit_pointer`           |                                  No | No ownership or lifetime rule               |
| `raw_internal_reference`     |                                  No | Leaks internal implementation               |
| `unbounded_external_pointer` |                                  No | Unsafe across ABI boundary                  |

### 11.4 Lifetime Non-Claims

Lifetime compatibility does not imply:

1. downstream delivery,
2. receiver processing,
3. add-on completion,
4. database persistence,
5. telemetry export.

## 13. Packet Readiness and ABI Readiness

### 13.1 Packet Readiness

`packet_ready` means:

```text id="asop-004-packet-ready"
The outbound material has been organized into a valid ASOP packet contract
shape and may be considered for ABI readiness.
```

It does not mean ABI passage has happened.

### 13.2 ABI Readiness

`abi_ready` means:

```text id="asop-004-abi-ready"
The packet contract, payload view, layout, size, version, lifetime,
and validation state are compatible with the target Thin C ABI passage rule.
```

It does not mean passage completed.

### 13.3 Readiness Distinction

```text id="asop-004-readiness-distinction"
packet_ready ≠ abi_ready
abi_ready ≠ passage_completed
passage_completed ≠ receiver_consumed
```

### 13.4 Readiness Table

| Readiness State     | Owner                      | Meaning                                | Not Equivalent To  |
| ------------------- | -------------------------- | -------------------------------------- | ------------------ |
| `packet_ready`      | ASOP packet contract       | Packet contract requirements satisfied | ABI passage        |
| `abi_ready`         | ASOP / Thin C ABI boundary | Packet can be passed through ABI       | Passage completion |
| `passage_completed` | Thin C ABI passage         | Boundary crossing completed            | Receiver success   |
| `receiver_consumed` | Receiver / ASAO            | Destination-side lifecycle event       | ASOP authority     |

## 20. Final Governance Statement

The Outbound Payload Packet Contract is the ASOP-owned packet-level law that defines how outbound material becomes packet-safe for Thin C ABI passage.

It is not the source payload.

It is not the bridge binding.

It is not the delivery round.

It is not the ABI function.

It is not the add-on protocol.

It is not the final receiver message.

It must remain:

```text id="asop-004-final-positive"
ASOP-owned,
packet-level,
version-aware,
layout-declared,
size-bounded,
lifetime-explicit,
payload-view-safe,
metadata-complete,
validation-ready,
ABI-compatible,
traceable,
and destination-neutral.
```

It must not become:

```text id="asop-004-final-negative"
ASIP internal container,
ASCC bridge frame,
Port-Owned Delivery Round,
Thin C ABI implementation,
full serialization engine,
database row,
telemetry event,
add-on command,
external runtime message,
receiver business schema,
or final delivery proof.
```

The governing rule is therefore:

```text id="asop-004-governing-rule"
ASIP produces stable outbound payload.
ASCC binds the payload provider to the outbound port.
ASOP delivery rounds prepare packet readiness.
ASOP packet contract defines boundary-safe packet shape.
Thin C ABI passage carries that packet across the ABI edge.
ASAO or external receivers own destination-specific interpretation beyond that edge.
```
