## 2. Document Purpose

### 2.1 Purpose Statement

This document defines how an **Independent Pipeline** is encapsulated, instantiated, configured, tuned, and exposed as a bounded object without allowing configuration to become a destination model.

It answers the question:

```text
How should an Independent Pipeline be wrapped and configured so that its
internal mechanics remain protected, its stable outbound payload exposure
remains destination-neutral, and its runtime behavior can be tuned without
turning configuration into bridge, port, ABI, database, telemetry, or receiver
semantics?
````

The document defines the architectural boundary between:

```text
pipeline object
pipeline internals
pipeline configuration
pipeline runtime tuning
pipeline exposure surfaces
```

It does not define final C++ implementation.

It does not define bridge execution.

It does not define outbound port configuration.

It does not define Thin C ABI configuration.

It does not define destination-specific add-ons.

### 2.2 Relationship to ASIP-001

`ASIP-001` established the Independent Pipeline as the unit of correctness, ownership, operation, and encapsulation.

It also established the rule:

```text id="asip-004-asip001-rule"
The Independent Pipeline produces stable outbound payloads.
It does not model final destination semantics.
```

This document specializes the encapsulation side of that rule.

The Independent Pipeline must be configurable enough to be useful, but bounded enough to prevent destination leakage.

### 2.3 Relationship to ASIP-002

`ASIP-002` defined the internal mechanics owned by the pipeline:

```text id="asip-004-internal-mechanics"
admission
validation
metadata preparation
timestamp stabilization
material preparation
containers
slots
waiting lists
round managers
moderation
internal dispatch readiness
stabilization
```

This document defines how those mechanics are hidden, parameterized, and governed through encapsulated configuration.

### 2.4 Relationship to ASIP-003

`ASIP-003` defined stable outbound payload exposure as the terminal ASIP boundary.

This document ensures that configuration cannot corrupt that boundary.

Configuration may tune:

1. how payloads are admitted,
2. how internal capacity is shaped,
3. how stabilization rules are selected,
4. how exposure readiness is evaluated,
5. how retention and release are bounded.

Configuration must not define:

1. where payloads finally go,
2. how bridge sessions execute,
3. how outbound ports deliver,
4. how Thin C ABI passage works,
5. how add-ons interpret payloads.

The ASIP sequence is:


## 4. Core Thesis

### 4.1 Primary Thesis

```text id="asip-004-primary-thesis"
An Independent Pipeline is an encapsulated payload-production object.
Its configuration may shape internal behavior and exposure readiness.
Its configuration must not encode final destination semantics.
```

### 4.2 Expanded Thesis

A pipeline must be configurable because different instances may require different:

1. admission rules,
2. validation participation,
3. metadata directives,
4. timestamp stabilization rules,
5. container sizes,
6. slot counts,
7. waiting-list limits,
8. round policies,
9. exposure readiness thresholds,
10. retention policies,
11. diagnostic verbosity.

However, these differences remain **pipeline-internal**.

They do not authorize the pipeline to know whether the payload will later be:

1. written to a database,
2. exported as telemetry,
3. passed through Thin C ABI,
4. received by Python,
5. received by C#,
6. received by MQL5,
7. consumed by an external process,
8. rendered as read-only material.

### 4.3 Configuration Rule

```text id="asip-004-configuration-rule"
Configuration tunes the Independent Pipeline.
Configuration must not become destination routing, bridge orchestration,
outbound port delivery policy, ABI mapping, or add-on projection logic.
```

## 5. Encapsulation Model

### 5.1 Encapsulation Definition

Independent Pipeline encapsulation is the architectural rule that pipeline internals are owned behind a stable pipeline object boundary.

The pipeline object may expose:

1. intake surface,
2. control surface,
3. observation surface,
4. stable payload exposure surface,
5. diagnostic surface,
6. lifecycle surface.

It must not expose:

1. mutable container internals,
2. mutable slot internals,
3. waiting-list internals,
4. round manager mutation handles,
5. internal dispatcher mutation handles,
6. metadata injector internals,
7. timestamp stabilizer internals,
8. validation internals,
9. raw allocator or lock internals,
10. destination-specific handles.

### 5.2 Encapsulation Purpose

Encapsulation protects:

1. correctness boundary,
2. single-writer discipline,
3. mutation closure,
4. exposure safety,
5. configuration integrity,
6. lifecycle management,
7. future implementation replaceability.

### 5.3 Encapsulation Rule

```text id="asip-004-encapsulation-rule"
Pipeline internals may be configured and observed only through governed surfaces.
They must not be externally manipulated as shared mutable structures.
```

## 6. Pipeline Instance Model

### 6.1 Instance Definition

An Independent Pipeline instance is a configured runtime unit that owns its internal mechanics and produces stable outbound payloads.

An instance may have:

1. pipeline identity,
2. configuration identity,
3. admission policy,
4. internal capacity profile,
5. stabilization profile,
6. exposure profile,
7. diagnostics profile,
8. lifecycle state.

### 6.2 Instance Boundary

The instance boundary separates:

```text id="asip-004-instance-boundary"
external configuration / construction
from
internal payload-production mechanics
```

A pipeline instance is not merely a bag of components.

It is the ownership boundary around those components.

### 6.3 Instance Rule

```text id="asip-004-instance-rule"
An Independent Pipeline instance owns its internal mechanics as one correctness
unit. External systems may configure, start, observe, and claim stable payloads,
but must not co-own the internals.
```

## 7. Configuration Object Model

### 7.1 Configuration Object Definition

A pipeline configuration object is a governed input that defines how a pipeline instance should be constructed and tuned.

It may include:

1. identity options,
2. admission policy selection,
3. validation mode,
4. metadata preparation mode,
5. timestamp stabilization mode,
6. container capacity,
7. slot model,
8. waiting-list bounds,
9. round-management policy,
10. exposure-readiness policy,
11. retention policy,
12. diagnostic policy.

### 7.2 Configuration Object Non-Meaning

A pipeline configuration object is not:

1. a bridge binding,
2. a channel profile,
3. a session policy,
4. an outbound port config,
5. a Thin C ABI mapping,
6. an add-on projection config,
7. a database schema,
8. a telemetry exporter config,
9. a language wrapper config,
10. a final receiver descriptor.

### 7.3 Configuration Object Rule

```text id="asip-004-config-object-rule"
A pipeline configuration object configures pipeline behavior only.
It must not carry downstream destination configuration.
```

## 9. Configuration Validation

### 9.1 Validation Purpose

Pipeline configuration must be validated before instance construction or activation.

Validation ensures that the configuration:

1. is internally coherent,
2. respects capacity bounds,
3. preserves destination neutrality,
4. does not leak ASCC/ASOP/ASAO semantics,
5. supports stable outbound payload exposure,
6. does not disable required safety constraints.

### 9.2 Required Validation Questions

|         Record ID | Question                                                                                  |
| ----------------: | ----------------------------------------------------------------------------------------- |
| ASIP-004-VALQ-001 | Does the configuration preserve the Independent Pipeline as the correctness unit?         |
| ASIP-004-VALQ-002 | Does the configuration avoid final destination names or destination-specific behavior?    |
| ASIP-004-VALQ-003 | Are admission rules compatible with internal preparation rules?                           |
| ASIP-004-VALQ-004 | Are stabilization rules compatible with exposure readiness rules?                         |
| ASIP-004-VALQ-005 | Are capacity limits compatible with retention and release policies?                       |
| ASIP-004-VALQ-006 | Does diagnostics configuration avoid leaking mutable internals?                           |
| ASIP-004-VALQ-007 | Does exposure configuration remain PluginAdapter-facing rather than PluginAdapter-owning? |
| ASIP-004-VALQ-008 | Does configuration avoid bridge/session/port/ABI/add-on semantics?                        |

### 9.3 Validation Rule

```text id="asip-004-validation-rule"
A pipeline configuration is valid only if it can tune ASIP behavior without
expanding ASIP authority beyond stable outbound payload exposure.
```

## 10. Configuration Authority

### 10.1 Configuration Authority Definition

Configuration authority is the right to instantiate or tune a pipeline instance.

Configuration authority may be held by a future factory, builder, bootstrap layer, test harness, or controlled application owner.

It is not destination authority.

### 10.2 Configuration Authority May Control

Configuration authority may control:

1. pipeline construction parameters,
2. pipeline capacity profile,
3. internal preparation policies,
4. internal lifecycle thresholds,
5. diagnostics and trace settings,
6. exposure readiness rules,
7. retention and expiration rules.

### 10.3 Configuration Authority Must Not Control

Configuration authority must not use pipeline configuration to control:

1. bridge binding topology,
2. session execution,
3. outbound port delivery,
4. Thin C ABI passage,
5. database projection,
6. telemetry export,
7. language wrapper mapping,
8. final receiver acknowledgement.

### 10.4 Authority Rule

```text id="asip-004-authority-rule"
Pipeline configuration authority is authority over ASIP instance behavior.
It is not authority over communication, delivery, ABI passage, add-on behavior,
or destination lifecycle.
```

## 11. Configuration vs Policy

### 11.1 Distinction

Configuration and policy may overlap in implementation shape, but they are architecturally distinct.

Configuration answers:

```text id="asip-004-config-question"
How should this pipeline instance be constructed and tuned?
```

Policy answers:

```text id="asip-004-policy-question"
Which governed behavior rule must be respected?
```

A configuration may select a policy.

A configuration must not silently become an untracked policy source.

### 11.2 Policy Selection Rule

```text id="asip-004-policy-selection-rule"
Configuration may reference governed policies.
It must not invent policy semantics outside the policy governance model.
```

### 11.3 Configuration/Policy Anti-Collapse

```text id="asip-004-policy-anti-collapse"
Configuration ≠ policy registry.
Configuration ≠ policy object.
Configuration ≠ policy assignment authority unless explicitly governed.
```

## 12. Configuration vs Bridge Binding

### 12.1 Distinction

Pipeline configuration determines the behavior of the Independent Pipeline.

Bridge binding determines how a payload-providing PluginAdapter and an outbound port are connected through ASCC.

These are separate.

### 12.2 Forbidden Bridge Leakage

Pipeline configuration must not include:

1. channel topology,
2. binding compatibility report,
3. session ID,
4. bridge protocol plan,
5. bridge step budget,
6. port obligation pair,
7. endpoint receiver descriptor.

### 12.3 Bridge Boundary Rule

```text id="asip-004-bridge-boundary-rule"
Pipeline configuration may prepare ASIP to expose payloads.
It must not pre-bind ASIP to a specific bridge, channel, session, or port.
```

---

## 13. Configuration vs Outbound Port Delivery

### 13.1 Distinction

Outbound ports own delivery rounds and Thin C ABI passage under ASOP.

The pipeline does not.

### 13.2 Forbidden Port Leakage

Pipeline configuration must not include:

1. outbound port delivery batch shape,
2. port-local queue/container state,
3. ABI function mapping,
4. receiver-side memory layout,
5. destination acknowledgement policy,
6. external retry strategy.

### 13.3 Port Boundary Rule

```text id="asip-004-port-boundary-rule"
Pipeline configuration ends before outbound port delivery configuration begins.
```

---

## 14. Configuration vs Add-On Semantics

### 14.1 Distinction

ASAO and external receivers own destination-specific interpretations.

Pipeline configuration must remain destination-neutral.

### 14.2 Forbidden Add-On Leakage

Pipeline configuration must not include:

1. database table mapping,
2. graph node/edge mapping,
3. telemetry span/resource mapping,
4. Python class mapping,
5. C# DTO mapping,
6. MQL5 receiver mapping,
7. file writer target path,
8. receiver-specific acknowledgement model.

### 14.3 Add-On Boundary Rule

```text id="asip-004-addon-boundary-rule"
Pipeline configuration may describe payload production.
It must not describe payload projection.
```

## 15. Encapsulated Surface Model

### 15.1 Allowed Surfaces

An encapsulated pipeline may expose the following governed surfaces:

|         Record ID | Surface               | Purpose                                                         |
| ----------------: | --------------------- | --------------------------------------------------------------- |
| ASIP-004-SURF-001 | Construction Surface  | Creates or initializes a pipeline instance                      |
| ASIP-004-SURF-002 | Configuration Surface | Accepts governed configuration inputs                           |
| ASIP-004-SURF-003 | Admission Surface     | Admits content or payload material                              |
| ASIP-004-SURF-004 | Control Surface       | Starts, stops, drains, or resets within ASIP authority          |
| ASIP-004-SURF-005 | Observation Surface   | Reports state without exposing mutable internals                |
| ASIP-004-SURF-006 | Exposure Surface      | Makes stable outbound payload views available                   |
| ASIP-004-SURF-007 | Diagnostic Surface    | Emits ASIP-owned diagnostics and trace references               |
| ASIP-004-SURF-008 | Lifecycle Surface     | Manages retention, release, and recycling within ASIP authority |

### 15.2 Forbidden Surfaces

An encapsulated pipeline must not expose:

|          Record ID | Forbidden Surface                | Reason                           |
| -----------------: | -------------------------------- | -------------------------------- |
| ASIP-004-FSURF-001 | Bridge Execution Surface         | Owned by ASCC                    |
| ASIP-004-FSURF-002 | Outbound Port Delivery Surface   | Owned by ASOP                    |
| ASIP-004-FSURF-003 | Thin C ABI Surface               | Owned by ASOP                    |
| ASIP-004-FSURF-004 | Add-On Projection Surface        | Owned by ASAO / receivers        |
| ASIP-004-FSURF-005 | Mutable Container Surface        | Leaks pipeline internals         |
| ASIP-004-FSURF-006 | Mutable Slot Surface             | Leaks runtime mechanics          |
| ASIP-004-FSURF-007 | Destination Routing Surface      | Violates destination neutrality  |
| ASIP-004-FSURF-008 | Receiver Acknowledgement Surface | Expands ASIP lifecycle authority |

### 15.3 Surface Rule

```text id="asip-004-surface-rule"
A pipeline surface must either configure, admit, observe, control, expose,
diagnose, or manage ASIP-owned lifecycle.
It must not perform communication or destination delivery.
```

## 17. Instance Lifecycle Relationship

### 17.1 Configuration and Instance Relationship

A configuration may be applied to create or tune a pipeline instance.

A pipeline instance may reference:

1. configuration ID,
2. configuration version,
3. applied timestamp,
4. active profile,
5. validation result.

But the instance owns runtime behavior after construction.

### 17.2 Runtime Mutation Rule

Configuration changes during runtime must be governed.

A later implementation may choose:

1. immutable configuration per instance,
2. controlled reconfiguration,
3. staged configuration replacement,
4. drain-and-restart reconfiguration.

This document does not select the final implementation strategy.

### 17.3 Instance/Configuration Rule

```text id="asip-004-instance-config-rule"
Configuration creates or tunes an instance.
It must not become shared mutable runtime state that bypasses pipeline authority.
```

## 18. Dependency Model

### 18.1 Configuration Dependency Direction

The dependency direction is:

```text id="asip-004-config-dependency"
governed configuration
→ pipeline construction
→ internal mechanics tuning
→ stable outbound payload exposure
```

The reverse direction is forbidden.

Stable payload exposure must not rewrite the meaning of configuration.

Destination behavior must not rewrite pipeline configuration.

### 18.2 Allowed Dependencies

Pipeline configuration may depend on:

1. ASIP vocabulary,
2. pipeline identity model,
3. internal mechanics model,
4. stable payload exposure model,
5. governance rules,
6. policy references where formally allowed,
7. diagnostic level definitions.

### 18.3 Forbidden Dependencies

Pipeline configuration must not depend on:

1. ASCC channel topology,
2. ASCC binding objects,
3. ASCC sessions,
4. ASOP outbound port internals,
5. ASOP Thin C ABI mappings,
6. ASAO add-on schemas,
7. external receiver internals.

### 18.4 Dependency Rule

```text id="asip-004-dependency-rule"
Pipeline configuration may depend on ASIP-owned concepts.
It must not depend on downstream communication, delivery, ABI, or receiver internals.
```

## 23. Summary

The Independent Pipeline must be encapsulated as a correctness unit.

Its configuration may define:

```text id="asip-004-summary-may"
identity
admission rules
preparation rules
capacity
slot and waiting-list bounds
stabilization rules
exposure readiness
retention and release
diagnostics
```

Its configuration must not define:

```text id="asip-004-summary-must-not"
bridge binding
session execution
outbound port delivery
Thin C ABI passage
database projection
telemetry export
language wrapper mapping
receiver acknowledgement
destination lifecycle
```

The stable architectural conclusion is:

```text id="asip-004-final-conclusion"
ASIP configuration shapes pipeline production behavior.
It does not shape final destination behavior.
```

This prepares the next document:

```text id="asip-005-next"
ASIP-005_Execution_Model_Single_Writer_And_Independent_Pipeline_Instances.md
```

which will define how configured Independent Pipeline instances execute under single-writer discipline without becoming a scheduler, bridge runtime, outbound port runtime, or destination lifecycle owner.
