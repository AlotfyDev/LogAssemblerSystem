## 2. Document Purpose

### 2.1 Purpose Statement

This document defines the **Port-Owned Delivery Round** model for the Assembler System Outbound Ports layer.

It answers the question:

```text
How does an ASOP-owned Outbound Port conduct delivery progression after
ASCC bridge-mediated binding and before Thin C ABI outbound passage without
becoming the Communication Bridge, an ASCC Session, an ABI implementation,
an add-on, a scheduler, or a final receiver lifecycle owner?
````

The Port-Owned Delivery Round is the ASOP-managed unit that moves outbound material through the port’s internal delivery progression.

It begins only after a valid bridge-mediated handoff into the port surface.

It ends before, or at readiness for, Thin C ABI outbound passage.

It does not prove destination success.

It does not prove receiver consumption.

It does not own add-on semantics.

### 2.2 Why This Document Exists

`ASOP-001` established the Unified Outbound Port as the ASOP-owned surface after ASCC binding.

However, the port foundation alone does not define how delivery progresses inside that port.

This document fills that gap.

It defines:

1. what a delivery round is,
2. when it may begin,
3. what it may inspect,
4. what it may prepare,
5. what it may reject,
6. what it may report,
7. when it completes,
8. where its authority closes.

### 2.3 Position in the ASOP Pack

The ASOP sequence is:

```text id="asop-002-sequence"
ASOP-001_Unified_Outbound_Port_Foundation_Model.md
ASOP-002_Port_Owned_Delivery_Round_Model.md
ASOP-003_Thin_C_ABI_Outbound_Passage_Model.md
ASOP-004_Outbound_Payload_Packet_Contract.md
```

This document is the second ASOP document.

It specializes the delivery-round responsibility introduced in ASOP-001.

It prepares the ground for ASOP-003 by defining the port-owned progression that produces ABI-passage readiness.

## 3. Architectural Position

### 3.1 Corrected Outbound Chain

```text id="asop-002-corrected-chain"
ASIP Independent Pipeline
    ↓
Stable Outbound Payload Exposure
    ↓
ASCC Payload-Providing PluginAdapter
    ↓
ASCC Bridge-Mediated Binding
    ↓
ASOP Unified Outbound Port
    ↓
ASOP Port-Owned Delivery Round
    ↓
ASOP Thin C ABI Outbound Passage
    ↓
ASAO Optional Add-ons / External Receivers
```

### 3.2 Delivery Round Entry

A Port-Owned Delivery Round begins only after:

1. ASIP has produced stable outbound payload material,
2. ASCC has bound the payload-providing PluginAdapter to a governed Outbound Port,
3. ASCC compatibility has been satisfied or policy-accepted,
4. ASCC session/protocol/bridge-core progression has reached a valid port handoff point,
5. the Outbound Port has accepted the handoff into its bridge-facing surface.

The delivery round must not start from raw ASIP internals.

The delivery round must not start from an unbound PluginAdapter.

The delivery round must not start from a Channel alone.

The delivery round must not start from an ASCC Binding declaration without port admission.

### 3.3 Delivery Round Exit

A Port-Owned Delivery Round exits by producing one of the following ASOP-owned outcomes:

|        Record ID | Outcome            | Meaning                                                                             |
| ---------------: | ------------------ | ----------------------------------------------------------------------------------- |
| ASOP-002-OUT-001 | `rejected`         | The port refuses delivery progression under ASOP-owned admission or readiness rules |
| ASOP-002-OUT-002 | `deferred`         | The port cannot safely progress delivery now but has not failed permanently         |
| ASOP-002-OUT-003 | `packet_ready`     | Outbound material is ready for packet formation or ABI-passage preparation          |
| ASOP-002-OUT-004 | `abi_ready`        | The round has produced a passage-ready state for Thin C ABI outbound passage        |
| ASOP-002-OUT-005 | `failed_port_side` | The port-side round failed within ASOP-owned responsibility                         |
| ASOP-002-OUT-006 | `cancelled`        | The round was terminated by ASOP-owned policy or lifecycle constraints              |

The delivery round does not exit with:

1. database committed,
2. telemetry exported,
3. external receiver consumed,
4. add-on completed,
5. downstream query-ready,
6. receiver acknowledged,
7. persistence durable.

Those are beyond ASOP delivery-round authority.

## 4. Core Definition

### 4.1 Port-Owned Delivery Round

A **Port-Owned Delivery Round** is an ASOP-owned bounded progression unit inside an Outbound Port.

It governs the movement of bridge-mediated outbound material from accepted port admission toward Thin C ABI passage readiness.

It is:

```text id="asop-002-round-definition"
port-owned,
post-ASCC-binding,
pre-ABI-passage,
destination-neutral,
diagnostic-capable,
authority-bounded.
```

### 4.2 Delivery Round Formula

```text id="asop-002-round-formula"
Port-Owned Delivery Round
= Port Admission Result
+ Delivery Eligibility State
+ Port-Owned Progression Step(s)
+ Packet Readiness Preparation
+ ABI Passage Readiness Decision
+ Port-Side Diagnostics
+ Authority Closure Before Destination Semantics
```

### 4.3 Delivery Round Non-Formula

The following definitions are invalid:

```text id="asop-002-non-formula"
Port-Owned Delivery Round = ASCC Session
Port-Owned Delivery Round = Bridge Protocol Step
Port-Owned Delivery Round = Bridge Core Run
Port-Owned Delivery Round = Thin C ABI Passage
Port-Owned Delivery Round = Add-on Execution
Port-Owned Delivery Round = Receiver Lifecycle
Port-Owned Delivery Round = Database Transaction
Port-Owned Delivery Round = Telemetry Export
Port-Owned Delivery Round = Scheduler Tick
```

## 5. Ownership Rules

### 5.1 Primary Ownership Rule

```text id="asop-002-primary-ownership-rule"
The Port-Owned Delivery Round belongs to ASOP and is owned by the Outbound Port.
```

The round is not owned by:

1. ASIP,
2. ASCC,
3. ASAO,
4. Thin C ABI alone,
5. external receiver,
6. database system,
7. telemetry system,
8. runtime scheduler.

### 5.2 Bridge Separation Rule

```text id="asop-002-bridge-separation-rule"
ASCC binding and session orchestration make port delivery possible.
They do not execute the Port-Owned Delivery Round.
```

ASCC may establish that a provider and port are compatible.

ASCC may create a binding.

ASCC may open a session.

ASCC may run a bridge protocol.

ASCC may step Bridge Core.

ASCC must not become the owner of port-side delivery progression.

### 5.3 ABI Separation Rule

```text id="asop-002-abi-separation-rule"
A delivery round may prepare for Thin C ABI passage.
It must not become the Thin C ABI passage model itself.
```

The delivery round may determine that outbound material is passage-ready.

The ABI layer defines the logic-free passage protocol.

The detailed ABI model belongs to ASOP-003.

### 5.4 Add-on Separation Rule

```text id="asop-002-addon-separation-rule"
A delivery round must not execute add-on destination semantics.
```

It may prepare material for a database add-on.

It may prepare material for telemetry add-on passage.

It may prepare material for external runtime passage.

It must not own:

1. database commit,
2. telemetry export completion,
3. external runtime lifecycle,
4. projection lifecycle,
5. receiver retry policy,
6. destination backpressure policy unless explicitly represented as port-side admission signal.

### 5.5 Scheduler Separation Rule

```text id="asop-002-scheduler-separation-rule"
A Port-Owned Delivery Round is not a scheduler.
```

A scheduler may later decide when a round is invoked.

The round itself defines bounded delivery progression semantics, not global scheduling policy.

## 6. Round Preconditions

### 6.1 Required Preconditions

A Port-Owned Delivery Round may begin only when all required conditions are satisfied.

|        Record ID | Precondition                           | Owner                     | Required Meaning                                               |
| ---------------: | -------------------------------------- | ------------------------- | -------------------------------------------------------------- |
| ASOP-002-PRE-001 | Stable outbound payload exists         | ASIP                      | Payload material has been stabilized before outbound exposure  |
| ASOP-002-PRE-002 | Payload provider surface exists        | ASCC-facing PluginAdapter | Payload can be represented through a governed provider surface |
| ASOP-002-PRE-003 | Outbound port surface exists           | ASOP                      | Port has a declared bridge-facing surface                      |
| ASOP-002-PRE-004 | Binding exists                         | ASCC                      | Provider and port are paired under a governed binding          |
| ASOP-002-PRE-005 | Compatibility is satisfied or accepted | ASCC                      | Compatibility has been evaluated under bridge governance       |
| ASOP-002-PRE-006 | Session scope is valid                 | ASCC                      | Binding progression is in a valid session context              |
| ASOP-002-PRE-007 | Bridge protocol allows port handoff    | ASCC                      | Protocol state permits handoff into the port                   |
| ASOP-002-PRE-008 | Port admission accepts material        | ASOP                      | Port-side admission allows round creation                      |
| ASOP-002-PRE-009 | Port is not closed or blocked          | ASOP                      | Port lifecycle permits delivery progression                    |
| ASOP-002-PRE-010 | Round budget is available              | ASOP                      | Port-side constraints permit a bounded delivery round          |

### 6.2 Forbidden Preconditions

The following are not valid preconditions:

|         Record ID | Invalid Precondition         | Reason                                                               |
| ----------------: | ---------------------------- | -------------------------------------------------------------------- |
| ASOP-002-FPRE-001 | Raw payload exists           | Bypasses ASIP stabilization                                          |
| ASOP-002-FPRE-002 | Channel exists               | Channel topology is not delivery progression                         |
| ASOP-002-FPRE-003 | PluginAdapter exists         | Adapter existence is not binding or admission                        |
| ASOP-002-FPRE-004 | Port exists                  | Port existence alone is not admission                                |
| ASOP-002-FPRE-005 | Add-on is available          | Add-on availability does not authorize port delivery round by itself |
| ASOP-002-FPRE-006 | Receiver requested material  | Receiver request does not bypass ASCC/ASOP governance                |
| ASOP-002-FPRE-007 | Database is connected        | Persistence availability is outside port-round authority             |
| ASOP-002-FPRE-008 | Telemetry sink is configured | Telemetry configuration is not port-round authorization              |

## 7. Round Lifecycle

### 7.1 Lifecycle Overview

```text id="asop-002-lifecycle"
1. Bridge-mediated handoff reaches the Outbound Port surface.
2. Port admission evaluates whether delivery progression may begin.
3. A delivery round is opened.
4. Round context is initialized.
5. Port-owned readiness is checked.
6. Delivery eligibility is confirmed.
7. Outbound material is normalized into a port-side delivery view.
8. Packet-readiness preparation is performed.
9. ABI-passage readiness is determined.
10. Round outcome is recorded.
11. Port-owned diagnostics are emitted.
12. Round closes without claiming destination lifecycle completion.
```

### 7.2 Lifecycle Table

|         Step ID | Step                             | Owner                | Output                  | Not Equivalent To               |
| --------------: | -------------------------------- | -------------------- | ----------------------- | ------------------------------- |
| ASOP-002-LC-001 | Handoff received at port surface | ASCC → ASOP boundary | Port handoff event      | Final delivery                  |
| ASOP-002-LC-002 | Port admission evaluated         | ASOP                 | Admission decision      | ASIP admission                  |
| ASOP-002-LC-003 | Delivery round opened            | ASOP                 | Round context           | ASCC session                    |
| ASOP-002-LC-004 | Round context initialized        | ASOP                 | Round state             | Scheduler task                  |
| ASOP-002-LC-005 | Port readiness checked           | ASOP                 | Readiness view          | Receiver readiness              |
| ASOP-002-LC-006 | Delivery eligibility confirmed   | ASOP                 | Eligibility state       | Compatibility evaluation        |
| ASOP-002-LC-007 | Delivery view prepared           | ASOP                 | Port-side outbound view | Payload mutation by destination |
| ASOP-002-LC-008 | Packet readiness prepared        | ASOP                 | Packet-ready material   | ABI passage                     |
| ASOP-002-LC-009 | ABI readiness determined         | ASOP                 | ABI-ready state         | Add-on execution                |
| ASOP-002-LC-010 | Outcome recorded                 | ASOP                 | Round outcome           | Receiver acknowledgement        |
| ASOP-002-LC-011 | Diagnostics emitted              | ASOP                 | Port diagnostic report  | Telemetry export                |
| ASOP-002-LC-012 | Round closed                     | ASOP                 | Closed round state      | Destination lifecycle closure   |

## 8. Round State Model

### 8.1 Candidate Round States

|          Record ID | State                | Meaning                                               |
| -----------------: | -------------------- | ----------------------------------------------------- |
| ASOP-002-STATE-001 | `not_started`        | No ASOP delivery round has begun                      |
| ASOP-002-STATE-002 | `admission_pending`  | Port admission is being evaluated                     |
| ASOP-002-STATE-003 | `admitted`           | Port admission accepted the handoff                   |
| ASOP-002-STATE-004 | `opened`             | Delivery round context exists                         |
| ASOP-002-STATE-005 | `checking_readiness` | Port-owned readiness checks are active                |
| ASOP-002-STATE-006 | `eligible`           | Material may progress through the round               |
| ASOP-002-STATE-007 | `preparing_packet`   | Port is preparing packet-readiness                    |
| ASOP-002-STATE-008 | `abi_ready`          | Material is ready for Thin C ABI passage              |
| ASOP-002-STATE-009 | `deferred`           | Round cannot safely progress now                      |
| ASOP-002-STATE-010 | `rejected`           | Round is rejected by port-owned rule                  |
| ASOP-002-STATE-011 | `failed_port_side`   | Round failed within ASOP responsibility               |
| ASOP-002-STATE-012 | `closed`             | Round has completed ASOP-owned progression            |
| ASOP-002-STATE-013 | `cancelled`          | Round terminated under ASOP-owned lifecycle or policy |

### 8.2 Terminal States

Terminal round states are:

```text id="asop-002-terminal-states"
abi_ready
deferred
rejected
failed_port_side
closed
cancelled
```

A terminal state does not imply destination success.

### 8.3 Invalid States

The following states must not appear in the delivery-round model:

```text id="asop-002-invalid-states"
database_committed
telemetry_exported
receiver_consumed
query_ready
external_runtime_completed
addon_finished
persistent_record_created
```

Those states belong beyond ASOP.

## 11. Delivery Eligibility

### 11.1 Eligibility Definition

Delivery eligibility is the ASOP-owned decision that outbound material may progress within the Outbound Port toward packet-readiness and ABI-passage readiness.

It is not equivalent to ASCC compatibility.

It is not equivalent to ASIP stabilization.

It is not equivalent to receiver readiness.

### 11.2 Eligibility Criteria

Candidate eligibility criteria include:

|         Record ID | Criterion                   | Meaning                                                        |
| ----------------: | --------------------------- | -------------------------------------------------------------- |
| ASOP-002-ELIG-001 | Port lifecycle open         | Port must be in a state that allows delivery progression       |
| ASOP-002-ELIG-002 | Admission accepted          | Port admission must accept the bridge-mediated handoff         |
| ASOP-002-ELIG-003 | Delivery budget available   | The round must respect bounded port-owned limits               |
| ASOP-002-ELIG-004 | Payload view valid          | Stable payload view must be suitable for port-side progression |
| ASOP-002-ELIG-005 | Packet contract available   | Packet-readiness contract must be known                        |
| ASOP-002-ELIG-006 | ABI contract available      | ABI passage target contract must be known                      |
| ASOP-002-ELIG-007 | Diagnostic policy available | The round must be able to report its outcome                   |
| ASOP-002-ELIG-008 | No port-side block          | Port must not be blocked by ASOP lifecycle or policy           |

### 11.3 Eligibility Non-Claims

Delivery eligibility does not claim:

1. destination acceptance,
2. add-on availability,
3. ABI passage completion,
4. final receiver consumption,
5. downstream persistence,
6. downstream query readiness.

The Delivery Round Context stores the ASOP-owned state needed to evaluate, progress, and close one port delivery round.

It is local to the Outbound Port.

It must not become a global scheduler record.

It must not become a downstream delivery receipt.

## 14. Diagnostics and Trace

### 14.1 Diagnostic Role

A Port-Owned Delivery Round must emit ASOP-owned diagnostic evidence sufficient to understand:

1. why the round began,
2. which port owned it,
3. which binding authorized it,
4. which admission result was reached,
5. which state transitions occurred,
6. whether packet readiness was produced,
7. whether ABI readiness was reached,
8. why the round failed, rejected, deferred, or closed.

### 14.2 Diagnostic Non-Role

Port diagnostics are not:

1. telemetry export,
2. downstream logging,
3. database audit record,
4. receiver acknowledgement,
5. persistent delivery proof,
6. query trace,
7. external runtime trace.

## 15. Anti-Collapse Rules

### 15.1 Delivery Round / ASIP Anti-Collapse

```text id="asop-002-anti-asip"
Port-Owned Delivery Round ≠ ASIP execution.
Port Admission ≠ Pipeline Admission.
Delivery Round Context ≠ Pipeline Internal State.
Packet Readiness ≠ Stable Payload Exposure.
Round Closure ≠ ASIP Recycling.
```

### 15.2 Delivery Round / ASCC Anti-Collapse

```text id="asop-002-anti-ascc"
Port-Owned Delivery Round ≠ ASCC Binding.
Port-Owned Delivery Round ≠ ASCC Session.
Port-Owned Delivery Round ≠ Bridge Protocol.
Port-Owned Delivery Round ≠ Bridge Core Step.
Delivery Eligibility ≠ Compatibility Evaluation.
```

### 15.3 Delivery Round / ABI Anti-Collapse

```text id="asop-002-anti-abi"
Port-Owned Delivery Round ≠ Thin C ABI.
Packet Readiness ≠ ABI Passage.
ABI Readiness ≠ ABI Completion.
Round Closure ≠ ABI Boundary Completion.
```

### 15.4 Delivery Round / ASAO Anti-Collapse

```text id="asop-002-anti-asao"
Port-Owned Delivery Round ≠ Add-on Execution.
Port-Owned Delivery Round ≠ Database Writer.
Port-Owned Delivery Round ≠ Telemetry Exporter.
Port-Owned Delivery Round ≠ External Runtime Adapter.
Port-Owned Delivery Round ≠ Projection Owner.
```

### 15.5 Delivery Round / Receiver Anti-Collapse

```text id="asop-002-anti-receiver"
Round Success ≠ Receiver Consumption.
Round Closure ≠ Destination Success.
ABI Readiness ≠ Receiver Acknowledgement.
Packet Prepared ≠ Downstream Persistence.
```

## 18. Final Governance Statement

The Port-Owned Delivery Round is the ASOP-owned unit of outbound delivery progression inside an Outbound Port.

It starts only after ASCC has completed bridge-mediated binding and valid handoff into the port surface.

It progresses material through port-owned admission, readiness, eligibility, packet-readiness, and ABI-readiness.

It must remain:

```text id="asop-002-final-positive"
port-owned,
post-binding,
pre-ABI-passage,
destination-neutral,
bounded,
diagnostic-capable,
and authority-closed.
```

It must not become:

```text id="asop-002-final-negative"
ASIP execution,
ASCC Binding,
ASCC Session,
Bridge Protocol,
Bridge Core,
Thin C ABI passage,
add-on execution,
database persistence,
telemetry export,
external runtime lifecycle,
receiver consumption,
or global scheduling.
```

The governing rule is therefore:

```text id="asop-002-governing-rule"
ASCC binds.
ASOP ports admit and run port-owned delivery rounds.
ASOP Thin C ABI passage carries outbound material across the ABI boundary.
ASAO or external receivers own destination-specific behavior beyond that boundary.
```
