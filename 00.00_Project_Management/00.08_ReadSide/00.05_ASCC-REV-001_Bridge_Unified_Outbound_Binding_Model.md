## 2. Document Purpose

### 2.1 Purpose Statement

This document defines the unified outbound binding model for the Assembler System Communication Context.

It answers the question:

```text
How does the Communication Bridge bind a stable outbound payload provider
to a governed outbound port without collapsing ASIP payload production,
ASCC bridge orchestration, ASOP outbound delivery, Thin C ABI passage,
or ASAO add-on semantics into one layer?
````

The document is a corrective revision.

It exists because the architecture has moved from an earlier write-side/read-side distribution model into a more precise outbound model:

```text
Independent Pipeline
→ Stable Outbound Payload Exposure
→ Payload-Providing PluginAdapter
→ Communication Bridge
→ Bound Outbound Port
→ Port-Owned Delivery Round
→ Thin C ABI Outbound Passage
→ Optional Add-ons / External Receivers
```

ASCC-REV-001 governs the middle of this chain:

```text
Payload-Providing PluginAdapter
→ Communication Bridge
→ Bound Outbound Port
```

It does not govern what happens inside the Independent Pipeline before payload exposure.

It does not govern what happens inside the Outbound Port after binding.

It does not govern Thin C ABI passage.

It does not govern final receiver semantics.

### 2.2 Why This Revision Exists

The ASIP documentation stabilized the Independent Pipeline as the unit of correctness and explicitly ended ASIP authority at stable outbound payload exposure. ASIP does not perform bridge binding, session execution, outbound port delivery rounds, Thin C ABI passage, add-on projection, or final receiver semantics .

At the same time, the Communication Context already has a mature bridge model: ASCC-001 established `communication_context/` as a root domain whose purpose is bridge-mediated communication rather than endpoint internals . Later ASCC implementation waves created the bridge building blocks: port/plugin-adapter obligations, binding declarations, compatibility evaluation, sessions, protocol model, and endpoint-neutral bridge core      .

This revision consolidates those pieces into one outbound binding doctrine.

### 2.3 Non-Purpose

This document does not define:

1. final C++ files,
2. final class signatures,
3. final bridge runtime loop,
4. final outbound port implementation,
5. port-owned delivery rounds,
6. Thin C ABI packet layout,
7. ABI memory ownership rules,
8. add-on receiver semantics,
9. OpenTelemetry exporter semantics,
10. database writer semantics,
11. external language receiver semantics,
12. persistence lifecycle,
13. production scheduler behavior.

Those belong to later ASOP and ASAO documents.

## 3. Architectural Position

### 3.1 Current Corrected Chain

The corrected outbound chain is:

```text
ASIP Independent Pipeline
    produces stable outbound payload

ASIP Stable Exposure Boundary
    exposes payload to a PluginAdapter-facing boundary

ASCC Payload-Providing PluginAdapter
    presents the payload as a bridge-compatible provider surface

ASCC Communication Bridge
    validates compatibility, opens a session, applies bridge protocol,
    and binds provider to outbound port surface

ASOP Outbound Port
    owns delivery round and outbound passage preparation

ASOP Thin C ABI
    provides logic-free cross-boundary payload passage

ASAO Optional Add-ons
    own destination-specific behavior beyond the Thin C ABI
```

### 3.2 ASCC Placement

ASCC sits after ASIP exposure and before ASOP delivery.

It owns:

1. bridge vocabulary,
2. channel topology,
3. participant roles,
4. port and plugin-adapter obligation surfaces,
5. binding declaration,
6. compatibility evaluation,
7. session lifecycle,
8. protocol model,
9. bridge-core model-step orchestration,
10. optional endpoint-neutral registries,
11. optional endpoint-neutral diagnostics.

It does not own:

1. Independent Pipeline internals,
2. stable payload production,
3. outbound port delivery rounds,
4. Thin C ABI passage,
5. add-on destination semantics,
6. final receiver lifecycle,
7. persistence lifecycle,
8. telemetry exporter behavior,
9. external language runtime behavior.

### 3.3 Revision Rule

```text
ASCC binds outbound participants.
ASCC does not deliver final payloads.
```

This means:

```text
Bridge binding is not outbound delivery.
Session validity is not ABI passage.
Protocol completion is not receiver consumption.
Bridge-core success is not destination success.
```

## 4. Core Thesis

### 4.1 Thesis Statement

The unified outbound binding model is the architectural rule that a stable payload provider and an outbound port must be connected only through a governed Communication Bridge.

The bridge connection must be explicit, typed, compatibility-checked, session-scoped, protocol-governed, and boundary-safe.

### 4.2 Binding Formula

```text
Payload-Providing PluginAdapter
+ Outbound Port
+ Channel Topology
+ Obligation Surface Pair
+ Compatibility Report
+ Binding Declaration
+ Session Scope
+ Bridge Protocol
+ Bridge Core Model Step
= Governed Outbound Binding
```

### 4.3 Binding Non-Formula

The following are invalid shortcuts:

```text
Stable Payload → Outbound Port
Stable Payload → Thin C ABI
Pipeline → Receiver
PluginAdapter → Receiver
Bridge → Database
Bridge → Telemetry Exporter
Bridge → External Runtime
```

The bridge must bind surfaces.

It must not bypass ASOP-owned port delivery.

## 5. Terminology

### 5.1 Stable Outbound Payload

A stable outbound payload is the ASIP-produced material exposed after pipeline internal mechanics have completed.

It is stable enough to be claimed or viewed by an outbound-facing adapter.

It is not yet delivered.

It is not yet an ABI packet.

It is not yet receiver-owned.

### 5.2 Payload-Providing PluginAdapter

A Payload-Providing PluginAdapter is the ASCC-facing provider surface that represents stable outbound payload material to the Communication Bridge.

It does not create the payload.

It does not own ASIP internal mechanics.

It does not decide the destination.

It presents the payload under ASCC bridge obligations.

### 5.3 Outbound Port

An Outbound Port is the ASOP-owned port surface that receives bridge-mediated outbound material for port-owned delivery.

In this document, the Outbound Port is treated only as a binding participant.

Its internal delivery rounds are defined later in ASOP-002.

### 5.4 Unified Outbound Binding

Unified Outbound Binding is the governed declaration that a payload-providing PluginAdapter and an Outbound Port are compatible and connected under a channel, obligation pair, session, and bridge protocol.

It is not direct invocation.

It is not final delivery.

It is not Thin C ABI passage.

### 5.5 Bridge Core

Bridge Core is the endpoint-neutral orchestrator over protocol steps.

W10 already fixed this boundary: Bridge Core is endpoint-neutral and is not a concrete endpoint bridge, adapter/port invocation mechanism, runtime scheduler, persistence layer, telemetry layer, or ABI layer .

### 5.6 First Integration Skeleton

The first integration skeleton demonstrated the model-level chain:

```text
Channel
→ Binding
→ Session
→ Protocol
→ Bridge Core model run
```

and explicitly remained only an integration skeleton, not production runtime, scheduler, slot/container implementation, persistence, telemetry, or ABI .

ASCC-REV-001 turns that integration lesson into formal architecture doctrine.

## 6. Governing Rules

### 6.1 ASIP-to-ASCC Rule

```text
ASIP exposes stable outbound payloads.
ASCC binds the exposed payload provider to an outbound port.
```

ASIP must not know:

1. final destination type,
2. outbound delivery rounds,
3. ABI passage,
4. add-on projection,
5. external receiver lifecycle.

ASCC must not reach backward into ASIP internals.

### 6.2 ASCC-to-ASOP Rule

```text
ASCC produces a governed outbound binding.
ASOP owns outbound port delivery after binding.
```

ASCC may validate that an outbound port surface is compatible.

ASCC may open a session around a binding.

ASCC may run bridge protocol steps.

ASCC may produce bridge-core traces or reports.

ASCC must not own the port’s delivery round.

### 6.3 Binding Declaration Rule

```text
A binding is a compatibility-checked declaration, not endpoint execution.
```

This is aligned with W06, where Binding was explicitly defined as a compatibility-checked declaration between Channel and ObligationSurfacePair, not Session, Protocol execution, Bridge orchestration, or endpoint invocation .

### 6.4 Compatibility Rule

```text
Compatibility is a validation and diagnostic layer over binding declarations.
```

Compatibility does not execute a session, protocol, bridge orchestration, or endpoint invocation. W07 fixed this boundary explicitly .

### 6.5 Session Rule

```text
A session is a lifecycle scope around a compatibility-checked binding.
```

A session is not protocol execution.

A session is not bridge orchestration.

A session is not endpoint invocation.

A session is not endpoint resource ownership. This boundary was fixed in W08 .

### 6.6 Protocol Rule

```text
A bridge protocol is an endpoint-neutral state-transition model over a valid session.
```

It is not TBridge itself.

It is not adapter/port call wiring.

It is not endpoint invocation.

This boundary was fixed in W09 .

### 6.7 Bridge Core Rule

```text
Bridge Core performs endpoint-neutral model-step orchestration over bridge protocol.
```

It does not become:

1. endpoint bridge,
2. adapter invocation engine,
3. port invocation engine,
4. runtime scheduler,
5. persistence engine,
6. telemetry exporter,
7. Thin C ABI layer.

### 6.8 Outbound Port Rule

```text
Outbound Port participation in ASCC is limited to its bridge-facing surface.
```

ASCC may bind to an Outbound Port.

ASCC may not absorb the port’s delivery round.

ASCC may not define ABI packet passage.

ASCC may not define add-on behavior beyond the port.

### 7.2 Provider Side

The provider side is:

```text
Stable Outbound Payload
→ Payload-Providing PluginAdapter
```

The provider side must preserve:

1. payload stability,
2. payload identity,
3. payload readiness state,
4. payload claim/release discipline,
5. ASIP non-destination rule.

The provider side must not expose:

1. ASIP internal containers,
2. ASIP internal slots,
3. ASIP admission internals,
4. ASIP recycling internals,
5. ASIP scheduler assumptions.

### 7.3 Port Side

The port side is:

```text
Outbound Port
→ Port-Owned Delivery Round
→ Thin C ABI Passage
```

ASCC sees only the bridge-facing port surface.

ASCC must not define the port’s delivery round.

ASCC must not define ABI packet structure.

ASCC must not define receiver completion.

## 8. Unified Binding Lifecycle

### 8.1 Lifecycle Overview

```text
1. Provider surface becomes available.
2. Outbound port surface is declared.
3. Channel topology is selected.
4. ObligationSurfacePair is formed.
5. Compatibility is evaluated.
6. Binding declaration is created.
7. Session is opened around binding.
8. Bridge protocol is selected.
9. Bridge Core performs endpoint-neutral model step.
10. Outbound port receives bridge-mediated outbound material.
11. ASOP delivery begins after ASCC boundary.
```

|             Step ID | Step                         | Owner                      | Output                               | Boundary                         |

| ASCC-REV-001-LC-003 | Port surface declaration     | ASOP / ASCC-facing surface | Outbound port obligation surface     | Port internals remain ASOP-owned |
| ASCC-REV-001-LC-004 | Channel selection            | ASCC                       | Channel topology                     | No session yet                   |
| ASCC-REV-001-LC-005 | Obligation pairing           | ASCC                       | ObligationSurfacePair                | No endpoint call                 |
| ASCC-REV-001-LC-006 | Compatibility evaluation     | ASCC                       | Compatibility report                 | No session execution             |
| ASCC-REV-001-LC-007 | Binding declaration          | ASCC                       | Binding                              | No delivery yet                  |

## 9. Binding Contract Model

### 9.1 Contract Purpose

The Unified Outbound Binding Contract defines the minimum architectural conditions required before stable outbound payload material may move from ASIP exposure toward ASOP outbound delivery.

### 9.2 Required Contract Fields

| -------------------: | ---------------------------- | -------------------------------------------------------------- |

| ASCC-REV-001-CON-004 | `channel_id`                 | Channel topology used for this binding                         |
| ASCC-REV-001-CON-005 | `obligation_surface_pair_id` | Pairing of provider and port obligations                       |
| ASCC-REV-001-CON-006 | `compatibility_report_id`    | Compatibility validation record                                |

| ASCC-REV-001-CON-012 | `trace_evidence`             | Bridge-side trace, report, or diagnostic reference             |

### 9.3 Contract Invariants

```text
A Unified Outbound Binding is valid only if:
1. the provider surface is declared,
2. the outbound port surface is declared,
3. a channel is selected,
4. obligations are paired,
5. compatibility evaluation passes or is explicitly accepted under policy,
6. a binding declaration is created,
7. a session can be opened,
8. a bridge protocol is applicable,
9. the bridge core can perform an endpoint-neutral model step,
10. ASCC authority closure before ASOP delivery is preserved.
```

## 11. Anti-Collapse Rules

### 11.1 ASIP / ASCC Anti-Collapse

```text
Independent Pipeline ≠ Communication Bridge.
Stable Payload Exposure ≠ Outbound Binding.
Payload Provider ≠ Final Destination.
```

### 11.2 ASCC / ASOP Anti-Collapse

```text
Bridge Binding ≠ Outbound Port Delivery.
Bridge Session ≠ Port Delivery Round.
Bridge Protocol ≠ Thin C ABI Passage.
Bridge Core ≠ Outbound Port Runtime.
```

### 11.3 ASOP / ASAO Anti-Collapse

```text
Thin C ABI Passage ≠ Add-on Semantics.
Outbound Port Delivery ≠ Receiver Lifecycle.
ABI Packet Transfer ≠ Database Write.
ABI Packet Transfer ≠ Telemetry Export.
ABI Packet Transfer ≠ External Language Ownership.
```

### 11.4 Binding Anti-Collapse

```text
Binding ≠ Session.
Session ≠ Protocol.
Protocol ≠ Bridge Core.
Bridge Core ≠ Endpoint Invocation.
Endpoint Invocation ≠ Destination Ownership.
```

### 11.5 Registry Anti-Collapse

Optional ASCC registries may support lookup.

They must not become:

1. service locators,
2. dependency injection containers,
3. persistence registries,
4. endpoint lifecycle owners,
5. runtime schedulers.

This aligns with W11, which fixed optional registries as bounded and optional, not runtime scheduler, service locator, DI container, persistence registry, or endpoint lifecycle authority .

## 12. Authority Boundaries

### 12.1 ASIP Authority Ends

ASIP authority ends when:

1. payload is stable,
2. exposure boundary is satisfied,
3. provider-facing surface may claim or view payload,
4. ASIP-owned release rules remain protected.

ASIP does not decide:

1. which outbound port receives the payload,
2. how the bridge session is opened,
3. how the port performs delivery,
4. how Thin C ABI passage is encoded,
5. how add-ons interpret the result.

### 12.2 ASCC Authority Ends

ASCC authority ends when:

1. a valid binding has been declared,
2. session and protocol constraints have been satisfied,
3. Bridge Core has completed its endpoint-neutral model step,
4. outbound material is handed to the ASOP-owned port surface.

ASCC does not decide:

1. port delivery round internals,
2. ABI packet layout,
3. ABI memory transfer semantics,
4. add-on retry behavior,
5. receiver-side lifecycle.

### 12.3 ASOP Authority Begins

ASOP authority begins at the outbound port surface after ASCC bridge-mediated binding.

ASOP owns:

1. outbound port identity,
2. port delivery round,
3. port-owned admission into outbound passage,
4. Thin C ABI passage model,
5. outbound payload packet contract,
6. ABI-facing safety rules.

### 12.4 ASAO Authority Begins

ASAO authority begins beyond the Thin C ABI.

ASAO owns optional destination-specific semantics, such as:

1. telemetry exporters,
2. database writers,
3. read-only projections,
4. external runtime receivers,
5. language-specific adapters,
6. optional persistence or projection add-ons.

## 16. Final Governance Statement

The Communication Context is the governed binding layer between stable payload exposure and outbound delivery.

It must remain:

```text
endpoint-neutral,
compatibility-governed,
session-scoped,
protocol-driven,
bridge-core-orchestrated,
and authority-bounded.
```

It must not become:

```text
an Independent Pipeline,
an Outbound Port runtime,
a Thin C ABI layer,
a destination add-on,
a receiver lifecycle owner,
a persistence engine,
a telemetry exporter,
or a scheduler.
```

The unified outbound model is therefore:

```text
ASIP produces.
ASCC binds.
ASOP delivers through ports and ABI.
ASAO interprets beyond ABI.
```
