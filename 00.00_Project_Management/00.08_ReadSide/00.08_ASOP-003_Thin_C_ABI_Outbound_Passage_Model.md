## 2. Document Purpose

### 2.1 Purpose Statement

This document defines the **Thin C ABI Outbound Passage Model** for the Assembler System Outbound Ports layer.

It answers the question:

```text
How should ASOP pass outbound material across a thin C ABI boundary after
a port-owned delivery round reaches ABI readiness, without turning the ABI
boundary into a delivery round, serialization engine, add-on executor,
database writer, telemetry exporter, or final receiver lifecycle owner?
````

The Thin C ABI is a boundary-passage model.

It is intentionally thin.

It exists to allow outbound material to cross from the C++ ASOP side into ABI-facing consumers or add-on adapters through a stable, minimal, logic-free protocol.

It does not decide destination meaning.

It does not transform payload semantics.

It does not own delivery success.

It does not own receiver lifecycle.

### 2.2 Why This Document Exists

`ASOP-002` defined the Port-Owned Delivery Round and its terminal readiness outcomes.

However, an `abi_ready` result is not itself ABI passage.

The project therefore needs a separate model for the crossing boundary.

This document defines:

1. what the Thin C ABI is,
2. why it is thin,
3. what passage means,
4. what material may cross,
5. how the boundary must avoid logic ownership,
6. what it may report,
7. what it must not claim,
8. how it prepares ASOP-004 packet contract definition.

### 2.3 Position in the ASOP Pack

The ASOP sequence is:


## 3. Architectural Position

### 3.1 Corrected Outbound Chain

```text id="asop-003-corrected-chain"
ASIP Independent Pipeline
    ↓
Stable Outbound Payload Exposure
    ↓
ASCC Payload-Providing PluginAdapter
    ↓
ASCC Bridge-Mediated Binding
    ↓
ASOP Unified Outbound Port
    ↓
ASOP Port-Owned Delivery Round
    ↓
ASOP Thin C ABI Outbound Passage
    ↓
ASAO Optional Add-ons / External Receivers
```

### 3.2 Thin C ABI Placement

The Thin C ABI boundary is positioned after:

1. stable outbound payload production,
2. bridge-mediated binding,
3. outbound port admission,
4. port-owned delivery round progression,
5. packet-readiness preparation,
6. ABI-readiness decision.

It is positioned before:

1. add-on execution,
2. external receiver interpretation,
3. database commit,
4. telemetry export,
5. downstream projection,
6. final receiver consumption,
7. destination retry policy.

### 3.3 Passage Entry

Thin C ABI passage may begin only when ASOP has a valid ABI-readiness state.

It must not begin from:

1. raw ASIP payload,
2. raw internal pipeline container,
3. ASCC Channel alone,
4. ASCC Binding declaration alone,
5. ASCC Session alone,
6. unadmitted Outbound Port material,
7. non-packet-ready delivery material,
8. add-on request alone.

### 3.4 Passage Exit

Thin C ABI passage exits with an ABI-side passage outcome only.

Valid exit outcomes include:

|        Record ID | Outcome             | Meaning                                                                          |
| ---------------: | ------------------- | -------------------------------------------------------------------------------- |
| ASOP-003-OUT-001 | `passage_accepted`  | The ABI boundary accepted the passage attempt                                    |
| ASOP-003-OUT-002 | `passage_rejected`  | The ABI boundary rejected the passage attempt under ABI-side precondition checks |
| ASOP-003-OUT-003 | `passage_failed`    | The boundary crossing failed within Thin C ABI responsibility                    |
| ASOP-003-OUT-004 | `passage_deferred`  | Passage cannot occur now without claiming downstream failure                     |
| ASOP-003-OUT-005 | `passage_completed` | The ABI handoff completed at the boundary level only                             |
| ASOP-003-OUT-006 | `passage_cancelled` | The passage attempt was cancelled under ASOP/ABI policy                          |

These outcomes do not mean:

1. add-on completed,
2. database committed,
3. telemetry exported,
4. receiver consumed,
5. external runtime completed,
6. downstream durable record created.

## 4. Core Definition

### 4.1 Thin C ABI

The **Thin C ABI** is a minimal, stable, C-compatible outbound passage boundary used to carry ASOP-prepared outbound material toward optional add-ons or external receivers.

It is thin because it must not contain destination logic.

It is C ABI because it must provide a low-friction boundary usable across language, runtime, or add-on boundaries.

It is outbound because it participates only in material leaving the ASOP-owned port side.

### 4.2 Thin C ABI Passage

**Thin C ABI Passage** is the act of presenting passage-safe outbound material through the Thin C ABI boundary after a Port-Owned Delivery Round has produced ABI readiness.

It is:

```text id="asop-003-passage-definition"
post-port-round,
ABI-boundary-owned,
logic-free,
destination-neutral,
passage-safe,
externally consumable,
authority-bounded.
```

### 4.3 Thin C ABI Formula

```text id="asop-003-formula"
Thin C ABI Outbound Passage
= ABI-Ready Port Material
+ Passage-Safe Packet View
+ Stable C-Compatible Boundary Shape
+ Ownership-Neutral Handles
+ Minimal Status / Error Surface
+ Boundary-Level Passage Trace
+ Destination-Semantic Exclusion
```

### 4.4 Thin C ABI Non-Formula

The following definitions are invalid:

```text id="asop-003-non-formula"
Thin C ABI = Port-Owned Delivery Round
Thin C ABI = Bridge Session
Thin C ABI = Bridge Protocol
Thin C ABI = Serialization Engine
Thin C ABI = Database Writer
Thin C ABI = Telemetry Exporter
Thin C ABI = Receiver Runtime
Thin C ABI = Add-on Execution Layer
Thin C ABI = Final Delivery Proof
```

## 5. Ownership Rules

### 5.1 Primary Ownership Rule

```text id="asop-003-primary-ownership-rule"
The Thin C ABI boundary belongs to ASOP as a passage boundary.
It does not own destination semantics beyond the ABI crossing.
```

ASOP owns:

1. the ABI passage contract,
2. the passage preconditions,
3. the passage-safe view shape,
4. the passage outcome categories,
5. boundary-level trace and diagnostics,
6. authority closure before add-on semantics.

ASOP does not own:

1. add-on execution,
2. external receiver interpretation,
3. database persistence,
4. telemetry export,
5. downstream retry behavior,
6. external runtime scheduling.

### 5.2 Thinness Rule

```text id="asop-003-thinness-rule"
The ABI boundary must remain thin: it may carry, expose, reference, or report;
it must not decide, transform, persist, query, retry, or interpret destination meaning.
```

The ABI may expose a stable packet view.

The ABI may return a passage status.

The ABI may expose a minimal error code.

The ABI may carry opaque handles.

The ABI must not perform destination-specific transformation.

### 5.3 C Compatibility Rule

```text id="asop-003-c-compatibility-rule"
The ABI shape must be representable through C-compatible constructs.
C++ ownership, templates, exceptions, references, STL internals, and object lifetimes
must not leak across the ABI boundary as uncontrolled public contract.
```

Allowed boundary concepts may include:

1. opaque handles,
2. plain status codes,
3. fixed-width scalar types,
4. pointer-and-size views,
5. explicit lifetime functions,
6. stable enum-like integer codes,
7. externally documented struct shapes where required.

Forbidden boundary leakage includes:

1. C++ exceptions crossing ABI,
2. STL containers as ABI surface,
3. C++ references as ABI surface,
4. templates as ABI surface,
5. CRTP types as ABI surface,
6. unowned internal pointers without lifetime contract,
7. implicit allocator ownership.

### 5.4 Destination Neutrality Rule

```text id="asop-003-destination-neutrality-rule"
Thin C ABI passage must not encode database, telemetry, projection,
language-runtime, or receiver-specific semantics as ASOP-owned meaning.
```

The ABI may carry a destination-neutral packet.

The receiver or add-on may interpret the packet.

ASOP must not become the interpreter.

### 5.5 Receiver Non-Ownership Rule

```text id="asop-003-receiver-nonownership-rule"
ABI passage completion is not receiver completion.
```

The ABI crossing may complete.

That does not mean:

1. receiver accepted,
2. receiver processed,
3. receiver persisted,
4. receiver acknowledged,
5. receiver projected,
6. receiver indexed.

## 6. Passage Preconditions

### 6.1 Required Preconditions

Thin C ABI outbound passage may begin only when the following conditions hold.

|        Record ID | Precondition                                      | Owner                | Required Meaning                                                      |
| ---------------: | ------------------------------------------------- | -------------------- | --------------------------------------------------------------------- |
| ASOP-003-PRE-001 | Stable outbound payload exists                    | ASIP                 | Payload material has been stabilized before outbound exposure         |
| ASOP-003-PRE-002 | Bridge-mediated binding exists                    | ASCC                 | Provider and Outbound Port were bound through ASCC                    |
| ASOP-003-PRE-003 | Port handoff reached ASOP                         | ASCC → ASOP          | The port received a valid bridge-mediated handoff                     |
| ASOP-003-PRE-004 | Port-Owned Delivery Round completed ABI readiness | ASOP                 | ASOP round produced `abi_ready` or equivalent                         |
| ASOP-003-PRE-005 | Packet-readiness state exists                     | ASOP                 | Material is organized for packet contract usage                       |
| ASOP-003-PRE-006 | ABI boundary contract is known                    | ASOP                 | The passage target is governed by a known ABI contract                |
| ASOP-003-PRE-007 | Packet lifetime policy is known                   | ASOP                 | The ABI view does not outlive its valid material                      |
| ASOP-003-PRE-008 | Ownership transfer / borrow rule is declared      | ASOP                 | ABI crossing knows whether it borrows, transfers, or copies           |
| ASOP-003-PRE-009 | Passage diagnostic policy is available            | ASOP                 | Passage outcome can be reported                                       |
| ASOP-003-PRE-010 | No destination-semantic claim is required         | ASOP / ASAO boundary | Passage does not require ASOP to interpret final destination behavior |

### 6.2 Forbidden Preconditions

The following must not be accepted as sufficient authorization for ABI passage.

|         Record ID | Invalid Precondition                                        | Reason                                                     |
| ----------------: | ----------------------------------------------------------- | ---------------------------------------------------------- |
| ASOP-003-FPRE-001 | Raw payload exists                                          | Bypasses ASIP stabilization                                |
| ASOP-003-FPRE-002 | PluginAdapter exists                                        | Adapter existence does not authorize ABI passage           |
| ASOP-003-FPRE-003 | Binding exists but port round has not reached ABI readiness | Skips ASOP delivery round                                  |
| ASOP-003-FPRE-004 | Add-on is configured                                        | Add-on configuration is not ABI passage authorization      |
| ASOP-003-FPRE-005 | Receiver requested material                                 | Receiver request does not bypass ASCC/ASOP governance      |
| ASOP-003-FPRE-006 | Database connection is open                                 | Persistence availability is not ASOP passage authorization |
| ASOP-003-FPRE-007 | Telemetry sink is reachable                                 | Telemetry reachability is destination semantics            |
| ASOP-003-FPRE-008 | External runtime is loaded                                  | Runtime availability does not prove packet readiness       |

## 7. Passage Lifecycle

### 7.1 Lifecycle Overview

```text id="asop-003-lifecycle"
1. Port-Owned Delivery Round reaches ABI readiness.
2. Thin C ABI passage context is opened.
3. Passage-safe packet view is resolved.
4. Lifetime and ownership rule is checked.
5. Boundary-compatible handle/view is prepared.
6. C-compatible status/error surface is initialized.
7. Passage function or boundary operation is invoked.
8. Boundary-level outcome is recorded.
9. Passage diagnostics are emitted.
10. ABI passage context is closed.
11. Destination lifecycle remains outside ASOP authority.
```

### 7.2 Lifecycle Table

|         Step ID | Step                       | Owner           | Output                     | Not Equivalent To             |
| --------------: | -------------------------- | --------------- | -------------------------- | ----------------------------- |
| ASOP-003-LC-001 | ABI readiness received     | ASOP            | Valid entry condition      | Final delivery                |
| ASOP-003-LC-002 | Passage context opened     | ASOP            | ABI context                | Port delivery round           |
| ASOP-003-LC-003 | Packet view resolved       | ASOP            | Passage-safe packet view   | Packet schema finalization    |
| ASOP-003-LC-004 | Lifetime checked           | ASOP            | Lifetime-valid view        | Receiver ownership            |
| ASOP-003-LC-005 | Handle/view prepared       | ASOP            | ABI-compatible reference   | C++ object export             |
| ASOP-003-LC-006 | Status surface initialized | ASOP            | Status/error target        | Telemetry event               |
| ASOP-003-LC-007 | ABI crossing invoked       | ASOP / boundary | Passage attempt            | Add-on execution              |
| ASOP-003-LC-008 | Outcome recorded           | ASOP            | Passage outcome            | Receiver acknowledgement      |
| ASOP-003-LC-009 | Diagnostics emitted        | ASOP            | Boundary diagnostic record | Downstream audit              |
| ASOP-003-LC-010 | Passage context closed     | ASOP            | Closed passage state       | Destination lifecycle closure |

Terminal states are:

```text id="asop-003-terminal-states"

passage_rejected
passage_failed
passage_deferred
closed
cancelled
```

A terminal passage state is not a destination terminal state.

## 10. Lifetime and Ownership Semantics

### 10.1 Lifetime Requirement

Thin C ABI passage must make payload lifetime explicit.

No ABI consumer may infer ownership from a pointer alone.

No ABI consumer may retain borrowed material unless the lifetime contract permits it.

No ABI passage may expose a view whose backing storage may be recycled before the declared passage window closes.

### 10.2 Candidate Lifetime Modes

|         Record ID | Lifetime Mode           | Meaning                                                                          |
| ----------------: | ----------------------- | -------------------------------------------------------------------------------- |
| ASOP-003-LIFE-001 | `borrowed_view`         | ABI consumer may read during a declared valid window only                        |
| ASOP-003-LIFE-002 | `retained_handle`       | ABI consumer receives a handle whose release is governed by explicit release API |
| ASOP-003-LIFE-003 | `copied_packet`         | ABI receives a copy independent from ASOP internal storage                       |
| ASOP-003-LIFE-004 | `transferred_buffer`    | Ownership of a buffer moves across the ABI boundary under explicit release rules |
| ASOP-003-LIFE-005 | `materialized_snapshot` | ABI receives a stable snapshot detached from ASIP/ASOP mutable lifecycle         |

### 10.3 Lifetime Non-Claims

Lifetime validity does not claim:

1. receiver success,
2. destination persistence,
3. external runtime health,
4. add-on semantic correctness,
5. downstream query readiness.

### 10.4 Recycling Boundary

ASOP must not release, recycle, or invalidate passage material until its own ABI lifetime obligations are satisfied.

However, satisfying ABI lifetime obligations does not prove downstream consumption.

## 14. Versioning and Compatibility

### 14.1 ABI Version Rule

```text id="asop-003-version-rule"
Thin C ABI passage must include an explicit version or compatibility indicator.
```

The version indicator is required because ABI boundaries are long-lived and must remain stable across:

1. C++ implementation changes,
2. add-on adapter changes,
3. external receiver evolution,
4. packet contract revisions,
5. language/runtime boundary changes.

### 14.2 Compatibility Scope

ABI compatibility covers:

1. struct layout expectations where applicable,
2. enum/status code stability,
3. pointer/size interpretation,
4. lifetime/release contract,
5. version tag recognition,
6. packet descriptor compatibility.

ABI compatibility does not cover:

1. destination schema compatibility,
2. database migration compatibility,
3. telemetry backend compatibility,
4. external runtime semantic compatibility,
5. receiver business-rule compatibility.

### 14.3 Compatibility Outcomes

|         Record ID | Outcome                   | Meaning                                         |
| ----------------: | ------------------------- | ----------------------------------------------- |
| ASOP-003-COMP-001 | `compatible`              | ABI passage may proceed                         |
| ASOP-003-COMP-002 | `compatible_with_warning` | ABI passage may proceed with diagnostic warning |
| ASOP-003-COMP-003 | `incompatible`            | ABI passage must not proceed                    |
| ASOP-003-COMP-004 | `unknown_version`         | Version cannot be recognized                    |
| ASOP-003-COMP-005 | `unsupported_layout`      | Packet/layout expectations cannot be satisfied  |

## 15. Diagnostics and Trace

### 15.1 Diagnostic Role

Thin C ABI passage diagnostics must explain:

1. why passage was attempted,
2. which port owned the passage,
3. which delivery round produced ABI readiness,
4. which packet view was used,
5. which lifetime policy applied,
6. which ABI version was targeted,
7. whether passage completed at the boundary level,
8. whether passage failed within ABI responsibility,
9. which authority boundary remains closed.

### 15.2 Diagnostic Non-Role

ABI passage diagnostics are not:

1. telemetry export,
2. downstream logging framework output,
3. database audit record,
4. receiver acknowledgement,
5. persistent delivery proof,
6. final add-on trace,
7. external runtime trace.

### 15.3 Trace Fields

|        Record ID | Trace Field                     | Purpose                                              |
| ---------------: | ------------------------------- | ---------------------------------------------------- |
| ASOP-003-TRC-001 | `passage_id`                    | Identify the passage attempt                         |

## 19. Final Governance Statement

Thin C ABI Outbound Passage is the ASOP-owned, logic-free boundary crossing that carries ABI-ready outbound material from an Outbound Port toward optional add-ons or external receivers.

It is not the producer of the payload.

It is not the bridge binding.

It is not the port-owned delivery round.

It is not the destination executor.

It is not the final receiver.

It must remain:

```text id="asop-003-final-positive"
thin,
C-compatible,
post-port-round,
passage-safe,
lifetime-explicit,
version-aware,
destination-neutral,
diagnostic-capable,
and authority-bounded.
```

It must not become:

```text id="asop-003-final-negative"
ASIP execution,
ASCC binding,
ASCC session,
Bridge Protocol,
Bridge Core,
Port-Owned Delivery Round,
serialization engine,
database writer,
telemetry exporter,
external runtime adapter semantics,
receiver lifecycle,
or final delivery proof.
```

The governing rule is therefore:

```text id="asop-003-governing-rule"
ASIP produces stable outbound payload.
ASCC binds the payload provider to the outbound port.
ASOP delivery rounds prepare ABI readiness.
Thin C ABI passage carries boundary-safe material across the ABI edge.
ASAO or external receivers own destination-specific behavior beyond that edge.
```
