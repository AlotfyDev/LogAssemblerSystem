## 2. Document Purpose

### 2.1 Purpose Statement

This document defines the phased implementation plan for the Assembler System Independent Pipeline.

It answers the question:

```text
In what dependency-first order should ASIP be implemented so that primitive
foundations, internal mechanics, exposure surfaces, lifecycle rules, pipeline
facade, and bridge-facing readiness are introduced without violating ASIP
authority boundaries or prematurely implementing ASCC, ASOP, ASAO, or receiver
semantics?
````

This document is not implementation code.

It is not a final file inventory.

It is not a final class hierarchy.

It is the implementation sequencing authority for ASIP.

### 2.2 Why This Plan Exists

ASIP has been stabilized as the domain that owns the Independent Pipeline.

The Independent Pipeline:

1. owns internal payload-production mechanics,
2. produces stable outbound payloads,
3. exposes stable material toward a PluginAdapter-facing boundary,
4. does not model final destination semantics,
5. does not own bridge execution,
6. does not own outbound port delivery,
7. does not own Thin C ABI passage,
8. does not own add-on behavior,
9. does not own final receiver lifecycle.

Implementation must therefore avoid starting with a large `TIndependentPipeline` object.

That would invert the dependency order.

The correct implementation strategy is:

```text id="asip-impl-000-core-order"
primitive foundations
→ internal payload/material primitives
→ admission and validation primitives
→ container and slot mechanics
→ metadata/timestamp preparation
→ internal coordination mechanics
→ stable outbound exposure
→ lifecycle release/recycling
→ independent pipeline facade
→ bridge-facing readiness
```

## 3. Implementation Doctrine

### 3.1 Dependency-First Rule

```text id="asip-impl-000-dependency-first-rule"
Write dependency foundations first.
Then write the abstractions that depend on them.
Then write internal mechanisms.
Then write the owning facade.
Then write integration-boundary readiness.
```

### 3.2 No Facade-First Rule

```text id="asip-impl-000-no-facade-first-rule"
Do not start ASIP implementation by writing `TIndependentPipeline.hpp`.
```

The pipeline facade depends on lower layers:

1. identifiers,
2. status types,
3. result/error types,
4. configuration objects,
5. admission results,
6. payload descriptors,
7. container and slot primitives,
8. validation reports,
9. metadata and timestamp preparation records,
10. exposure views,
11. lifecycle claims,
12. release/recycling decisions.

Therefore the facade must come late.

### 3.3 No Destination Semantics Rule

```text id="asip-impl-000-no-destination-semantics-rule"
ASIP implementation must not contain destination-specific semantics.
```

ASIP code must not implement:

1. database writes,
2. telemetry export,
3. file sinks,
4. receiver retry,
5. Thin C ABI passage,
6. outbound port delivery,
7. bridge orchestration,
8. add-on projection,
9. external runtime lifecycle,
10. final receiver acknowledgement.

### 3.4 Header-Only-First Rule

```text id="asip-impl-000-header-only-first-rule"
ASIP implementation waves should prefer header-only C++17 artifacts where
appropriate until the implementation boundary requires otherwise.
```

This preserves compile-smoke-test simplicity and keeps the early implementation phase close to the ASCC precedent.

### 3.5 Compile-Smoke-Test Rule

Every implementation wave must include:

1. minimal smoke test,
2. `g++ -std=c++17` compile command,
3. dependency notes,
4. changed files manifest,
5. deferred decisions,
6. validation notes,
7. tree listing,
8. wave README.

## 4. Planned Wave Structure

### 4.1 Wave Naming

Implementation waves use this pattern:

```text id="asip-impl-000-wave-naming"
ASIP-IMPL-W##_Wave_Name
```

Each wave should produce a zip package or repository patch containing:

```text id="asip-impl-000-wave-package-shape"
include/assembler/independent_pipeline/...
tests/...
W##_FILE_HEADER_DOCTRINE.md
WAVE_README.md
WAVE_MANIFEST.json
CHANGED_FILES.txt
DEPENDENCY_NOTES.md
DEFERRED_DECISIONS.md
VALIDATION_NOTES.md
COMPILE_CHECK.json
TREE.txt
```

### 4.2 Namespace Baseline

The default namespace should be:

```cpp id="asip-impl-000-namespace"
namespace assembler::independent_pipeline
{
    // ...
}
```

Specialized sub-namespaces may be introduced only when they clarify ownership.

### 4.3 Include Boundary Baseline

ASIP core implementation must not include:

```text id="asip-impl-000-forbidden-core-includes"
assembler/communication_context/...
assembler/outbound_ports/...
assembler/optional_addons/...
assembler/thin_c_abi/...
assembler/database/...
assembler/telemetry/...
```

Bridge-facing readiness may define adapter-neutral surfaces later, but core ASIP must remain independent.

## 22. Final Governance Statement

ASIP implementation must proceed from dependency foundations to the owning facade.

The implementation plan must preserve this order:

```text id="asip-impl-final-order"
foundations first
mechanics second
stable exposure third
lifecycle fourth
facade fifth
execution harness sixth
bridge-facing readiness last
```

The central rule is:

```text id="asip-impl-final-rule"
ASIP produces and exposes stable outbound payloads.
ASIP does not deliver them to final destinations.
```

Therefore this plan authorizes phased ASIP implementation readiness, not unrestricted implementation of bridge orchestration, outbound delivery, Thin C ABI passage, optional add-ons, or receiver lifecycle.
