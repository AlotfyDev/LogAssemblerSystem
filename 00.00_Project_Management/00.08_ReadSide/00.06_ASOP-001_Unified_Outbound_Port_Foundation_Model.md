## 2. Document Purpose

### 2.1 Purpose Statement

This document establishes the **Unified Outbound Port** as the foundational ASOP concept.

It answers the question:

```text
What is an Outbound Port, what does it own, where does it begin after ASCC
bridge-mediated binding, where does its authority end before optional add-ons
or external receivers, and why must it not be collapsed into the bridge,
the pipeline, the Thin C ABI, or final destination semantics?
````

The Outbound Port is the ASOP-owned surface that receives bridge-mediated outbound material after ASCC has completed its binding, session, protocol, and bridge-core responsibilities.

The Outbound Port is not:

1. the Independent Pipeline,
2. the stable payload producer,
3. the Payload-Providing PluginAdapter,
4. the Communication Bridge,
5. the ASCC Binding,
6. the ASCC Session,
7. the Bridge Protocol,
8. the Bridge Core,
9. the Thin C ABI itself,
10. an add-on,
11. a database writer,
12. a telemetry exporter,
13. a final receiver lifecycle owner.

It is the port-side delivery authority that prepares outbound material for controlled passage through the ASOP-owned outbound path.

### 2.2 Why This Document Exists

The architecture now separates outbound responsibility into four distinct ownership zones:

```text id="4nzo0v"
ASIP produces stable outbound payloads.
ASCC binds payload providers to outbound ports.
ASOP owns outbound ports and Thin C ABI passage.
ASAO owns optional destination-specific behavior beyond the ABI.
```

`ASIP` does not own final destinations or delivery. It ends at stable outbound payload exposure.

`ASCC` does not own outbound delivery semantics. It binds a stable outbound payload provider to an outbound port without absorbing outbound delivery or Thin C ABI passage.

`ASOP` therefore requires its own foundation document.

This document defines that foundation.

### 2.3 Position After ASCC-REV-001

`ASCC-REV-001` stabilized the rule:

```text id="tnhvw9"
ASCC binds.
ASOP delivers through ports and ABI.
```

This document starts the ASOP side of that split.

It does not revise ASCC.

It consumes the ASCC output:

```text id="ere3k7"
Governed Outbound Binding
```

and defines the ASOP-owned participant that receives it:

```text id="oi5qes"
Unified Outbound Port
```

### 2.4 Non-Purpose

This document does not define:

1. final C++ headers,
2. final class declarations,
3. final method signatures,
4. final delivery-loop implementation,
5. final ABI packet binary layout,
6. final allocator or memory-pool design,
7. final callback ABI,
8. final add-on contract,
9. database persistence semantics,
10. telemetry export semantics,
11. external runtime ownership,
12. receiver retry semantics,
13. production scheduler behavior.

Those belong to later ASOP and ASAO documents.

## 3. Architectural Position

### 3.1 Corrected Outbound Chain

The corrected outbound architecture is:

```text id="ksoerc"
ASIP Independent Pipeline
    ↓
Stable Outbound Payload Exposure
    ↓
ASCC Payload-Providing PluginAdapter
    ↓
ASCC Bridge-Mediated Binding
    ↓
ASOP Unified Outbound Port
    ↓
ASOP Port-Owned Delivery Round
    ↓
ASOP Thin C ABI Outbound Passage
    ↓
ASAO Optional Add-ons / External Receivers
```

### 3.2 ASOP Entry Point

ASOP begins after ASCC has produced a valid bridge-mediated binding.

The ASOP entry point is not raw payload exposure.

The ASOP entry point is not the PluginAdapter.

The ASOP entry point is not the Channel.

The ASOP entry point is not compatibility evaluation.

The ASOP entry point is:

```text id="igrh2t"
a bound outbound port surface that is eligible to begin port-owned delivery preparation.
```

### 3.3 ASOP Exit Point

ASOP exits at the boundary where Thin C ABI passage has transferred or exposed outbound material to the next external or add-on-owned layer.

The ASOP exit point is not final destination success.

The ASOP exit point is not database commit.

The ASOP exit point is not telemetry export completion.

The ASOP exit point is not receiver consumption.

The ASOP exit point is:

```text id="wbb7zn"
completion of ASOP-owned outbound passage obligations through the Thin C ABI boundary.
```

## 4. Core Thesis

### 4.1 Thesis Statement

A Unified Outbound Port is the ASOP-owned delivery surface that receives bridge-mediated outbound material and governs its port-side progression toward Thin C ABI passage.

It is unified because the same foundation applies regardless of whether the eventual add-on or external receiver is:

1. read-only projection,
2. database persistence,
3. telemetry export,
4. external runtime receiver,
5. cross-language consumer,
6. local diagnostic sink,
7. test harness,
8. future specialized receiver.

The Outbound Port unifies the port-side contract.

It does not unify destination semantics.

### 4.2 Port Formula

```text id="dq7qw5"
Unified Outbound Port
= Bridge-Facing Port Surface
+ Port-Owned Admission
+ Delivery-Round Eligibility
+ Outbound Payload Packet Preparation Boundary
+ Thin C ABI Passage Adjacency
+ Authority Closure Before Add-on Semantics
```

### 4.3 Port Non-Formula

The following are invalid definitions:

```text id="1n9hn1"
Outbound Port = Communication Bridge
Outbound Port = ASCC Binding
Outbound Port = Session
Outbound Port = Thin C ABI
Outbound Port = Add-on
Outbound Port = Final Receiver
Outbound Port = Database Writer
Outbound Port = Telemetry Exporter
Outbound Port = External Runtime
```

## 5. Terminology

### 5.1 Unified Outbound Port

A Unified Outbound Port is an ASOP-owned boundary surface that accepts bridge-mediated outbound material and initiates port-owned delivery preparation.

It is unified across possible outbound destination categories, but it does not encode destination-specific semantics.

### 5.2 Bridge-Facing Port Surface

The Bridge-Facing Port Surface is the part of the Outbound Port visible to ASCC binding.

It exposes enough obligation information for ASCC to validate compatibility, form a binding, open a session, and run bridge protocol steps.

It does not expose port internals.

### 5.3 Port-Owned Delivery

Port-Owned Delivery is the ASOP-owned progression of outbound material after bridge-mediated handoff into the Outbound Port.

It begins after ASCC binding and before Thin C ABI passage.

It is not ASCC bridge orchestration.

It is not add-on execution.

### 5.4 Port-Owned Delivery Round

A Port-Owned Delivery Round is the ASOP-managed unit of delivery progression inside an outbound port.

It will be defined fully in:

```text id="xvubnr"
ASOP-002_Port_Owned_Delivery_Round_Model.md
```

This document only establishes that such rounds belong to ASOP, not ASCC or ASIP.

### 5.5 Outbound Payload Packet

An Outbound Payload Packet is the ASOP-prepared passage unit intended for Thin C ABI outbound passage.

It is not the original ASIP stable payload.

It is not a final receiver object.

It is not database record material by default.

It will be defined in a later packet-contract document.

### 5.6 Thin C ABI Outbound Passage

Thin C ABI Outbound Passage is the ASOP-owned logic-free boundary protocol through which outbound payload packets cross into add-on or external receiver space.

It is adjacent to the Outbound Port but must not be collapsed into the port itself.

### 5.7 Optional Add-on

An Optional Add-on is an ASAO-owned destination-specific extension beyond the Thin C ABI.

Examples may include database writers, telemetry exporters, external language receivers, or projection builders.

Add-ons are not ASOP ports.

## 6. Governing Rules

### 6.1 Port Ownership Rule

```text id="2y2zgr"
The Outbound Port owns port-side delivery preparation after ASCC binding.
```

It does not own ASIP payload production.

It does not own ASCC binding.

It does not own ASAO destination semantics.

### 6.2 Bridge Separation Rule

```text id="zqfnyt"
Bridge binding is not port delivery.
```

ASCC may bind to a port surface.

ASCC may validate port obligations.

ASCC may run a bridge-core model step.

ASCC must not execute the port-owned delivery round.

### 6.3 Pipeline Separation Rule

```text id="2q574j"
Stable payload exposure is not outbound port delivery.
```

ASIP produces stable outbound payloads.

ASOP receives bridge-mediated outbound material after ASCC binding.

The Outbound Port must not reach backward into ASIP internal containers, slots, admission mechanics, recycling structures, or pipeline execution state.

### 6.4 ABI Separation Rule

```text id="h8xsto"
An Outbound Port prepares for Thin C ABI passage.
It is not itself the Thin C ABI.
```

The port may produce or prepare an outbound payload packet.

The Thin C ABI governs the logic-free outbound passage mechanism.

The ABI layer must remain separately documented.

### 6.5 Destination Separation Rule

```text id="jgdz96"
An Outbound Port is not a final receiver.
```

The port may deliver toward a boundary.

It must not claim database write success, telemetry export success, external receiver consumption, or add-on lifecycle completion.

### 6.6 Unified Port Rule

```text id="ewd8le"
Unified means shared outbound port doctrine, not shared destination semantics.
```

The same port doctrine can govern different outbound target categories.

Those target categories remain beyond ASOP unless explicitly defined as ASOP-owned passage mechanics.

### 6.7 Authority Closure Rule

```text id="ma4wvb"
ASOP authority closes at completion of ASOP-owned port and Thin C ABI passage obligations.
```

After that point, ASAO or external receiver authority begins.

## 8. Port Boundary Model

### 8.1 Inbound Side of the Port

The inbound side of the Outbound Port receives material only after:

1. ASIP has stabilized payload exposure,
2. ASCC has represented the payload through a PluginAdapter-facing provider surface,
3. ASCC has paired provider and port obligations,
4. ASCC compatibility evaluation has passed or been accepted under policy,
5. ASCC has created a binding declaration,
6. ASCC has opened or validated a session scope,
7. ASCC has applied a bridge protocol model,
8. ASCC Bridge Core has completed its endpoint-neutral model step.

The port does not accept arbitrary raw payloads.

### 8.2 Internal Side of the Port

The port internally owns:

1. port admission state,
2. port readiness state,
3. port delivery eligibility,
4. port delivery round state,
5. port-side packet preparation state,
6. port diagnostics,
7. port failure categories,
8. port-owned release or rejection conditions.

The details of these internal structures are deferred to later ASOP documents.

### 8.3 Outbound Side of the Port

The outbound side of the port prepares material for:

```text id="iup49e"
Thin C ABI Outbound Passage
```

The port does not decide what the add-on or receiver does after ABI passage.

## 9. Unified Outbound Port Lifecycle

### 9.1 Lifecycle Overview

```text id="4a7xb4"
1. Port surface is declared.
2. Port obligations are exposed to ASCC.
3. ASCC forms a governed binding.
4. Bridge-mediated material reaches the port surface.
5. Port admission evaluates the inbound material.
6. Port delivery eligibility is established.
7. Port-owned delivery round begins.
8. Outbound payload packet is prepared or authorized.
9. Thin C ABI passage is invoked or made eligible.
10. ASOP authority closes before add-on destination semantics.
```

### 9.2 Lifecycle Table

| ASOP-001-LC-003 | Binding completion             | ASCC               | Governed outbound binding         | Port delivery              |
| ASOP-001-LC-004 | Port-side admission            | ASOP               | Admission decision                | ASIP admission             |

## 11. Port Kinds

### 11.1 Purpose of Port Kinds

Port kinds classify outbound ports by passage posture, not by final destination ownership.

A port kind helps ASOP reason about port-side delivery obligations.

It must not turn ASOP into the owner of add-on semantics.

## 12. Relationship to ASCC Binding

### 12.1 What ASCC Sees

ASCC sees:

1. port identity,
2. bridge-facing port obligation surface,
3. compatibility-relevant constraints,
4. binding eligibility,
5. session-relevant surface,
6. protocol-relevant surface.

ASCC does not see:

1. port internal delivery queue,
2. port delivery round internals,
3. packet construction internals,
4. Thin C ABI internal passage rules,
5. add-on receiver behavior.

### 12.2 What ASOP Receives from ASCC

ASOP receives the result of governed binding:

```text id="fk62x9"
a bridge-mediated, compatibility-checked, session-scoped, protocol-governed
handoff into the outbound port surface.
```

ASOP does not receive:

1. raw ASIP internals,
2. arbitrary endpoint invocation,
3. destination semantics,
4. receiver success state.

### 12.3 Boundary Handoff Statement

```text id="asop-001-boundary-handoff"
ASCC hands into the outbound port surface.
ASOP owns what happens inside the port after that handoff.
ASOP then hands toward Thin C ABI passage.
ASAO or external receivers own what happens beyond that passage.
```

## 13. Relationship to Thin C ABI

### 13.1 ABI Adjacency

The Outbound Port is adjacent to the Thin C ABI.

Adjacency means:

1. the port prepares material for ABI passage,
2. the port may produce an outbound payload packet candidate,
3. the port may validate ABI passage readiness,
4. the port may invoke or delegate to an ABI passage boundary.

Adjacency does not mean:

1. the port is the ABI,
2. the ABI contains port delivery logic,
3. the ABI owns add-on semantics,
4. ABI passage proves receiver consumption.

### 13.2 ABI Non-Ownership Rule

```text id="vvc3od"
Thin C ABI is a logic-free outbound passage protocol.
It must not absorb port delivery logic or add-on destination semantics.
```

The detailed ABI model is deferred to:

```text id="5e0fgy"
ASOP-003_Thin_C_ABI_Outbound_Passage_Model.md
```

## 14. Relationship to ASAO

### 14.1 ASAO Starts Beyond ABI

ASAO starts beyond Thin C ABI passage.

ASAO may own optional behavior such as:

1. database writing,
2. telemetry exporting,
3. external language receiver adaptation,
4. read-only projection,
5. diagnostic sink handling,
6. external runtime delivery semantics.

### 14.2 ASOP Must Not Pre-Own ASAO

ASOP must not define:

1. database schemas,
2. telemetry exporter protocols,
3. receiver retry policies,
4. destination lifecycle,
5. add-on internal state,
6. external runtime ownership,
7. query model semantics.

### 14.3 ASOP / ASAO Rule

```text id="hvj589"
ASOP prepares and passes.
ASAO interprets and owns destination-specific behavior beyond the ABI.
```

## 18. Final Governance Statement

The Unified Outbound Port is the ASOP-owned delivery surface that starts after ASCC has completed bridge-mediated binding and ends before ASAO or external receiver semantics begin.

It must remain:

```text id="qq9k9s"
port-owned,
bridge-compatible,
delivery-round-capable,
ABI-adjacent,
destination-neutral,
and authority-bounded.
```

It must not become:

```text id="uc5vwa"
the Independent Pipeline,
the Communication Bridge,
the ASCC Binding,
the ASCC Session,
the Bridge Protocol,
the Bridge Core,
the Thin C ABI itself,
an add-on,
a database writer,
a telemetry exporter,
an external runtime,
or a final receiver lifecycle owner.
```

The outbound ownership rule is therefore:

```text id="u3wjvm"
ASIP produces stable payloads.
ASCC binds provider to port.
ASOP owns outbound port delivery and Thin C ABI passage.
ASAO owns optional destination semantics beyond ABI.
```
