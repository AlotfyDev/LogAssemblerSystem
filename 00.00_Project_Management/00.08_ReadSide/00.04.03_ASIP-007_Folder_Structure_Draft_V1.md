## 2. Document Purpose

### 2.1 Purpose Statement

This document defines the first folder-structure draft for the Assembler System Independent Pipeline domain.

It answers the question:

```text
Given the stabilized ASIP architecture, which folders should represent the
Independent Pipeline as a digital twin before final file inventory is derived?
````

This document is a folder taxonomy draft.

It does not create final implementation files.

It does not freeze class names.

It does not define final CMake targets.

It does not introduce bridge, outbound port, Thin C ABI, add-on, or receiver folders inside ASIP.

### 2.2 Why This Document Comes Now

The ASIP sequence has stabilized the main architectural layers:

1. `ASIP-001` established the Independent Pipeline as the primary unit of correctness, ownership, operation, and encapsulation.
2. `ASIP-002` defined internal mechanics: admission, preparation, containers, slots, rounds, waiting lists, stabilization, and internal dispatch readiness.
3. `ASIP-003` defined stable outbound payload exposure as the terminal ASIP boundary.
4. `ASIP-004` defined encapsulation and configuration boundaries.
5. `ASIP-005` defined single-writer execution and independent pipeline instances.
6. `ASIP-006` defined admission, retention, release, expiration, and recycling lifecycle boundaries.

The next correct step is not file inventory.

The next correct step is folder structure.

### 2.3 Folder-First Rule

```text id="asip-007-folder-first-rule"
ASIP folders are stabilized before ASIP files.
```

The folder structure must first answer:

1. what ASIP owns,
2. what ASIP does not own,
3. where internal pipeline mechanics live,
4. where stable outbound payload exposure lives,
5. where lifecycle state and recycling rules live,
6. where configuration lives,
7. where docs and tests live,
8. which folders are forbidden because they would collapse ASIP into downstream delivery.

### 2.4 ASIP Folder Meaning

An ASIP folder is not a convenience bucket.

An ASIP folder is a semantic owner.

A folder may exist only when it represents a stable ASIP-owned responsibility, object category, lifecycle region, boundary, or validation concern.

## 4. ASIP Root Placement

### 4.1 Root Folder

The ASIP root implementation folder is:

```text id="asip-007-root-folder"
assembler/independent_pipeline/
```

### 4.2 Root Folder Meaning

`independent_pipeline/` owns the Independent Pipeline domain.

It represents the ASIP-bounded unit that:

1. admits material,
2. prepares material,
3. coordinates internal payload-production mechanics,
4. stabilizes outbound payloads,
5. exposes stable outbound payload material,
6. retains exposed material under ASIP-owned rules,
7. releases material under ASIP-owned conditions,
8. recycles ASIP-owned internal structures.

### 4.3 Root Folder Non-Meaning

`independent_pipeline/` does not own:

1. ASCC bridge orchestration,
2. outbound port delivery rounds,
3. Thin C ABI passage,
4. add-on semantics,
5. database persistence,
6. telemetry export,
7. cross-language receiver lifecycle,
8. final receiver acknowledgement.

### 4.4 Root Folder Rule

```text id="asip-007-root-folder-rule"
The ASIP root folder owns payload production lifecycle.
It must not own destination delivery lifecycle.
```

## 20. Dependency Direction

### 20.1 Folder Dependency Principle

```text id="asip-007-dependency-principle"
Dependency first, then what depends on it.
Public boundary and contracts before internal derivation.
Private detail last and never upstream.
```

### 20.2 Suggested Dependency Order

| Order | Folder Group     | May Depend On                                                                |
| ----: | ---------------- | ---------------------------------------------------------------------------- |
|     1 | `public/`        | Stable contract vocabulary only                                              |
|     2 | `contracts/`     | `public/` vocabulary where needed                                            |
|     3 | `configuration/` | `public/`, `contracts/`                                                      |
|     4 | `execution/`     | `public/`, `contracts/`, `configuration/`                                    |
|     5 | `admission/`     | `public/`, `contracts/`, `configuration/`, `execution/`                      |

## 22. Root-Level Anti-Collapse Rules

### 22.1 Independent Pipeline vs Write Side

```text id="asip-007-anticollapse-001"
Independent Pipeline ≠ legacy Write Side package.
```

The Independent Pipeline may reuse write-side mechanics, but it is a new bounded pipeline object model.

### 22.2 Independent Pipeline vs Read Side

```text id="asip-007-anticollapse-002"
Independent Pipeline ≠ Read Side router.
```

ASIP produces stable outbound payloads. It does not own read-only or end-receiver paths.

### 22.3 Stable Payload Exposure vs Delivery

```text id="asip-007-anticollapse-003"
Stable outbound payload exposure ≠ final delivery.
```

Exposure ends ASIP’s production boundary.

### 22.4 Bridge-Facing Claim vs Receiver Consumption

```text id="asip-007-anticollapse-004"
Bridge-facing claim ≠ receiver consumption.
```

Claim may affect ASIP release, but does not prove downstream success.

### 22.5 Recycling vs Delivery Proof

```text id="asip-007-anticollapse-005"
Recycling ≠ delivery proof.
```

Recycling only means ASIP-owned release conditions were satisfied.

### 22.6 Runtime Containers vs Message Broker

```text id="asip-007-anticollapse-006"
ASIP containers ≠ message broker.
```

Containers support internal lifecycle, not generic message routing.

### 22.7 Diagnostics vs Telemetry Export

```text id="asip-007-anticollapse-007"
ASIP diagnostics ≠ telemetry exporter.
```

Diagnostics may later feed telemetry, but do not implement external telemetry.
