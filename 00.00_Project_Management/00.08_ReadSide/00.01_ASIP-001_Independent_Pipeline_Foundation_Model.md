## 2. Document Purpose

### 2.1 Purpose Statement

This document establishes the **Independent Pipeline** as the primary unit of correctness, ownership, operation, and future encapsulation inside the Assembler System.

It answers the foundational question:

```text
What is the Independent Pipeline, what does it own, what does it produce,
where does its authority end, and why must it not model final destination
semantics?
````

The Independent Pipeline is not a read-side router.

It is not a destination dispatcher.

It is not a database adapter.

It is not an OpenTelemetry adapter.

It is not a read-only reader container.

It is not a cross-language receiver.

It is the internal pipeline object that receives or assembles content, executes its own internal mechanics, stabilizes outbound material, and exposes that outbound payload through a PluginAdapter-facing boundary.

Destination-specific meaning belongs beyond the pipeline.

### 2.2 Position in the Documentation Set

This document starts the ASIP documentation sequence.

The intended ASIP sequence is:


### 2.3 Relationship to ASCC, ASOP, and ASAO

The stable model is divided as follows:

```text
ASIP
    Independent Pipeline
    Internal mechanics
    Stable outbound payload exposure

ASCC
    Communication Bridge
    PluginAdapter / Port binding
    Mechanical bridge-mediated delivery

ASOP
    Outbound Ports
    Port-owned delivery rounds
    Thin C ABI outbound passage

ASAO
    Optional Add-ons beyond Thin C ABI
    Database adapters
    OpenTelemetry adapters

    Other receiver-specific extensions
```

The Independent Pipeline does not replace the Communication Context.

The Independent Pipeline does not absorb Outbound Ports.

The Independent Pipeline does not implement optional add-ons.

It only produces stable outbound payloads and exposes them toward the communication boundary.

## 3. Background and Supersession Context

### 3.1 Previous Read Side Direction

Earlier modeling explored a Read Side as a distinct internal layer that would:

1. receive from the Write Side,
2. stabilize containers,
3. fork material into read-only readers and end receivers,
4. manage dual dispatching,
5. possibly copy between dispatching rounds,
6. own internal read-side routing decisions.

That model was useful as an exploratory stage.

However, later modeling clarified that this would introduce unnecessary duplication and would pull destination semantics into the wrong layer.

### 3.2 Supersession Decision

The previous Read Side internal routing model is superseded by the Independent Pipeline model.

The stable decision is:

```text
The core pipeline does not know whether the final destination is:
- read-only readers,
- end receivers,
- database,
- graph database,
- OpenTelemetry,
- Python object,
- C# object,
- MQL5 receiver,
- shared file map writer,
- memory cache container,
- Chronicle-like store,
- or any other receiver.
```

The pipeline only produces stable outbound payloads.

The bridge and outbound ports handle delivery.

Anything beyond Thin C ABI owns destination semantics.

### 3.3 Why the Supersession Is Necessary

The older Read Side model risked creating these problems:

|         Record ID | Risk                         | Explanation                                                                              |
| ----------------: | ---------------------------- | ---------------------------------------------------------------------------------------- |
| ASIP-001-RISK-001 | Destination leakage          | Read-only and end-receiver semantics could leak into pipeline internals                  |
| ASIP-001-RISK-002 | Dual-dispatch rigidity       | The system could incorrectly require two dispatch paths even when one is enough          |
| ASIP-001-RISK-003 | Premature copying            | The system could copy between internal rounds before knowing whether a port requires it  |
| ASIP-001-RISK-004 | Bridge bypass                | The pipeline could begin acting like a destination-aware bridge                          |
| ASIP-001-RISK-005 | Port responsibility collapse | Port-owned mechanics could be pulled back into the pipeline                              |
| ASIP-001-RISK-006 | Add-on confusion             | Database, telemetry, or language-specific logic could be mistaken as core pipeline logic |

The Independent Pipeline model removes these risks by enforcing a strict separation:

```text
Pipeline produces payload.
Bridge binds and delivers.
Port owns outbound delivery mechanics.
Thin C ABI passes payload outward.
Add-ons / receivers interpret payload.
```

## 4. Core Thesis

### 4.1 Primary Thesis

```text
The Independent Pipeline is the unit of correctness, ownership, and encapsulated
operation for a single writer-oriented payload production path.
```

### 4.2 Expanded Thesis

An Independent Pipeline is a complete internal pipeline object that owns its own internal mechanics and produces stable outbound payload material.

It may be configured.

It may be instantiated per writer.

It may expose payloads through a PluginAdapter-facing boundary.

It may participate in bridge-mediated delivery.

It must not know final destination semantics.

It must not encode read-only reader behavior.

It must not encode database behavior.

It must not encode OpenTelemetry behavior.

It must not encode language-specific receiver behavior.

It must not expose internal container, slot, round, or waiting-list details through outbound delivery contracts.

### 4.3 One-Line Definition

```text
Independent Pipeline = encapsulated internal payload-production mechanism
that ends at stable outbound payload exposure.
```

## 5. Independent Pipeline Identity

### 5.1 What the Independent Pipeline Is

An Independent Pipeline is:

1. a self-contained internal mechanism,
2. a unit of correctness,
3. a unit of ownership,
4. a unit of future configuration,
5. a unit of execution planning,
6. a unit of writer-oriented encapsulation,
7. a stable outbound payload producer,
8. a bridge-facing payload provider through a PluginAdapter boundary.

### 5.2 What the Independent Pipeline Is Not

An Independent Pipeline is not:

1. a final receiver,
2. a read-only reader API,
3. a database injector,
4. an OpenTelemetry exporter,
5. a cross-language FFI object,
6. a Thin C ABI implementation,
7. a bridge,
8. a port,
9. a broker,
10. a shared file map,
11. a generic messaging system,
12. a destination router,
13. a semantic interpreter of final receiver needs.

### 5.3 Identity Rule

```text
The Independent Pipeline owns internal payload preparation and stabilization.
It does not own external destination meaning.
```

## 7. Core Structural Model

### 7.1 High-Level Structure

```text
Independent Pipeline
    ├── internal content / material intake
    ├── validation participation
    ├── metadata preparation
    ├── timestamp stabilization
    ├── internal containers / slots
    ├── bundle agent
    ├── container moderator
    ├── waiting lists
    ├── round managers
    ├── internal dispatcher mechanics
    └── stable outbound payload exposure
```

### 7.2 External Relationship

```text
Independent Pipeline
        ↓
Stable Outbound Payload
        ↓
Pipeline PluginAdapter Boundary
        ↓
Communication Bridge
        ↓
Bound Outbound Ports
        ↓
Thin C ABI Outbound Passage
        ↓
External Receiver or Optional Add-on
```

### 7.3 Meaning of the Boundary

The pipeline exposes payload material.

It does not expose:

1. slot identity,
2. container identity,
3. round state,
4. waiting-list state,
5. internal offsets,
6. internal ownership state,
7. internal pointer contracts,
8. final destination expectations.

The pipeline’s outbound boundary must be payload-centered, not internals-centered.

## 8. Stable Outbound Payload

### 8.1 Definition

A stable outbound payload is the material produced by the Independent Pipeline after internal preparation and stabilization, and before bridge-mediated outbound delivery.

It is the pipeline’s external product.

### 8.2 Stable Payload Properties

A stable outbound payload should satisfy:

|            Record ID | Property            | Meaning                                                                  |
| -------------------: | ------------------- | ------------------------------------------------------------------------ |
| ASIP-001-PAYLOAD-001 | Prepared            | Internal pipeline preparation is complete                                |
| ASIP-001-PAYLOAD-002 | Stable              | The payload is not still being mutated by pipeline internals             |
| ASIP-001-PAYLOAD-003 | Outbound            | The payload is eligible to be presented toward the bridge boundary       |
| ASIP-001-PAYLOAD-004 | Internals-free      | It does not require external receivers to know pipeline internals        |
| ASIP-001-PAYLOAD-005 | Destination-neutral | It does not encode final receiver semantics                              |
| ASIP-001-PAYLOAD-006 | ABI-passable        | It can later be passed through a Thin C ABI boundary by an outbound port |

### 8.3 Payload Naming Rule

The preferred architectural term is:

```text
Outbound Payload
```

Terms that should be avoided at this foundation level:

```text
PayloadRef
ReadSideContainer
ReceiverPayloadView
DestinationEnvelope
ReaderMaterial
EndReceiverMessage
```

Those terms either imply access back into the pipeline or prematurely encode destination semantics.

## 9. Destination Semantics Exclusion

### 9.1 Core Rule

```text
Destination semantics are not pipeline semantics.
```

### 9.2 Meaning

The pipeline must not know whether an outbound payload will be used by:

1. a read-only reader container,
2. a database add-on,
3. a graph database add-on,
4. an OpenTelemetry add-on,
5. a Python FFI wrapper,
6. a C# FFI wrapper,
7. an MQL5 FFI wrapper,
8. a C API receiver,
9. a memory cache container,
10. a shared file map writer,
11. a Chronicle-like local store,
12. a custom receiver.

Those are external destinations or optional add-ons beyond the core pipeline.

### 9.3 Configuration Implication

Configuration may bind ports or add-ons.

Configuration may enable or disable bridge/port relationships.

Configuration may select which outbound ports receive payload.

But configuration must not cause the pipeline core to internalize destination-specific semantics.

## 10. Bridge Relationship

### 10.1 Bridge Role

The Communication Bridge belongs to ASCC.

The bridge:

1. binds a payload-providing PluginAdapter to outbound ports,
2. owns or consults a bound port registry,
3. mechanically delivers or copies payload into bound ports,
4. waits for port admission,
5. does not wait for final receiver completion,
6. does not interpret payload semantics,
7. does not own final destination internals.

### 10.2 Pipeline-to-Bridge Relationship

The pipeline does not call final receivers directly.

The pipeline exposes payload to a PluginAdapter boundary.

The bridge then handles binding and delivery to ports.

```text
Pipeline
    → PluginAdapter boundary
        → Bridge
            → Bound Port
```

### 10.3 Anti-Collapse Rule

```text
The pipeline must not become a bridge.
The bridge must not become a pipeline.
```

The pipeline produces stable payload.

The bridge performs communication orchestration.

---

## 11. Outbound Port Relationship

### 11.1 Port Role

Outbound Ports belong to ASOP.

Each outbound port owns its own delivery mechanics.

A port may own:

1. port-local intake,
2. port-local containers,
3. port-local waiting lists,
4. port-local delivery round manager,
5. port-local dispatcher,
6. Thin C ABI outbound passage,
7. admission/result reporting to the bridge.

### 11.2 Port Boundary

The bridge delivers payload into the port boundary.

After admission, the port owns its local delivery mechanics.

The pipeline does not know what happens inside the port.

The bridge does not wait for final receiver completion.

### 11.3 Anti-Collapse Rule

```text
The pipeline must not absorb port-local delivery rounds.
```

Port-local mechanics are repeated write-side-like mechanics in a narrower outbound scope.

---

## 12. Thin C ABI Relationship

### 12.1 Thin C ABI Role

Thin C ABI belongs to the outbound passage model.

It is:

```text
a logic-free outbound payload passage protocol.
```

It is not:

1. a schema engine,
2. a validator,
3. a JSON parser,
4. a C struct requirement,
5. a database mapper,
6. an OpenTelemetry projector,
7. a receiver manager,
8. a read-only reader API,
9. a pipeline internals exposure mechanism.

### 12.2 Thin C ABI Direction

The Thin C ABI is outbound.

It passes payload from the system toward an external destination or add-on.

It does not invite external receivers to enter the pipeline and read internal data.

### 12.3 Thin C ABI Payload Rule

```text
Thin C ABI passes outbound payload material.
It must not expose pipeline internals.
```

Allowed payload content may include:

1. C-style structs,
2. JSON bytes,
3. raw binary blocks,
4. FlatBuffers,
5. custom records,
6. market data blocks,
7. any receiver-understood representation.

The Thin C ABI does not care.

The final receiver or add-on interprets the payload.

## 13. Add-on Relationship

### 13.1 Add-ons Are Beyond Core

Optional add-ons belong beyond the Thin C ABI boundary or around it as extension packages.

Examples include:

1. relational database adapter,
2. graph database adapter,
3. OpenTelemetry adapter,
4. Python FFI wrapper,
5. C# FFI wrapper,
6. MQL5 FFI wrapper,
7. read-only container writer,
8. shared file map writer,
9. memory cache container writer,
10. Chronicle-like local store writer.

### 13.2 Add-on Ownership

Add-ons own destination-specific semantics.

The pipeline does not own them.

The bridge does not own them.

The Thin C ABI does not interpret them.

### 13.3 Add-on Rule

```text
Add-ons may understand destination semantics.
Core pipeline must not.
```

## 14. Single Writer and Pipeline Instance Model

### 14.1 Single Writer Pipeline

The initial correctness model assumes:

```text
one writer-oriented flow
→ one independent pipeline instance
```

This does not mean that the entire system can only have one writer.

It means each writer should be able to own or instantiate an independent pipeline instance.

### 14.2 Multiple Pipeline Instances

Multiple writers may run as multiple independent pipeline instances:

```text
Writer A → Independent Pipeline A
Writer B → Independent Pipeline B
Writer C → Independent Pipeline C
```

Parallelism emerges at the pipeline-instance level.

### 14.3 Multithreading Rule

```text
A pipeline must be correct without internal multithreading.
```

Internal multithreading is an execution policy, not a structural requirement.

Possible execution policies include:

1. single-thread deterministic pipeline,
2. one pipeline per writer thread,
3. bounded bridge copy workers,
4. port-local workers,
5. add-on-local workers.

These do not change the architectural identity of the Independent Pipeline.

## 15. Configuration Role

### 15.1 What Configuration May Do

Configuration may define:

1. pipeline capacity,
2. slot count,
3. container count,
4. validation mode,
5. timestamp mode,
6. metadata mode,
7. bridge binding enablement,
8. outbound port selection,
9. add-on enablement,
10. execution policy,
11. diagnostic level.

### 15.2 What Configuration Must Not Do

Configuration must not:

1. move destination semantics into the pipeline core,
2. make the pipeline know read-only reader internals,
3. make the pipeline know database internals,
4. make the pipeline know OpenTelemetry internals,
5. make the pipeline know language-specific object internals,
6. turn Thin C ABI into a semantic layer,
7. turn the bridge into a pipeline,
8. turn ports into hidden pipeline internals.

### 15.3 Configuration Principle

```text
Configuration instantiates and tunes the pipeline.
It does not redefine the pipeline’s architectural responsibility.
```

## 16. Core Lifecycle

### 16.1 Lifecycle Overview

```text
1. Content or material enters the pipeline.
2. Pipeline internal mechanics prepare the material.
3. Metadata and timestamp semantics are stabilized where applicable.
4. Internal containers / slots / rounds coordinate placement and readiness.
5. Pipeline produces stable outbound payload.
6. Pipeline exposes payload at PluginAdapter-facing boundary.
7. Communication Bridge binds PluginAdapter to outbound ports.
8. Bridge delivers payload to ports.
9. Ports pass payload through Thin C ABI.
10. Add-ons or external receivers interpret the payload.
```

### 16.2 Pipeline Authority End

Pipeline authority ends at step 6.

The bridge owns step 7 and step 8.

Ports own step 9.

Add-ons / receivers own step 10.

## 17. Correctness Model

### 17.1 Correctness Unit

The Independent Pipeline is the correctness unit for internal payload production.

A pipeline is correct when:

1. its internal mechanics preserve ownership,
2. its internal state transitions are valid,
3. its outbound payload is stable before exposure,
4. its outbound boundary does not leak internals,
5. it does not interpret final destination semantics,
6. it does not depend on final receiver behavior.

### 17.2 Correctness Independence

A pipeline must remain correct even if:

1. no database add-on exists,
2. no OpenTelemetry add-on exists,
3. no read-only container writer exists,
4. no language FFI wrapper exists,
5. only one outbound port is bound,
6. multiple outbound ports are bound,
7. the bridge runs single-threaded,
8. the bridge later enables bounded copy parallelism.

### 17.3 Correctness Rule

```text
Pipeline correctness is independent of final receiver semantics.
```

## 18. Anti-Collapse Rules

### 18.1 Pipeline vs Bridge

```text
Independent Pipeline ≠ Communication Bridge
```

The pipeline produces outbound payload.

The bridge binds and delivers to ports.

### 18.2 Pipeline vs Port

```text
Independent Pipeline ≠ Outbound Port
```

The pipeline exposes payload.

The port owns post-bridge delivery mechanics.

### 18.3 Pipeline vs Thin C ABI

```text
Independent Pipeline ≠ Thin C ABI
```

The pipeline prepares payload.

Thin C ABI passes payload outward.

### 18.4 Pipeline vs Add-on

```text
Independent Pipeline ≠ Add-on
```

The pipeline does not own database, telemetry, read-only container, or language-specific semantics.

### 18.5 Pipeline vs Read Side

```text
Independent Pipeline ≠ mandatory Read Side dual-dispatch layer
```

The pipeline does not require internal read-only/end-receiver branching.

### 18.6 Payload vs Internal Container

```text
Outbound Payload ≠ internal container / slot / round state
```

Outbound payload must not expose internal pipeline structure.

### 18.7 Configuration vs Architecture

```text
Configuration ≠ architectural responsibility redefinition
```

Configuration may enable ports and add-ons, but it must not move their semantics into the pipeline.

## 19. Non-Responsibilities

The Independent Pipeline is not responsible for:

1. selecting read-only vs end-receiver semantics,
2. managing read-only reader cursors,
3. managing shared file map internals,
4. managing memory cache container internals,
5. parsing JSON for final receivers,
6. requiring C-style struct payloads,
7. interpreting database schemas,
8. projecting OpenTelemetry spans, metrics, or logs,
9. translating Python objects,
10. translating C# objects,
11. translating MQL5 objects,
12. retrying final receiver failures,
13. handling final receiver backlog,
14. owning final receiver acknowledgements,
15. running a generic message broker,
16. owning external receiver lifecycle.

These responsibilities belong beyond the pipeline boundary.

## 20. Documentation Consequences

### 20.1 ASRS Status

The previous ASRS direction should be considered superseded as a primary internal-domain documentation path.

The useful material from ASRS should be absorbed into:

1. ASIP for independent pipeline mechanics,
2. ASOP for outbound ports and Thin C ABI,
3. ASAO for add-ons and external destination semantics.

### 20.2 ASCC Status

ASCC remains valid.

However, ASCC should be revised or extended to reflect the unified outbound binding model:

```text
Pipeline PluginAdapter
    ↔ Bridge
        ↔ Bound Outbound Ports
```

### 20.3 ASOP Requirement

A new ASOP documentation sequence is required.

It must define:

1. unified outbound port foundation,
2. port-owned delivery rounds,
3. Thin C ABI outbound passage,
4. outbound payload packet contract.

### 20.4 ASAO Requirement

A new ASAO documentation sequence is required.

It must define optional add-ons beyond Thin C ABI, including:

1. database add-ons,
2. telemetry add-ons,
3. language FFI wrappers,
4. read-only container add-ons,
5. custom receiver add-ons.

### 21.2 Implementation Boundary

The future pipeline implementation must not include:

```text

opentelemetry/

mql5_ffi/
```

```text

```

## 24. Summary

The Independent Pipeline model stabilizes the Assembler System around a simpler and stronger architecture:

```text
One pipeline mechanism.
One stable outbound payload boundary.
One bridge-mediated communication model.
One outbound port model.
One Thin C ABI passage protocol.
Many optional add-ons beyond the boundary.
```

The pipeline no longer needs to distinguish read-only readers from end receivers.

It no longer needs a mandatory dual-dispatch layer.

It no longer needs to copy between internal dispatching rounds to satisfy destination diversity.

Instead:

```text
The pipeline produces stable outbound payloads.
The bridge binds payload providers to outbound ports.
The ports pass payloads through Thin C ABI.
External receivers and add-ons own final semantics.
```

This foundation allows the system to remain mechanical, scalable, testable, and destination-neutral while preserving the ability to support many different external receiver models.
