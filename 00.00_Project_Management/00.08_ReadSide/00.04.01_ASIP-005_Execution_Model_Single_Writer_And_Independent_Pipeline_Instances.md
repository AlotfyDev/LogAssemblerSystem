## 2. Document Purpose

### 2.1 Purpose Statement

This document defines the execution model for Independent Pipeline instances in the Assembler System.

It answers the question:

```text
How should Independent Pipeline instances execute under a single-writer
correctness model, how may multiple instances coexist, and how can optional
parallelism be introduced without changing ASIP ownership, expanding ASIP into
a scheduler, or leaking destination semantics into the pipeline?
````

The document establishes that:

```text
single-writer correctness comes first
instance independence comes second
optional parallelism comes after correctness
```

The Independent Pipeline must be valid even if executed deterministically by one thread.

Parallelism is an execution policy.

Parallelism is not the architectural identity of the pipeline.

### 2.2 Relationship to ASIP-001

`ASIP-001` established the Independent Pipeline as the primary correctness unit.

This document specializes that principle into execution terms:

```text
One configured pipeline instance must be correct as an independent execution unit.
```

### 2.3 Relationship to ASIP-002

`ASIP-002` defined the internal mechanics that may execute inside the pipeline:

```text
admission
validation
metadata preparation
timestamp stabilization
material preparation
container placement
slot occupancy
moderation
waiting-list coordination
round management
internal dispatch readiness
stabilization
```

This document defines how these mechanics are sequenced, executed, and protected under the single-writer baseline.

### 2.4 Relationship to ASIP-003

`ASIP-003` established that stable outbound payload exposure is the terminal ASIP boundary.

This document preserves that execution boundary:

```text
ASIP execution ends at stable outbound payload exposure.
ASIP execution does not include bridge delivery, port delivery, Thin C ABI passage,
or final receiver behavior.
```

### 2.5 Relationship to ASIP-004

`ASIP-004` established that pipeline configuration tunes only ASIP-owned behavior.

This document applies that to execution:

```text
Execution policy may tune how the pipeline runs.
Execution policy must not redefine what the pipeline owns.
```

## 4. Core Thesis

### 4.1 Primary Thesis

```text
An Independent Pipeline must be correct under single-writer execution.

Multiple pipelines may run independently.

Optional parallelism may be introduced only as a bounded execution policy that
does not alter ASIP ownership or destination neutrality.
```

### 4.2 Expanded Thesis

The Assembler System does not need to define its pipeline correctness around mandatory internal multithreading.

The pipeline should first be valid as a mechanically stable, deterministic, single-writer payload-production path.

Once that is true, the system may scale through:

1. multiple independent pipeline instances,
2. one writer per pipeline instance,
3. bounded bridge-side copy workers outside ASIP,
4. port-local delivery workers outside ASIP,
5. add-on-local workers outside ASIP,
6. optional ASIP-internal bounded concurrency only where formally justified.

### 4.3 Execution Rule

```text
Execution policy may improve throughput.
It must not become architecture.
```

## 5. Single-Writer Baseline

### 5.1 Definition

Single-writer baseline means that one writer-oriented flow owns mutation authority for one Independent Pipeline instance at a time.

The baseline assumption is:

```text
one writer
→ one Independent Pipeline instance
→ one internally coherent payload-production lifecycle
```

### 5.2 Meaning

Single-writer does not mean:

1. the whole system has only one writer,
2. the process has only one thread,
3. only one pipeline can exist,
4. bridge or ports cannot run separately,
5. diagnostics cannot observe safely,
6. downstream delivery cannot be asynchronous.

It means:

```text
within one pipeline instance, internal mutation is governed by one writer authority.
```

### 5.3 Single-Writer Responsibilities

The single writer may drive:

1. admission,
2. validation participation,
3. metadata preparation,
4. timestamp stabilization,
5. material preparation,
6. internal container placement,
7. slot occupancy,
8. internal waiting-list movement,
9. round execution,
10. stabilization,
11. stable outbound payload exposure.

### 5.4 Single-Writer Non-Responsibilities

The single writer does not drive:

1. bridge session execution,
2. outbound port delivery,
3. Thin C ABI passage,
4. database writes,
5. telemetry export,
6. language-wrapper invocation,
7. final receiver acknowledgement.

### 5.5 Single-Writer Rule

```text
A pipeline must preserve correctness when all ASIP-owned mutation is driven by
one writer authority.
```

## 6. Independent Pipeline Instance Model

### 6.1 Instance Definition

An Independent Pipeline instance is a configured execution unit that owns its internal mechanics and produces stable outbound payloads.

It is the execution realization of the ASIP correctness unit.

### 6.2 Instance-Owned Execution

A pipeline instance may own execution over:

1. its configuration-derived profile,
2. its admission state,
3. its internal containers,
4. its slots,
5. its waiting lists,
6. its round managers,
7. its internal dispatcher readiness,
8. its stable outbound payload exposure state,
9. its diagnostics within ASIP authority,
10. its recycling state within ASIP authority.

### 6.3 Instance Non-Owned Execution

A pipeline instance must not own execution over:

1. ASCC bridge core,
2. ASCC channel/session/binding lifecycle,
3. ASOP outbound port delivery round,
4. ASOP Thin C ABI passage,
5. ASAO add-on execution,
6. final receiver lifecycle,
7. downstream persistence lifecycle.

### 6.4 Instance Rule

```text
Each Independent Pipeline instance is a complete ASIP execution unit.
It is not a partial subroutine inside another pipeline.
```

## 7. Multiple Pipeline Instances

### 7.1 Multi-Instance Model

The system may support multiple independent pipeline instances:

```text
Writer A → Independent Pipeline A
Writer B → Independent Pipeline B
Writer C → Independent Pipeline C
```

Each pipeline instance remains internally independent.

### 7.2 Independence Requirements

Pipeline instances must not share mutable internal structures unless a later formal model introduces an explicit shared boundary.

By default, each instance owns its own:

1. configuration profile,
2. containers,
3. slots,
4. waiting lists,
5. round managers,
6. stabilization state,
7. exposure state,
8. recycling state.

### 7.3 Shared Components

Shared read-only definitions may exist, such as:

1. type definitions,
2. static policies,
3. immutable configuration templates,
4. compile-time traits,
5. constants,
6. diagnostic category definitions.

Mutable runtime state should not be shared by default.

### 7.4 Multi-Instance Rule

```text
Scale first by adding independent pipeline instances, not by making one
pipeline internally concurrent by default.
```

## 8. Execution Phases

### 8.1 Phase Overview

The canonical single-writer execution sequence is:

```text
configure
→ construct
→ admit
→ prepare
→ place
→ moderate
→ coordinate rounds
→ stabilize
→ expose
→ retain until bridge claim or expiration
→ release / recycle
```

### 8.2 Phase Table

|          Record ID | Phase             | ASIP-Owned | Meaning                                                         |
| -----------------: | ----------------- | ---------- | --------------------------------------------------------------- |
| ASIP-005-PHASE-001 | configure         | yes        | Configuration is validated and applied                          |
| ASIP-005-PHASE-002 | construct         | yes        | Pipeline instance is created                                    |
| ASIP-005-PHASE-003 | admit             | yes        | Material enters pipeline authority                              |
| ASIP-005-PHASE-004 | prepare           | yes        | Validation, metadata, timestamp, and material preparation occur |
| ASIP-005-PHASE-005 | place             | yes        | Material is placed into containers/slots where applicable       |
| ASIP-005-PHASE-006 | moderate          | yes        | Internal state is checked for safe progression                  |
| ASIP-005-PHASE-007 | coordinate_rounds | yes        | Internal round managers coordinate work                         |
| ASIP-005-PHASE-008 | stabilize         | yes        | Payload becomes internally stable                               |
| ASIP-005-PHASE-009 | expose            | yes        | Stable outbound payload becomes available                       |
| ASIP-005-PHASE-010 | bridge_claim      | boundary   | ASCC may claim/admit exposed payload                            |
| ASIP-005-PHASE-011 | release           | yes        | ASIP releases internal ownership after safe boundary event      |
| ASIP-005-PHASE-012 | recycle           | yes        | Internal structures may become reusable                         |

### 8.3 Phase Rule

```text
ASIP phases end at stable outbound exposure and ASIP-side release/recycle.
Bridge claim is a boundary event, not ASIP delivery execution.
```

## 9. Deterministic Execution Requirement

### 9.1 Definition

Deterministic execution means that, for a given input sequence, configuration, and internal state, the single-writer pipeline produces predictable state transitions and exposure outcomes.

### 9.2 Determinism Applies To

Determinism should apply to:

1. admission decisions,
2. validation participation results,
3. metadata preparation behavior,
4. timestamp stabilization behavior where policy allows,
5. material preparation,
6. container/slot placement rules,
7. waiting-list ordering,
8. round progression,
9. exposure readiness,
10. release/recycle eligibility.

### 9.3 Determinism Does Not Require

Determinism does not require:

1. all timestamps be identical across executions if wall-clock input is part of policy,
2. all bridge behavior be deterministic under ASIP,
3. all downstream receivers behave deterministically,
4. all diagnostic emission ordering be identical under future concurrency,
5. all optional external add-ons run synchronously.

### 9.4 Determinism Rule

```text
The baseline pipeline execution model should be explainable as a deterministic
state-transition path before optional concurrency is added.
```

## 10. Internal Round Execution

### 10.1 Round Role

Internal rounds organize repeatable phases of ASIP-owned work.

They may coordinate:

1. admission,
2. placement,
3. preparation,
4. stabilization,
5. exposure readiness,
6. recycling.

### 10.2 Round Execution Under Single Writer

Under the single-writer baseline, a round does not require a separate thread.

A round is a logical execution phase.

It may be represented by:

1. a function call,
2. a state transition,
3. a bounded loop,
4. a policy-driven step,
5. a future processing unit.

### 10.3 Round Non-Meaning

A round is not automatically:

1. a thread,
2. a worker,
3. a scheduler task,
4. a coroutine,
5. a queue consumer,
6. a bridge protocol step,
7. a port delivery round.

### 10.4 Round Rule

```text
An ASIP round is an internal coordination phase.
It must not be treated as a mandatory thread or scheduler primitive.
```

## 11. Optional Bounded Concurrency

### 11.1 Definition

Optional bounded concurrency means that some ASIP-owned work may later be executed with limited parallelism if doing so preserves correctness, mutation safety, and boundary discipline.

### 11.2 Allowed Optional Concurrency Candidates

Future ASIP design may consider bounded concurrency for:

1. validation pre-checks,
2. metadata derivation,
3. payload material preparation,
4. diagnostic collection,
5. non-mutating observation,
6. batch readiness scanning,
7. safe recycling scans.

### 11.3 High-Risk Concurrency Areas

Concurrency is risky around:

1. container mutation,
2. slot occupancy changes,
3. waiting-list mutation,
4. round-state mutation,
5. exposure readiness transition,
6. mutation closure,
7. release/recycle boundary.

These require stronger formal modeling before parallelization.

### 11.4 Concurrency Permission Rule

```text
Concurrency may be introduced only where ownership, mutation, ordering,
and release boundaries remain explicit.
```

### 11.5 Concurrency Non-Rule

```text
The existence of multiple internal mechanics does not imply that they must run
in multiple threads.
```

## 12. Execution Policy

### 12.1 Execution Policy Definition

Execution policy is the configured or derived rule set that determines how a pipeline instance executes its internal mechanics.

It may define:

1. single-step execution,
2. batch execution,
3. drain mode,
4. bounded round budget,
5. diagnostic verbosity,
6. optional internal parallelism,
7. admission pressure behavior,
8. idle behavior.

### 12.2 Execution Policy Non-Meaning

Execution policy is not:

1. bridge protocol,
2. outbound port delivery policy,
3. Thin C ABI policy,
4. database retry policy,
5. telemetry batching policy,
6. final receiver scheduling policy.

### 12.3 Execution Policy Rule

```text
Execution policy governs ASIP-owned internal execution.
It must not govern communication, delivery, ABI passage, add-on projection,
or final receiver lifecycle.
```

## 13. Pipeline Execution State Model

### 13.1 Candidate Execution States

A pipeline instance may later derive execution states such as:

|          Record ID | State        | Meaning                                                                  |
| -----------------: | ------------ | ------------------------------------------------------------------------ |
| ASIP-005-STATE-001 | unconfigured | No valid configuration has been applied                                  |
| ASIP-005-STATE-002 | configured   | Configuration is valid                                                   |
| ASIP-005-STATE-003 | constructed  | Pipeline instance exists                                                 |
| ASIP-005-STATE-004 | idle         | Pipeline is ready but not processing material                            |
| ASIP-005-STATE-005 | admitting    | Admission is active                                                      |
| ASIP-005-STATE-006 | preparing    | Internal preparation is active                                           |
| ASIP-005-STATE-007 | coordinating | Internal rounds are active                                               |
| ASIP-005-STATE-008 | stabilizing  | Payload is approaching exposure readiness                                |
| ASIP-005-STATE-009 | exposing     | Stable payload is being made available                                   |
| ASIP-005-STATE-010 | draining     | Pipeline is processing remaining admitted material without new admission |
| ASIP-005-STATE-011 | releasing    | ASIP is releasing retained payload material                              |
| ASIP-005-STATE-012 | recycling    | Internal structures are being prepared for reuse                         |
| ASIP-005-STATE-013 | stopped      | Pipeline is not executing                                                |
| ASIP-005-STATE-014 | faulted      | Pipeline encountered an ASIP-owned execution fault                       |

### 13.2 Forbidden Execution States

ASIP execution state must not include:

```text
bridge_running
session_executing
port_delivering
abi_passing
database_writing
telemetry_exporting
python_callback_running
receiver_acknowledged
```

### 13.3 State Rule

```text
Pipeline execution state describes ASIP-owned execution only.
It must not encode bridge, port, ABI, add-on, or receiver execution.
```

## 14. Admission Pressure and Backpressure

### 14.1 Admission Pressure Definition

Admission pressure occurs when the pipeline cannot immediately accept more material without violating capacity, correctness, or lifecycle rules.

### 14.2 ASIP-Owned Pressure Handling

ASIP may handle admission pressure by:

1. rejecting new admission,
2. delaying new admission,
3. entering drain mode,
4. reporting pressure diagnostics,
5. applying configured admission limits,
6. exposing a capacity status.

### 14.3 ASIP Non-Ownership

ASIP admission pressure is not the same as:

1. bridge backpressure,
2. port backpressure,
3. ABI receiver delay,
4. database write backlog,
5. telemetry export backlog,
6. final receiver outage.

### 14.4 Pressure Rule

```text
ASIP admission pressure describes the pipeline’s ability to admit material.
It must not be treated as final receiver backpressure.
```

## 15. Observation During Execution

### 15.1 Observation Purpose

A pipeline may expose safe observation surfaces so that diagnostics, tests, or controllers can inspect ASIP-owned state without mutating internals.

### 15.2 Safe Observation May Include

1. pipeline execution state,
2. admission pressure state,
3. exposed payload count,
4. pending internal count,
5. fault status,
6. last internal transition,
7. diagnostic records,
8. configuration ID,
9. execution policy summary.

### 15.3 Unsafe Observation Must Exclude

1. mutable container references,
2. mutable slot references,
3. waiting-list mutation handles,
4. round manager mutation handles,
5. allocator internals,
6. lock internals,
7. destination handles,
8. bridge session handles,
9. port-local structures.

### 15.4 Observation Rule

```text
Observation must be read-safe and ASIP-bounded.
Observation must not become external co-ownership of pipeline internals.
```

## 16. Exposure During Execution

### 16.1 Exposure Event

A stable outbound payload may become exposed while the pipeline continues operating.

Exposure is not delivery.

Exposure is ASIP boundary availability.

### 16.2 Exposure Must Preserve

1. mutation closure,
2. payload identity,
3. exposure state,
4. retention policy,
5. release conditions,
6. destination neutrality.

### 16.3 Exposure Must Not Require

1. bridge completion,
2. port completion,
3. ABI completion,
4. database completion,
5. telemetry completion,
6. receiver acknowledgement.

### 16.4 Exposure Rule

```text
A pipeline may continue executing after exposing payloads, provided exposed
payload material remains stable until valid release conditions are met.
```

## 18. Relationship to ASCC

### 18.1 ASCC Execution Ownership

ASCC owns communication execution after stable payload exposure.

ASIP does not execute:

1. channel binding,
2. session opening,
3. bridge protocol,
4. bridge core orchestration,
5. bridge diagnostics,
6. bridge registries.

### 18.2 Boundary Interaction

ASIP may expose payloads that ASCC can claim or admit.

That is a boundary interaction.

It is not ASIP executing ASCC.

### 18.3 ASCC Boundary Rule

```text
ASIP execution produces and exposes stable payloads.
ASCC execution communicates them.
```

---

## 19. Relationship to ASOP

### 19.1 ASOP Execution Ownership

ASOP owns outbound port delivery execution and Thin C ABI passage.

ASIP does not execute:

1. port-local delivery rounds,
2. port-local containers,
3. port-local waiting lists,
4. port dispatchers,
5. Thin C ABI function calls,
6. receiver-facing result handling.

### 19.2 ASOP Boundary Rule

```text
ASIP execution stops before outbound port delivery begins.
```

---

## 20. Relationship to ASAO

### 20.1 ASAO Execution Ownership

ASAO owns optional add-on behavior beyond the Thin C ABI or receiver boundary.

Examples include:

1. database writer execution,
2. graph database adapter execution,
3. OpenTelemetry exporter execution,
4. Python wrapper execution,
5. C# wrapper execution,
6. MQL5 wrapper execution,
7. read-only container writer execution.

### 20.2 ASAO Boundary Rule

```text
ASIP execution does not project payloads into destination-specific meaning.
```

## 21. Execution Safety Principles

### 21.1 Mutation Safety

ASIP-owned mutation must remain under controlled execution authority.

### 21.2 Ordering Safety

Internal phase ordering must remain explainable.

### 21.3 Exposure Safety

Stable outbound payload exposure must preserve mutation closure.

### 21.4 Instance Safety

One pipeline instance must not accidentally mutate another instance’s internal state.

### 21.5 Boundary Safety

Execution must not cross into ASCC, ASOP, ASAO, or final receiver authority.

### 21.6 Safety Rule

```text
Execution safety means preserving ASIP ownership, ordering, mutation closure,
and boundary exclusion under the selected execution policy.
```

## 26. Summary

The ASIP execution model is intentionally conservative.

The baseline is:

```text
one writer
→ one independent pipeline instance
→ deterministic internal execution
→ stable outbound payload exposure
```

System scaling should first occur through:

```text
multiple independent pipeline instances
```

not through mandatory internal multithreading.

Optional concurrency may be introduced later, but only where:

```text
ownership is explicit
mutation is controlled
ordering is explainable
exposure remains stable
release boundaries are safe
ASIP does not absorb ASCC, ASOP, ASAO, or receiver execution
```

The final conclusion is:

```text
ASIP execution is payload-production execution.

It is not bridge execution.
It is not outbound port delivery.
It is not Thin C ABI passage.
It is not add-on projection.
It is not final receiver lifecycle.
```

This prepares the next document:

```text
ASIP-006_Recycling_Lifecycle_And_Admission_Boundaries.md
```

which will define release, retention, recycling, and admission-boundary lifecycle in detail.
